# 본선 논증 척추 — 인사이트 → 심사·제품·시연

## 🎤 한 문장 포지셔닝 (발표 첫 문장)
JB LocalGuard OS는 AI 챗봇이 아니라, 전북은행과 JB우리캐피탈 RM이 민감정보를 밖으로 내보내지 않고 근거·승인·감사를 묶어 실행하는 지역은행 거버넌드 AI 운영체계다. [[인사이트맵]] [[연결-1]] [[연결-3]] [[연결-6]]

## 🏆 심사기준별 득점 논리 (1~5)

### 1. 주제·문제정의
**핵심 주장**: 문제는 “AI 답변 품질”이 아니라 지역은행 현업에서 반복되는 판단·승인·감사 단절이다.

**뒷받침 인사이트·근거**: LocalGuard OS의 본체는 모델이 아니라 `Case→AgentRun→Agent→Skill→Evidence→Approval→Audit` 운영계약이다. 소상공인·전세·보이스피싱은 RM 화면에서 실제로 얽히는 고객·물건·계좌 리스크다. [[인사이트맵]] [[연결-1]] [[연결-2]] [[연결-6]]

**제품·MVP 증거**: MVP는 세 도메인을 같은 Case 구조로 태운다. `computeRiskDecision`, `buildDashboardData`, `auditChainRecords`, `moveCaseToColumn`이 판단·근거·승인·상태변화·감사로그를 반복한다.

**시연 장면**: `?demo=sme`에서 한 케이스가 위험판단, 근거, 승인대기, 감사로그로 이어지는 흐름을 먼저 보여준다.

**예상 반론·방어**: “범위가 넓다”는 반론에는 기능 나열이 아니라 같은 운영계약을 다른 도메인에 적용한 것이라고 답한다. 데이터 레인과 실행권한은 분리한다. [[인사이트맵]] [[연결-5]]

### 2. 금융연계·고객가치
**핵심 주장**: 고객가치는 토큰비 절감이 아니라 RM 재작업 감소, 사후관리 누락 방지, 승인 가능한 결정패키지 생산이다.

**뒷받침 인사이트·근거**: JB 타깃은 전북은행+JB우리캐피탈 2계열사로 좁힐수록 강하다. 전북은행은 지역 RM·소상공인·사후관리, JB우리캐피탈은 비오토·기업금융·연체 변동성 파일럿으로 잡는다. [[인사이트맵]] [[연결-4]] [[연결-6]]

**제품·MVP 증거**: 대시보드는 단순 점수판이 아니라 승인대기, 재작업, 사후관리, 위험코드, 근거포인터를 보여준다. 채택률보다 `approval_lead_time`, `rework_rate`, `override_reason_distribution`, `follow_up_completion`을 본지표로 둔다. [[연결-5]]

**시연 장면**: `?demo=jeonse`에서 물건·계약·근거를 한 Case로 묶고, 승인 전 실행이 막히는 장면을 보여준다.

**예상 반론·방어**: “ROI가 과하다”는 공격에는 보수 모델을 기본값으로 두고, 자본구제·리스크 회피는 확장 시나리오로 분리한다고 답한다. canon 승인 전 정량 헤드라인은 고정하지 않는다. [[인사이트맵]] [[연결-1]] [[연결-4]]

### 3. AI Agent 설계·구현
**핵심 주장**: Agent는 자동 심사관이 아니라 특화모델·룰·검색·RPA를 호출하고 승인 전까지 실행을 막는 운영 레이어다.

**뒷받침 인사이트·근거**: PII 비반출은 보안 문구가 아니라 아키텍처다. 원본 PII·개인신용정보는 내부에 두고, 외부에는 토큰·가명·최소맥락만 라우팅한다. 승인우선은 데모 버튼이 아니라 구매 장벽 제거 장치다. [[인사이트맵]] [[연결-2]] [[연결-3]]

**제품·MVP 증거**: `Approval` 없이는 고객 알림·약정 생성·자금 release 같은 고위험 action이 진행되지 않는다. `Evidence`와 `Audit`은 판단 결과와 함께 남는다. `verify_static.py`와 Playwright는 이 계약을 검증 대상으로 삼는다.

**시연 장면**: 승인 큐에서 근거 묶음, 이유 코드, 승인/반려 경로, 감사로그 생성을 한 화면에서 보여준다.

**예상 반론·방어**: “AI가 자동결정하면 위험하다”는 공격에는 완전자동 승인·거절 엔진이 아니라고 답한다. RM/심사역의 인간 승인과 사후 감사를 전제로 한 결정패키지 생성 콘솔이다. [[연결-6]]

### 4. MVP 완성도·검증
**핵심 주장**: MVP 완성도는 예쁜 화면보다 상태 변화 불변식으로 증명한다.

**뒷받침 인사이트·근거**: `입력→판단→근거→승인게이트→상태변화→감사로그`가 세 데모에 반복되면 OS다움이 증명된다. “좋은 답변”이 아니라 승인 전 실행 차단, 근거 없는 주장 차단, 감사 append를 검증한다. [[인사이트맵]] [[연결-5]] [[연결-6]]

**제품·MVP 증거**: 정적 바닐라JS로 구동한다. 함수계약은 `computeRiskDecision`, `buildDashboardData`, `auditChainRecords`, `moveCaseToColumn`이다. `?demo=sme`, `?demo=jeonse`, `?demo=phishing`가 같은 계약을 다른 업무로 보여준다.

**시연 장면**: 세 데모를 빠르게 전환하고, 각 케이스가 같은 감사 체인에 쌓이는 장면을 보여준다.

**예상 반론·방어**: “정적 MVP라 실도입과 멀다”는 공격에는 MVP는 샘플 Evidence와 토큰화 이벤트를 증명하고, 실도입은 정보계 CDC Read-only, Tokenization Proxy, Nexacro WebView, RM 승인 후 MCI/EAI Write로 분리한다고 답한다. [[연결-6]]

### 5. 혁신·확장·운영리스크
**핵심 주장**: 혁신은 더 큰 모델이 아니라 Zero-PII Case Hub와 승인우선 운영계약이다.

**뒷받침 인사이트·근거**: 통합 고객지능은 계열사 원장 통합이 아니다. 토큰, 위험코드, 근거포인터, 승인상태를 통합 관측하는 구조다. 광주은행은 본선 본문이 아니라 확장 근거로만 둔다. [[인사이트맵]] [[연결-2]] [[연결-4]]

**제품·MVP 증거**: 공개형·유료 승인형·계약형 데이터 레인을 구분한다. 원본 PII 비반출, 승인 전 차단, 감사로그 실패 시 fail-closed를 운영 원칙으로 둔다. [[연결-3]] [[연결-5]]

**시연 장면**: `?demo=phishing`에서 긴급 위험을 띄우되, 고객 대상 실행은 승인 전 차단되는 장면을 보여준다.

**예상 반론·방어**: “유출 위험이 없나”라는 질문에는 “전무”라고 말하지 않는다. 원본 PII 비반출과 다층통제로 위험을 실질 저감하고, 프롬프트 인젝션·로그·캐시·재식별 위험은 감사와 스캔 대상으로 관리한다고 답한다. [[연결-3]]

메시지: “빠른 AI보다 금융권에 필요한 것은 통제된 실행입니다.” [[인사이트맵]] [[연결-3]] [[연결-5]]

## 🛡 리스크·과장 방어

- “고객정보 유출 가능성 전무”라고 말하지 않는다. “원본 PII 비반출과 다층통제로 위험을 실질 저감”이라고 말한다. [[연결-3]]
- 본선 타깃은 전북은행+JB우리캐피탈 2계열사로 고정한다. 광주은행은 동일 지주 확장 근거로만 둔다. [[연결-4]]
- ROI는 보수 모델을 기본값으로 둔다. 상방값은 본문 헤드라인이 아니라 확장 시나리오로 분리한다. [[연결-1]] [[연결-4]]
- “망분리 폐지”라고 말하지 않는다. 망분리 원칙, 예외, 보안통제, 결과책임으로 말한다. [[연결-3]]
- “AI 자동 승인/거절”이라고 말하지 않는다. 인간 승인과 사후 감사를 전제로 한 결정패키지 생성·승인 콘솔이라고 말한다. [[연결-6]]
- “생산성 평균 향상”을 단정하지 않는다. 리드타임, 재작업률, 블라인드 품질점수, faithfulness score로 검증하겠다고 말한다. [[연결-5]]
- “승인 안전 100%”라고 말하지 않는다. 평가 범위에서 critical flow 불변식을 검증한다고 말한다. [[연결-5]]
- 등기 API 기간, JB FY2025 수치, MOU 원문, 보존기간은 canon 확정 전 단정하지 않는다. [[연결-2]] [[연결-3]] [[연결-4]]
