# 연결-6 — 도메인·시스템 — 도메인지식·코어뱅킹 정합·아키텍처·데이터품질

## 연결된 발견
- 여신 실무의 핵심은 “상담→품의→전결/본부이관→실행→EWS·자금용도·EOD 사후관리”의 트리거 체인이다. AI가 들어갈 자리는 판단 대체가 아니라 이 체인의 증빙·기한·승인 경로를 묶는 자리다. [[D2-결과.md]]
- 은행 시스템은 계정계·정보계·채널계가 분리되어 있다. JB LocalGuard OS는 계정계에 쓰지 않고, CDC Read-only와 Tokenization Proxy로 위험 신호를 읽고, Nexacro WebView에서 RM에게 보이고, 최종 Write는 RM 승인 후 MCI/EAI가 수행하는 구조가 맞다. [[D6-결과.md]]
- 금융형 에이전트 구현 패턴은 “검색은 grounding, 규칙은 deterministic gate, 에이전트는 분해·조정”이다. 승인/차단/제출/발송 같은 불가역 행동은 tool approval·human-in-the-loop·checkpoint 뒤에 둬야 한다. [[D9-결과.md]]
- 데이터·모델 신뢰성 쪽 결론도 같다. `신용정보법` 제36조의2, `개인정보 보호법` 제37조의2 때문에 완전자동 승인/거절 엔진보다 인간 승인과 사후 감사가 결합된 보조수단이 법·감사 적합성이 높다. [[D14-결과.md]]
- 소상공인 여신은 thin-file와 분절 데이터가 구조적이다. 그래서 “좋은 모델”보다 `생성시점`, `수집시점`, `마지막 갱신시점`, `결측률`, `이상치`, `원천-정합성`을 먼저 보여주는 데이터 신뢰도 게이트가 제품 기능이 된다. [[D14-결과.md]]
- RM의 반복 병목은 수십 장 서류 정합성 확인, 품의서 정성 의견 작성, EWS·3개월 사후점검·14일 EOD 기한 추적이다. RAG/OCR/Rule Engine/Approval Trace를 붙일 업무가 이미 명확하다. [[D2-결과.md]] [[D9-결과.md]]

## 1차 인사이트
- JB LocalGuard OS는 “AI 심사관”이 아니라 “승인 가능한 결정패키지 공장”이어야 한다. 패키지는 원문 근거, 데이터 freshness, 룰 ID, 모델 버전, RM 승인자, override 사유를 한 번에 남긴다. [[D14-결과.md]] [[D9-결과.md]]
- 운영계약 `Case→AgentRun→Agent→Skill→Evidence→Approval→Audit`는 추상 구조가 아니다. 여신 1건이 Case, 서류파싱·EWS·자금용도·EOD가 AgentRun, 세금계산서·거래내역·약관·RM 메모가 Evidence, RM/MCI 실행이 Approval, 재심·감리가 Audit로 대응된다. [[D2-결과.md]] [[D6-결과.md]] [[D9-결과.md]]
- “PII 비반출”은 보안 문구가 아니라 아키텍처 차별점이다. AI 콘솔은 가상 토큰 UUID와 비식별 feature만 보고, 실명 매핑과 실제 Write는 Nexacro/MCI 내부에서 끝낸다. [[D6-결과.md]] [[D14-결과.md]]
- 5천만 원 초과 사업자대출 3개월 사후점검과 기업여신 이자 연체 14일·3영업일 전 통지는 데모에 바로 쓸 수 있는 하드룰이다. 모델이 애매해도 룰 게이트와 Evidence 대조만으로 금융가치를 보인다. [[D2-결과.md]] [[D9-결과.md]]
- 한국어 금융 LLM은 단독 판단 엔진으로 밀면 약하다. `KFinEval-Pilot`에서 공개모델 최고 성적은 지식 64.19%, 추론 6.30이고, `financial-mmlu-ko`는 455행 규모다. 그러니 내부문서 RAG + 규칙엔진 + 인간 승인 조합이 정답이다. [[D14-결과.md]] [[D9-결과.md]]
- JB의 강점은 “최첨단 모델”보다 지방은행 실무 적합성이다. 전북은행은 자바 기반·MDA/MDD 코어뱅킹 현대화 경험이 있고, RM은 지역 소상공인·정책자금·지역신보 보증서 상담에 시간을 쓴다. 이 둘을 잇는 RM에이전트가 심사 설득력이 높다. [[D6-결과.md]] [[D2-결과.md]]

## 모순·텐션
- D6는 전북은행+광주은행 통합 전산센터 철회와 개별 전산망을 다룬다. 현재 제품 정본은 JB 2계열사를 전북은행+JB우리캐피탈로 둔다. reconcile: D6의 “개별 전산망·어댑터 필요”는 그룹 내 멀티 엔티티 배포 원칙 근거로 쓰고, 시연 타깃 표기는 전북은행+JB우리캐피탈로 고정한다. [[D6-결과.md]]
- D2는 오픈뱅킹·펌뱅킹으로 사업용 계좌 거래내역과 전자세금계산서를 대조하는 사후점검 Agent를 제안한다. D6는 내부망 직접 연결에는 MCI/EAI·내부 표준 전문 어댑터가 필요하다고 못 박는다. reconcile: MVP는 샘플 Evidence와 토큰화 이벤트로 검증하고, 실도입은 은행별 MCI/EAI 매핑 과제로 분리한다. [[D2-결과.md]] [[D6-결과.md]]
- D9는 agentic retrieval·traffic splitting 등 최신 패턴을 제시하지만 일부 기능은 2026년 preview/beta 성격이 있다. reconcile: 본선에서는 구현 패턴 근거로만 쓰고, MVP는 classic RAG + rules + approval trace로 좁힌다. [[D9-결과.md]]
- D14는 데이터·모델만으로 자동 의사결정하면 안 된다고 경고한다. 이는 제품 약화가 아니라 승인우선·감사우선 포지셔닝을 강화한다. [[D14-결과.md]]

## 제품·심사 매핑
- 1=주제: 지역은행 RM의 실제 여신·사후관리 흐름을 겨냥한다. EWS, 자금용도외 유용, EOD, 품의, 전결권이 모두 금융 실무 언어다. [[D2-결과.md]]
- 2=금융가치: RM의 수작업 병목을 줄이고, 5천만 원 초과 사업자대출 3개월 점검과 14일 EOD 기한 누락을 막는다. 이는 시간 절감과 채권보전 가치가 같이 난다. [[D2-결과.md]]
- 3=Agent설계: RAG는 근거화, Rule Engine은 하드룰, 멀티에이전트는 문서추출·규정검색·정책판단·액션준비를 맡는다. 고객 대상 행동은 tool approval과 human-in-the-loop 뒤에 둔다. [[D9-결과.md]]
- 4=MVP검증: Nexacro WebView 모사 화면, 토큰화 Case, Evidence 대조, Approval hold/release, Audit trace를 한 흐름으로 보이면 된다. 계정계 Write 없이도 프로덕션 이식 서사를 입증한다. [[D6-결과.md]]
- 5=혁신·운영: 모델 성능 경쟁이 아니라 데이터 신뢰도 표시, 설명패킷, 재심·정정 파이프라인, 독립 검증, PII 비반출을 운영 OS로 묶는다. [[D14-결과.md]]

## 시연 한 컷
- RM 콘솔에 “사업자대출 7,000만 원 / 실행 후 82일 / 자금용도 점검 D-8” Case가 뜬다. Agent가 거래내역·세금계산서 Evidence를 대조하고, 일부 송금이 사업 목적 증빙과 불일치한다고 표시한다. 데이터 신뢰도는 출처별 freshness와 결측률로 보인다. Rule Engine은 “5천만 원 초과·3개월 내 점검” 하드룰을 붙이고, LLM은 1분 브리핑과 고객 요청자료 초안을 쓴다. RM이 승인하기 전까지 발송/회수 요청은 hold 상태다. 승인하면 WebView는 `APPROVED` 토큰만 넘기고, 실제 실행은 MCI/EAI 경로로 간다는 설명을 붙인다. [[D2-결과.md]] [[D6-결과.md]] [[D9-결과.md]] [[D14-결과.md]]
