## 🎯 교차 인사이트 Top (6~8)
- **LocalGuard OS의 본체는 모델이 아니라 운영계약이다.** LLM은 특화모델·룰·RPA를 호출하는 라우터이고, `Case→AgentRun→Agent→Skill→Evidence→Approval→Audit`가 법·서버·업무 흐름을 묶는다. 근거 [[연결-1]] [[연결-3]] [[연결-6]] / 심사기준 1·3·5 / so what: 발표 첫 문장을 “AI 챗봇”이 아니라 “지역은행 거버넌드 AI 운영체계”로 고정한다.

- **PII 비반출은 보안 문구가 아니라 제품 아키텍처다.** 원본 PII·개인신용정보는 내부에 두고, 외부에는 토큰·가명·최소맥락만 승격 라우팅한다. 계열사 공통 허브도 원장 통합이 아니라 토큰·위험코드·근거포인터·승인상태 통합이어야 한다. 근거 [[연결-2]] [[연결-3]] [[연결-6]] / 심사기준 3·5 / so what: “Zero-PII Case 360”을 핵심 차별어로 쓴다.

- **승인우선은 데모 버튼이 아니라 구매 장벽 제거 장치다.** 고위험 action은 승인 토큰 전까지 고객 알림·약정 생성·자금 release가 막혀야 하고, 승인에는 근거 묶음·이유 코드·재심·SLA·kill switch가 붙어야 한다. 근거 [[연결-3]] [[연결-4]] [[연결-5]] / 심사기준 2·3·5 / so what: MVP의 핵심 장면은 “AI 실행”이 아니라 “승인 전 차단과 해제 경로”다.

- **JB 타깃은 전북은행+JB우리캐피탈 2계열사로 좁힐수록 강하다.** 전북은행은 지역 RM·소상공인·사후관리, JB우리캐피탈은 비오토·기업금융·연체 변동성 문제를 갖는다. 광주은행 근거는 확장성으로만 둔다. 근거 [[연결-2]] [[연결-4]] [[연결-6]] / 심사기준 1·2·4 / so what: 시연과 ROI는 2계열사 파일럿 서사로 통일한다.

- **금융가치는 토큰비 절감이 아니라 재작업·사후관리·자본구제다.** 경제성 헤드라인은 보수 NPV와 RM 생산성으로 잡고, 확장 논리는 EL·RWA·자기자본비용 절감까지 연결한다. 검증 KPI는 총 리드타임·재작업률·승인 가능한 산출물 수로 둔다. 근거 [[연결-1]] [[연결-4]] [[연결-5]] / 심사기준 2·4 / so what: “모델 단가표” 대신 “승인대기, 재작업, 사후관리, RWA” 패널을 만든다.

- **멀티도메인은 기능 욕심이 아니라 RM의 실제 화면이다.** 소상공인, 전세, 보이스피싱은 같은 고객·물건·계좌 리스크로 얽힌다. 다만 공개형·유료 승인형·계약형 데이터 레인은 분리해야 한다. 근거 [[연결-2]] [[연결-4]] [[연결-5]] / 심사기준 1·2·3 / so what: 한 화면에 묶되 데이터 출처와 실행권한은 레인별로 표시한다.

- **MVP 검증은 “좋은 답변”이 아니라 “상태 변화 불변식”이다.** `입력→판단→근거→승인게이트→상태변화→감사로그`가 소상공인·전세·피싱에 반복되면 OS다움이 증명된다. PII leak 0/N, 승인 전 실행 0건, 근거 없는 주장 차단을 테스트로 보인다. 근거 [[연결-1]] [[연결-3]] [[연결-5]] [[연결-6]] / 심사기준 4·5 / so what: Playwright와 `verify_static.py` 결과를 기술 데모 증거로 격상한다.

- **정책룰화는 RM 편의 기능이 아니라 내부통제 기능이다.** 시행일·대상·예외·한도·의무를 Skill과 Rule Engine으로 만들고, 변경 로그와 승인 로그를 남기면 “AI 판단”이 아니라 “감사 가능한 정책 반영”이 된다. 근거 [[연결-2]] [[연결-3]] [[연결-6]] / 심사기준 3·5 / so what: 정책 문서 추출 화면은 요약보다 룰 ID·적용근거·변경이력을 보여준다.
