# 도메인 벼락치기 — 발표 30분 전

> 발표·Q&A에서 실제로 입 밖에 낼 법한 것만. 수치는 전부 출처 파일명 표기. 소스에 없는 수치는 절대 지어내지 말 것.
> **제품명**: 발표에서는 **JByond**(구 LocalGuard OS)로 통일해서 말한다. `_canon.md`는 아직 구 명칭("JB LocalGuard OS, 변경 금지")으로 남아있음 — 덱 실물 확인 기준 표기이니 canon 갱신 전까지는 이 문서·덱 표기를 우선한다.

## 1. 한 줄 정의

1. **RM (Relationship Manager)** — 기업여신·소상공인 고객을 직접 응대하며 심사·사후관리를 담당하는 은행 영업담당자.
2. **FDS (Fraud Detection System)** — 이상거래·보이스피싱 등 금융사기를 실시간 탐지해 차단하는 시스템.
3. **전세사기** — 임대인이 보증금을 반환할 능력·의사가 없거나 권리관계를 속여 세입자가 보증금을 떼이는 사기 유형(깡통전세·이중계약·무자본 갭투자 등).
4. **여신심사** — 대출 신청 고객의 상환능력·담보·신용도를 평가해 승인 여부를 결정하는 은행 핵심 업무.
5. **PII 비반출** — 고객 원본 개인신용정보(주민번호·계좌 등)는 내부에만 두고, 외부 프런티어 LLM에는 가명·토큰화된 최소 정보만 보내는 설계 원칙.
6. **승인 게이트 L0~L4** — 고객 대상 행동을 위험도별로 5단계(L0 내부기록 ~ L4 보류·차단)로 나눠 사람 승인을 강제하는 안전장치.
7. **폴백 사다리** — LLM 호출이 실패하면 다른 엔진으로 자동 재시도하고, 그마저 실패하면 사람 큐로 자동 격상하는 오류 복구 구조.
8. **운영계약(Case→AgentRun→Agent→Skill→Evidence→Approval→Audit)** — 위험신호가 케이스가 되고 에이전트가 근거를 남기며 판단하고, 사람이 승인해야 실행되고, 모든 과정이 감사로그로 남는 우리 제품의 뼈대.
9. **EWS (Early Warning System)** — 부실·리스크 징후를 사전에 포착해 조기 경보하는 체계. 사후관리 업무의 핵심.
10. **thin-file 고객** — 신용거래 이력이 얕아 기존 신용평가모델로는 정확한 평가가 어려운 소상공인·개인사업자 등 고객군.
11. **Enter Interaction** — 근거 패키지를 눈으로 확인한 담당자가 Enter(스페이스) 하나로 승인하는 keyboard-first 승인 UI — 사람이 판단에만 집중하게 하는 장치.
12. **CaseOps Engine** — 유입된 금융 이슈를 계열사×역할×케이스 구조로 정규화하고 우선순위를 계산하는 코어 엔진.
13. **Zero-PII Case Hub** — 원본 PII가 토큰화·게이트를 통과한 뒤에만 케이스로 합류하는 비식별 허브(외부 LLM엔 최소필드만).
14. **Audit LogAgent** — 실행·RM 판단·준법 승인·보류 사유를 자동 기록하는 감사 전담 에이전트.

## 3. 법령·규제 5개

1. **신용정보법 §40조의2 ①②⑥⑦⑧** — 가명처리 + 토큰↔원본 키 분리보관, 재식별 금지·재식별 시 즉시 중단·회수·파기. 우리 PII 4중방어(토큰화·모델 라우팅)의 1차 법적 근거. [`적법성-근거팩.md`]

## 4. 공격 질문 방어 원라이너 10개

1. **"생산성 X% 향상 숫자를 대라"** → 그 숫자를 성과지표로 팔지 않는다. 속도만 재면 거의 틀린다(숙련 개발자 RCT에선 완료시간이 오히려 +19%). 리드타임·품질·재작업률·근거충실성을 함께 잰다.
2. **"Enter가 마우스보다 49% 빠르다는 근거는?"** → KLM 기반 이론적 상한(전문가·무오류 가정)일 뿐이고 판단시간은 못 잰다. 성과가 아니라 "왜 이렇게 설계했나"의 설계근거로만 쓴다.
3. **"실제로 작동하나, 목업인가?"** → 운영계약과 4개 함수계약(`computeRiskDecision` 등)은 이미 브라우저에서 작동한다. 히어로 1케이스는 실 LLM이 목표이고, 나머지는 아직 개발 목표라고 정직하게 말한다.
4. **"ROI를 대라"** → 프레임은 연체 방어가 아니라 RM 반복행정 절감이다. 기준 시나리오 연 7.66억, 보수적으로 잡아도 0.83억이다.
5. **"고객 개인정보를 AI에 넣는 게 합법인가?"** → 원본 개인신용정보는 내부에만 두고, 외부 LLM에는 가명·토큰화 최소정보만 보낸다. 신용정보법 §40조의2를 앵커로 설계했다.
6. **"ChatGPT나 대형 SI가 그냥 하면 되지 않나?"** → 범용 LLM은 원본 PII를 못 만진다. 진입장벽은 모델 성능이 아니라 규제 정합 운영계약이다.
7. **"3도메인이 두 계열사에 똑같이 확장되나?"** → 아니다, 모듈식 비대칭 확장이다. 전세는 JB우리캐피탈 사업영역 밖이라 "3도메인 모두 캐피탈로 확장"이라고는 절대 말하지 않는다.
8. **"이 문제가 정말 큰가, AI 붙일 명분 아닌가?"** → 2026-06-30 시행된 사후관리 규제 강화로 점검의무 대상이 하향되며 처리 케이스가 폭증하는데, 4대 은행 RM은 1년 새 1,000명 이상 줄었다 — 수요·공급이 벌어지는 시점 문제다.
9. **"위험점수가 취약 소상공인을 과잉 표적하지 않나?"** → 단일 점수로 끝내지 않고 승인율·오류율 격차·사유코드 분포를 모니터한다. 확정 공정성 임계치 대신 조기경보 체계로 대응한다.
10. **"로컬모델이 진짜 되나, API 호출 흉내 아닌가?"** → 독립적으로 만들어진 두 게이트웨이가 같은 결론(관문에서 통제)에 도달했다. 다만 두 코드베이스는 아직 통합되지 않았고 관련 PR은 머지 대기 상태다.
11. **"에이전트가 14종이라더니 코드엔 몇 개인가?"** → 코드 레지스트리에는 58종 에이전트·40종 스킬이 등록돼 있고, 발표 조직도에는 사람이 감독 가능한 핵심 14종·스킬 25종만 노출했다 — 보여주는 수와 등록된 수를 구분해 답한다.

## 5. 절대 말하면 안 되는 것

- "생산성 X% 향상"을 성과지표처럼 단정하지 말 것 — 벤더 홍보 수치는 참고용일 뿐, 우리 정량주장에 쓰지 않는다.
- "Enter가 마우스보다 49% 빠르다"를 성과라고 말하지 말 것 — 설계근거일 뿐이다.
- "중단 후 복귀 23분 15초"(Gloria Mark) — 오귀속된 수치, 아예 입 밖에 내지 않는다.
- 에이전트 출력(판단문·초안 텍스트)이 전부 실 LLM이라고 말하지 말 것 — 현재는 JS 템플릿 목업이며, 히어로 1케이스만 실 LLM화가 목표다.
- "3도메인 모두 JB우리캐피탈로 확장"이라 말하지 말 것 — 전세는 캐피탈 사업영역 밖이다.
