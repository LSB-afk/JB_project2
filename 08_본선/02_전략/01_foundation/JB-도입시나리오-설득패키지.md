## 🖼 슬라이드 1 — "이렇게 연결되고, 이렇게 적용됩니다"
> 이 한 장이 시스템연결·데이터·메모리·외부/로컬모델·보안·확장을 **동시에** 답한다.

```
┌─ 외부·공공 레인 ────────────────────────────────────────┐
│  등기정보광장 · 보증기관 · 금융결제원 · 뉴스 · 정책 API   │
│         │  [반입 게이트] 분류·라이선스·캐싱                │
│  외부 LLM(Claude/Codex) ← 토큰·가명·최소맥락만 ↑          │
│                            원본 PII 반출 ✗                │
╞══════════ 🔒 은행 보안경계 (망분리) ══════════════════════╡
│  LocalGuard OS  ─ 로컬모델 + 판단→행동초안→검증 에이전트    │
│                 ─ 고객 메모리 = Case 360 (Zero-PII)       │
│         ▲ 읽기(CDC Read-only)          ▼ 쓰기(승인 후만)   │
│  내부 은행 시스템: 코어뱅킹·여신·CRM·EWS                   │
│         │  토큰화 프록시                                   │
│  담당자 승인 게이트 → Audit ledger(감사기록)              │
└──────────────────────────────────────────────────────────┘
   확장: 계열사 허브 = 토큰·위험코드·근거·승인상태만 통합
         (원장/PII 통합이 아니라 "신호"만 통합)
```

흐름 한 줄: **외부신호 + 내부신호 → Case → 로컬 판단 → 행동초안 → 담당자 승인 → (승인 후에만) 실행·기록.**

## 📥 슬라이드 2 — 외부 도구·공공데이터, 안전하게 가져오기
> "밖에서 뭘 끌어오면 위험하지 않냐"에 대한 정답. 근거 [[P1-데이터-연동기반]]

| 레인 | 예시 | 반입 규칙 | 활용 |
|---|---|---|---|
| **공개형(즉시)** | 등기정보광장·정책·뉴스 공개 API | 라이선스·캐싱 게이트 통과 | 자유 활용 |
| **승인형(제휴필요)** | 보증기관·금융결제원 한도/약관 | 한도·약관 확인 과제로 격리 | 계약 후 |
| **계약형/상용(반출제약)** | CB·카드매출·사기방지 데이터 | 재배포·반출 제약표 준수 | 내부만·외부전송금지 |

핵심 원칙 3개: **들어올 때 분류·라이선스 게이트 / 나갈 때 토큰만(원본 PII 반출 금지) / 은행 시스템은 읽기만, 쓰기는 승인 후 MCI·EAI로만.** 코어뱅킹 직접 쓰기는 원천 제외.

⚖️ **법령 근거([[D25-결과]])**: "공개형 = 비개인정보"가 아니다(등기·실거래가에 식별요소 섞이면 PIPA 적용). 원본 개인신용정보 **외부 LLM 반출은 금지에 가깝다** — 2026.4.20 금융권 SaaS 예외도 **"고유식별정보·개인신용정보는 망분리 예외 불허"**, 생성형 AI는 2026.5 기준 **샌드박스·연내 제도화 중**. → "레인 분리"가 곧 준법 설계.

## 🛡 슬라이드 3 — 보안 리스크 → 통제 → 증거 (은행은 여기서 산다)
> 과장 금지. "유출 전무"❌ / "원본 PII 비반출 + 다층통제로 실질 저감"⭕. 근거 [[P3-보안-거버넌스]] · [[D5b-결과]] · [[연결-3]]

| 리스크 | 통제 | 증거(시연·테스트) |
|---|---|---|
| 외부 LLM에 원본 PII 유출 | 토큰화 프록시 · 반출 DLP 스캔 · 모델 라우팅 게이트 · 망분리 | 감사로그 · synthetic PII leak 0/N 테스트 |
| AI 오판·오실행 | 승인우선 L0–L4 · 승인 전 고위험 action 차단 · fail-closed | 승인로그 · 이유코드 · kill switch |
| 프롬프트 인젝션·재식별·캐시유출 | 반출스캔 · 캐시 정책 · 재식별 위험 스캔 | 스캔 리포트 · 감사 |
| 코어뱅킹 오염 | 직접쓰기 제외 · CDC Read-only · 승인 후 MCI/EAI Write | 연동 경계표 |
| 계열사 데이터 오남용 | 식별형 내부경영관리 / 가명분석 / 클린룸 분리 · 토큰만 통합 | 데이터공동이용 옵션표 |

⚖️ **법령 근거([[D25-결과]])** — 문구가 아니라 조문에 걸린 설계:
- **외부 반출**: 원본 PII/개인신용정보 반출 금지근접(신용정보법 §32·§33·§17 위탁, PIPA). **토큰화 ≠ 익명화**(내부 키 복원가능 → 가명/개인정보) → "Zero-PII = 무규제"❌ / **"반출·위험 최소화 설계"⭕**.
- **계열사 이식**: 금융지주회사법 **§48조의2** → 가능하되 **내부경영관리 목적 한정(영업권유 제외)·원장 제공 금지·분리보관·이용기간 1개월·고객정보관리인 승인·연1회 통지**. → 3단계(원천 식별형 / 그룹 가명형 결합 / 외부 익명·Zero-PII).
- **승인우선 = 법적 방어**: 자동화결정 거부·설명요구권(PIPA)·자동화평가 이의(신용정보법)·금융위 AI가이드 "보조수단·인적개입".
- **보존**: 신용정보 처리기록 3년 / 접속기록 1~2년 / 중요원장 5년. RM 메모·프롬프트 중간산출은 목적달성 즉시 파기.

## 🧠 슬라이드 4 — 외부 vs 로컬 모델 운영 (모델 주권)
> "은행이 외부 AI 꺼린다"의 해소. 근거 [[D22-결과]] · [[P5-통합-로컬모델-시연]]

- **로컬 우선, 통제 승격**: 원본·판단은 로컬모델이 은행 안에서 실행. 외부 LLM은 토큰·가명 맥락만 받아 보조.
- **철학**: "하네스가 좋으면 모델은 그 안에서 잘 작동한다" — 초거대 모델이 아니라 운영계약이 본체.
- **시연 포인트**: 히어로 도메인 판단 단계를 **M4 Pro 로컬모델로 실제로 돌려** "밖으로 안 나간다"를 체감시킨다. (로컬스택 = 아래 빌드 청사진)

## 🏗 슬라이드 5 — 어떻게 만드나: "히어로 1개는 풀 리얼, 나머지는 목업"
> paperclip처럼 **진짜 도는 완성도**를 확보하되 리스크는 목업 폴백으로 관리. 히어로 = **보이스피싱 `[잠정]`**.

| 층 | 히어로 도메인 (풀 리얼) | 나머지 도메인 |
|---|---|---|
| 데이터 | 실 공개 API 1개 (확보되는 것) | 목업 fixture |
| 모델 | **로컬모델 실추론** (판단 단계 진짜 실행) | 스크립트/캐시 응답 |

| UI | JB 디자인 풀 적용 (디자이너 PR `codex/jb-uxwriting-20260702`) | 동일 셸 |

## 🔍 슬라이드 8 — 시장분석
> 근거 [[D30-결과]](GPT-5.5 DR, 산정식·출처 노출) · 보조 [[Dplus-a-결과]] · [[D23-결과-gpt55pro]]. 모든 수치 **보수·범위**.

**시장은 실재한다 (보수 추정)**
| 층 | 값 | 근거 |
|---|---|---|
| **TAM** | 1,100~2,200억원 | 국내 AI시장(2026e 3.9조)×금융비중 20%+×운영콘솔 비중 ∩ IT예산 9.44조×GRC 1~2% |
| **SAM** | 110~200억원 (제품형 ARR) | 지방은행 6·저축은행 79·카드 8·리스할부 50 bottom-up seat |
| **SOM** | 4.6~13.5억원 (기준 ~8억) | JB 2계열사 초기 seat 150~350석 |

**빈칸(white space)이 있다** — 원본 PII 내부처리 + 승인우선 + 계열사 이식 + 지역은행/여신 특화 + 감사증적 내장을 **동시에** 제품 기본값으로 주는 포지션은 공개시장 부재. (Palantir=무겁고 도메인계약 별도 / 국내 SI=프로젝트형·재사용성 낮음 / GRC·AML=현업 행동초안 약함 / 내부 RPA=파편화·이식성 낮음)

**⭐ 왜 지금 — 시장이 이미 우리 패턴을 산다**
- **은행연합회-업스테이지 AI 광고심의(2026)**: AI가 보고서, **최종 판정은 사람** = 우리 승인우선 패턴이 **이미 제도권에서 조달**됨. (자동응답이 아니라 승인·감사 내장형 AI 조달 시작)
- **삼성SDS-우리은행 90개 에이전트**: 대형사도 챗봇→에이전트·내부통제로 이동. 단 **프로젝트형** → 우리는 **제품형 운영 콘솔**로 차별.
- **금융위 2026.4 절차 간소화**(단순 모델변경 서면확인 즉시출시) + 망분리 완화 → "도입장벽↓, 책임은 **운영거버넌스**로 이동".
- **진입 방식**: 케이뱅크=KT+kt cloud+업스테이지 **3자 컨소시엄** → JB 진입도 **보안·인프라 파트너 붙인 형태**가 성공확률↑.

**진입 서사(GTM)**: ① 전북은행 RM·여신·사후관리 + JB우리캐피탈 여신 사후관리로 beachhead(KPI 3개: 검토시간↓·승인전 오류/누락↓·감사추적성↑) → ② **공동 코어 + 계열사별 정책팩**으로 룰셋 이식 → ③ 광고심의·내부통제·이상징후로 확장 = 그룹 공통 통제 SaaS. 초기 8억 SOM → 그룹 상향.

⚠️ 심사 주의: "지역은행" 정의 **4개 지방은행 vs 6개 지역기반 은행** 혼용 → 자료에 정의 병기. TAM/SAM은 직접 시장통계 아닌 **파생 추정**임을 명시(과장 방어).
