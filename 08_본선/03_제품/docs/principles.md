# 03 Principles — 제품 원칙

> `principles-definitions` 스킬(Principle Test: "no라고 말할 수 있어야 원칙") 적용. 각 원칙은 **Enforcement(어떻게 강제)** + **근거등급(E0~E5)** + **한계(못 막는 것)** 를 함께 명시한다. 근거: `_canon.md` §0·§2·§4·§8, [[08_본선/03_제품/00_vision/core-bet|core-bet]], [[08_본선/03_제품/00_vision/차별성-경험레이어-서사|차별성-경험레이어-서사]], [[08_본선/03_제품/01_결정-준비/키스톤-확정|키스톤-확정]], [[08_본선/03_제품/01_결정-준비/근거팩/적법성-근거팩|적법성-근거팩]].

## 근거등급 스케일

| 등급 | 의미 |
|---|---|
| E0 | 가정·[미검증] |
| E1 | 약근거(정황·유추) |
| E2 | 문헌·설계 정합(리서치 요약 기반) |
| E3 | 규제 조문·1차 실증 문헌 |
| E4 | **데모로 실증 가능**(콘솔 코드/화면으로 시연) |
| E5 | 현장 PoC 검증(미보유) |

목표는 각 원칙 E2 이상, 데모 대상 원칙은 E4.

## Product Principles (6)

| ID | Principle | Enforcement | 등급 |
|---|---|---|---|
| P-001 | **사용자 가치·경험 우선** — AI는 기능을 자동화하는 게 아니라 직원이 사람답게 판단할 여유를 만든다. 자동화의 대상은 기능이 아니라 경험(직원·조직·그룹·고객). | 반복 행정(대조·초안·기록)은 Agent가, 최종 판단·고객 설명·승인은 사람이 맡도록 역할 기반 게이트로 흐름을 재구성한다. Demo-critical path(CCL-0001 전주 카페 운전자금)부터 완결 시연한다. | E2 |
| P-002 | **근거 우선·설명가능** — 근거(Evidence) 없는 판단은 노출하지 않는다. | 모든 AgentRun 산출물은 인용 신호(Evidence)를 동반해야 하며, Evidence·불확실성을 숨긴 추천 카드는 승인 화면에 노출되지 않는다. 카드는 `왜 지금 떴는지·근거 원문·규정 근거·불확실성·대안 행동` 순으로 구성(D18). 신용정보법 §36조의2 자동화평가 설명요구권 대응. | E4 |
| P-003 | **사람 승인 게이트(고위험)** — 사람 승인 없이는 고객 접촉·계좌/계약 변경을 실행하지 않는다. AI는 보조수단이며 최종책임은 임직원에게 있다. | 승인 레벨 L0~L4(`approvalLevelMatrix`, `app.js`) — L2+ 고객 통지는 RM 편집/승인, L3+ 는 RM+준법 2역할 승인 필수. 저위험=빠른 승인, 중위험=사유 선택, 고위험=원문확인+이중확인+발송 preview(D18). AI=권고 / 심사역=실무책임 / 센터장=최종책임(금융위 2026 가이드라인 RACI, AI기본법 §34 고영향 사람관리·감독). | E4 |
| P-004 | **불확실하면 닫는다(Fail-closed)** — 정책 규칙·근거가 없으면 진행하지 않는다. | 정책 규칙(policy rules) 없이 생성된 행동 초안은 승인 대기열로 넘기지 않고 차단. Approval Gate는 fail-closed(승인 전 발송 원천 차단). 신용/법률/계약상 확정 판단은 AI가 단정하지 않고 보류로 라우팅. | E4 |
| P-005 | **데이터 최소화·PII 비반출** — 원본 고객 PII·개인신용정보를 외부 프런티어 LLM에 반출하지 않는다. | PII 4중 방어(데이터등급제·토큰화·모델 라우팅·반출 스캔·감사원장). 레인 분리(D25): 공개·집계형→외부 LLM, 원본 PII·개인신용정보→내부 레인, 내부 생성물→Zero-PII 파생. 외부에는 토큰·위험코드·근거 포인터·승인상태·모델버전만 전송. 신용정보법 §40조의2 ①②⑥⑦⑧(가명처리·재식별금지), 전자금융감독규정 §15조(망분리 예외 요건)[^1]. | E4 |
| P-006 | **감사가능성(Auditability)** — 모든 상태 전환은 감사기록으로 남는다. | Case→AgentRun→Agent→Skill→Evidence→Approval 전이마다 audit chain(`auditChainRecords`) 레코드 생성 — 누락 시 해당 전이 자체가 무효. append-only·해시/전자서명·변경 히스토리·forensic export를 목표 형태로 둔다(D15). | E4 |

[^1]: 전자금융감독규정 §15조 ①항 3·5호는 [미검증] — 법령 개정 진행 중이므로 제출 전 law.go.kr 현행 항·호 번호 재확인 필요(canon §4).

> **스키마 필수원칙 커버리지**: User value first→P-001, Demo-critical path first→P-001, Evidence over claims→P-002, Explainability→P-002, Human approval for high-risk→P-003, Fail closed on uncertainty→P-004, Data minimization→P-005, Auditability→P-006. 8개 필수 항목 전부 6원칙에 매핑됨.

## Reject-If 규칙 (출력 거부 조건)

- **[REJECT] 무단 실행** — 승인 없이 고객에게 직접 발송·접촉·계좌변경을 실행하는 출력.
- **[REJECT] 근거 은닉** — Evidence·confidence(신뢰도)를 숨기거나 생략한 출력.
- **[REJECT] AI 단정** — 법률·신용·계약상 확정 판단(전세사기 확정, 대출 승인/거절 확정, 금리·한도 확정)을 AI가 단정하는 출력.
- **[REJECT] PII 반출** — 원본 PII(주민번호·주소·계좌·상담원문 등 재식별 가능 정보)가 외부 LLM 호출 페이로드에 포함 → 반출 스캔에서 즉시 차단·회수.
- **[REJECT] 정책 부재 초안** — policy rules 없이 생성된 action draft를 승인 대기열로 승격.
- **[REJECT] 실데이터 데모** — 데모·발표·제출물에 실제 고객 데이터 사용(히어로 CCL-0001 포함 전부 synthetic만 허용).

## 한계 — 이 원칙들이 못 하는 것 (정직하게)

리서치는 원칙의 방어력만큼 **원칙의 구멍**도 가리킨다. 심사위원 반문 대비.

| 반문 | 정직한 답 | 출처 |
|---|---|---|
| "승인 게이트 붙였으니 안전한가?" | 아니다. 사람 승인만으론 **rubber-stamping·입력 데이터 오염·벤더 유출·경보 무시·kill switch 미연결·승인 대기 병목**을 못 막는다. 그래서 P-003을 Reject-If 사유코드 강제·이중승인·kill switch(채널/배치/서드파티 API까지 차단)·immutable evidence로 **구조화**한다. Knight Capital(45분 4.6억달러), Wells Fargo(100만 계좌 오동결)가 "승인 없는/구조 없는 자동화"의 실패 사례. | D15 |
| "그럼 생산성은 얼마나 오르나?" | 단정 못 한다. 생성형 AI 효과는 과업·숙련도에 따라 크게 갈리고(글쓰기·CS는 +, 숙련 개발자 복잡작업은 완료시간 +19%), **속도만 재면 거의 틀린다**. 리드타임·품질·faithfulness·재작업률을 함께 재야 한다. 한국 금융 RM 직접 RCT 근거는 [미검증]. | D8 |
| "설명을 많이 달면 채택률 오르나?" | 아니다. 그럴듯한 설명은 AI가 틀려도 수용을 늘린다(rubber-stamping ↑). **설명량보다 검증 구조**(수정 가능한 제안·근거 원문·override taxonomy)가 먼저. | D18 |
| "토큰화하면 외부 반출 자유인가?" | 아니다. 토큰화·Zero-PII도 **복원 키가 내부에 있거나 재식별 가능하면** 여전히 가명정보·개인정보 평가 대상. 국외이전(리전·서브프로세서) 쟁점도 남는다. | D25 |
| "감사원장 있으면 법적 증거인가?" | 로그가 있다는 것만으론 부족하다. **무결성·동일성·chain-of-custody** 입증이 있어야 증거가 된다. 감사원장은 사후 추적엔 강하나 개발·학습 단계 데이터 경계가 무너지면 선제 예방엔 한계. | D15 |

> [Open Question] 승인 직군(RM·준법 2역할) 유지 vs 확대는 7/4 팀 확정 대기 — Enforcement의 승인 오너 확정 전까지 최선 가정.
