# Data Strategy — JB LocalGuard OS

> [미결/7-4] 도메인 조합·데모②·데이터 제공 부서 확인(최영욱)은 7/4 팀 확정 전까지 최선 가정으로 채웠다. 근거층(리서치 요약 D5a·D6·D7a·D7b·D17·D25)은 '왜?'와 조달·경계 설계에만 인용했고, 백본 사실·수치·이름은 canon·키스톤·business-model을 따른다.

## 근거등급 표기 (E0~E5)
| 등급 | 의미 |
|---|---|
| E5 | 법령·감독규정 원문(근거팩 표현 그대로) |
| E4 | 데모에서 실동작으로 보여줄 수 있음 |
| E3 | 다수 외부출처/문헌 정합 |
| E2 | 단일 외부출처 확인 |

목표는 핵심 주장 E2+, 데모 주장 E4.

## Judge-Facing Thesis — 데이터가 해자다
JB LocalGuard OS의 데이터 우위는 "더 많은 데이터"가 아니라 **원본을 밖으로 내보내지 않고도 외부 LLM의 추론력을 쓰는 경계 설계**에서 나온다. 고객 원본 PII·개인신용정보는 계열사 내부 레인에만 두고, 외부 프런티어 LLM에는 토큰·위험코드·근거 포인터·비식별 요약만 흐른다. 이 경계는 데이터 등급제·토큰화·모델 라우팅·반출 스캔·감사원장의 **4중 방어**(canon §0)로 구현되며, 신용정보법 §40조의2가 요구하는 통제를 설계 시점부터 만족시킨다(E5). 범용 LLM 서비스는 이 경계를 사후에 붙이기 어렵고, 규제 준수 자체가 경쟁자가 단기 복제하기 어려운 진입장벽이 된다(E1).

## 데이터 등급제 (5단계)
비식별 강도에 따라 등급을 매기고, 등급이 AI 입력 가능 여부와 라우팅 대상을 결정한다(D5a, E2).

| 등급 | 예시 | 저장 위치 | 외부 LLM |
|---|---|---|---|
| G0 원본 개인신용정보 | CB 스코어 원값, 계좌·카드번호, 상담 녹취 원문 | 계열사 내부 실명 레인(Vault) | **불가** |
| G1 가명정보 | 토큰화 식별자, 구간화 신용등급, 폐업확률 구간 | 계열사 내부 가명 레인 | 조건부(추가정보 분리 전제) |
| G2 익명·집계 | 권역 평균, 상권 통계, 셀 억제 후 집계 | 그룹 공통/클린룸 | 조건부 |
| G3 공개·공공 | law.go.kr 법령, MOLIT 실거래가, ECOS 금리 | 자유 | 가능(라이선스·식별요소 재확인 후) |
| G4 내부 생성물(Zero-PII 파생) | 위험코드, 근거 포인터, 승인상태, 모델 버전 | Zero-PII 파생 레인 | **가능** |

> ⚠️ G4/토큰화라도 복원 키가 내부에 있으면 대체로 가명정보 — "무규제"가 아니다(D25, E3). 공개 데이터도 자연인 식별요소 혼입 여부를 다시 본다.

## 데이터 레인 4단 분리
같은 그룹이라도 원본을 섞지 않는다. 읽기/쓰기 경계를 레인으로 고정한다(D17·D6, E2 — 내부 연동 스펙 비공개로 일부 [미검증]).

| 레인 | 담는 것 | 안 담는 것 | 흐름 |
|---|---|---|---|
| L-실명(계열사 내부) | 원본 원장, 원본 식별자, 상담 원문 | — | 계정계 로그 → CDC → 내부 가명처리 → Kafka |
| L-가명(그룹 공통 Hub) | `subject_token, affiliate_id, domain, risk_code, score_band, evidence_pointer, event_time_bucket, approval_state` 최소필드 | 원본 원장·원본 식별자 | 계열사 토큰화 게이트웨이 경유 |
| L-결합(전문기관) | 행 단위 결합·외부 공공데이터 결합 | Case Hub와 혼용 금지 | 데이터전문기관/결합전문기관 레인 |
| L-외부(비식별 요약) | draft용 비식별 요약, 토큰·위험코드·근거 포인터 | 이름·주민번호·계좌·주소·녹취 | 반출 스캔 통과 후에만 외부 LLM |

**쓰기 경로**(고객 대상 행동): `AI 권고 → RM 승인 클릭 → (Nexacro 런타임) → MCI/EAI 표준 전문 → 계정계/발송계`. AI가 직접 Write하지 않는다(D6, E2).

## 4중 방어 ↔ 등급·레인 매핑
| 방어층 | 구현 | 등급/레인 연결 | 등급 |
|---|---|---|---|
| ① 데이터 등급제 | G0~G4 분류, AI 입력 가능/불가 게이팅 | 위 등급표 | E4(데모 표시 가능) |
| ② 토큰화 | 계열사 토큰화 게이트웨이, 키 분리보관 | L-실명 → L-가명 | E4 |
| ③ 모델 라우팅 | 원본=로컬 LLM, 비식별=외부 LLM | L-외부 진입 게이트 | E4 |
| ④ 반출 스캔 | 출력 DLP(주민번호·계좌·주소·희소조합) | L-외부 직전 | E4 |
| ⑤ 감사원장 | 반출프록시·DLP·토큰화·승인·모델/리전 로그 묶음, 3년 보존 | 전 레인 횡단 | E5(보존기간 법정) |

## 데이터 조달 3층
조달 방식이 다르면 라이선스·리드타임·MVP 편입 여부가 달라진다(D7a·D7b, E2).

| 층 | 성격 | 예시 | MVP 편입 |
|---|---|---|---|
| ① 무상 공개형 API | 별도 동의 불요, 즉시 | ECOS(금리·경기), RTMS 실거래가, 상권정보, 국세청 사업자상태조회, 행안부 인허가, 경찰청·KISA 통계, law.go.kr | **즉시 편입** |
| ② 유료 건별 조회 | 건당 과금, 원문 오래 쥐지 않음 | 인터넷등기소 등기 열람 700원/발급 1,000원, CRETOP(10건 7,000원~) | 편입(정규화 피처만 잔존) |
| ③ 기관계약형 연동 | 이용계약·이용승인 필요, 후속 트랙 | 금결원 오픈뱅킹, NICE/KCB BizAPI, 카드매출, 더치트 | **후속 제휴 트랙**(MVP는 Fallback) |

- 국세청 상태조회: `1회 100건 / 1일 100만건 / 30분 주기` 전제로 캐시·재조회 룰(D7a, E2).
- 등기: 원문 PDF 대신 `소유자일치·선순위권리 총액·근저당 최고채권액·가압류·말소기준권리` 정규화 피처만 저장(D7a, E2/E4 데모).
- 상용 대체데이터: 원문 응답 대신 요약 지표만 LLM 주입 — 카드매출은 `Sales Drop 24.5% vs quarterly avg`, 더치트는 `사기 피해 등록 유무` 플래그, CB는 `부도확률 구간`(D7b, E2).

## 원장 비접촉 연동 (CDC / MCI)
- 읽기: 운영 DB 직접조회·계정계 API 추가는 피하고 `Debezium 같은 로그 기반 CDC`로 고정(OLTP 부하·원장 가용성 이유). PII 경계는 은행 내부 Tokenization Proxy에 둔다(D6, E2).
- 전북은행·광주은행은 통합 전산센터 계획 철회 → **배포·연동·보안심사 이원화**(별도 호스트/프로젝트)(D6, E2).
- [미검증] 넥사크로 WebView 엔진 버전 호환성, MCI/EAI 표준 전문 스키마·암호화 규칙(비공개).

## Zero-PII 메모리·보존정책
- 고객 메모리는 원장 통합이 아니라 **Zero-PII 메모리** — 저장 대상은 `조치 이력·승인 이력·설명 생성 근거`까지로 제한(D25, E3).
- 보존정책(D25, E5):
  - 개인신용정보 수집·이용·제공·폐기 기록 **3년**
  - 개인정보처리시스템 접속기록 **1년 또는 2년**
  - 전자금융거래기록·중요원장 직접기록 **5년**
  - RM 초안·중간 프롬프트 = **단기 파기 원칙**
- [미검증] 최소 샘플 수·셀 억제 기준·재식별 허용치·프롬프트 차단 룰 등 정량 반출 기준은 JB 내부 별도 제정 필요(D17).

## 반출 차단·감사
- 출력단 반출 차단기: 주민번호·계좌식별자·주소 조합·희소 속성 조합 스캔(D5a, E4 데모 가능).
- 감사원장 기록: `가명처리 규칙·외부모델 라우트·승인·접근자·처리 목적/방법/일시·스캔 결과`, 3년 이상 보존(D5a, E5).
- 국외이전 쟁점: 외부 LLM은 리전·서브프로세서·원격지원·로그 저장 위치까지 계약 명시(D25, E3). [Open Question] JB 외부 LLM 계약 조건 공개자료로 미확인(D17).
