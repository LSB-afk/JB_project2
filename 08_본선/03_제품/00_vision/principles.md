# 03 Principles — 제품 원칙

> `principles-definitions` 스킬(Principle Test: "no라고 말할 수 있어야 원칙") 적용. 근거: `_canon.md` §2·§4·§8, [[08_본선/03_제품/00_vision/core-bet|core-bet]], [[08_본선/03_제품/00_결정-준비/키스톤-확정|키스톤-확정]], [[08_본선/03_제품/00_결정-준비/근거팩/적법성-근거팩|적법성-근거팩]].

## Product Principles

| ID | Principle | Enforcement |
|---|---|---|
| P-001 | 원본 고객 PII를 외부 프런티어 LLM에 반출하지 않는다 | PII 4중 방어(데이터등급제·토큰화·모델 라우팅·반출 스캔·감사원장) — 로컬(EXAONE/Qwen2.5)=PII 업무, 외부(Claude/Codex API)=비민감 요청만. 신용정보법 §40조의2 ①②⑥⑦⑧(가명처리·재식별금지), 전자금융감독규정 §15조(망분리 예외 요건)[^1] |
| P-002 | 사람 승인 없이는 고객 접촉·계좌/계약 변경을 실행하지 않는다 | 승인 레벨 L0~L4(`approvalLevelMatrix`, `app.js`) — L2 이상 고객 통지는 RM 편집/승인, L3 이상은 RM+준법 2역할 승인 필수(canon §2·§8) |
| P-003 | 근거(Evidence) 없는 판단은 차단한다 | 모든 AgentRun 산출물은 인용 신호(Evidence)를 동반해야 하며, Evidence 없는 추천 카드는 승인 화면에 노출되지 않는다 |

> 별도 조문 병기: 신용정보법 **§36조의2**(자동화평가 결과에 대한 설명요구권·이의제기, §40조의2와는 별개 조문)는 P-001(PII 비반출)·P-003(근거 있는 판단) 두 원칙의 법적 근거를 보강한다(적법성-근거팩).

[^1]: 전자금융감독규정 §15조 ①항 3·5호는 [미검증] — 법령 개정 진행 중이므로 제출 전 law.go.kr 현행 항·호 번호 재확인 필요(canon §4).
| P-004 | 모든 상태 전환은 감사기록으로 남는다 | Case→AgentRun→Agent→Skill→Evidence→Approval 전이마다 audit chain(`auditChainRecords`) 레코드 생성 — 누락 시 해당 전이 자체가 무효 |
| P-005 | AI는 보조수단이며 최종책임은 임직원에게 있다 | 신용평가·대출승인·금리/한도 확정을 AI가 대신하지 않는다(AI=권고, 심사역=실무책임, 센터장=최종책임 — 금융위 2026 가이드라인 RACI, AI기본법 §34 고영향 사람관리·감독) |
| P-006 | 데모·공개 자료는 합성 데이터만 사용한다 | 실제 고객 데이터는 데모·발표·제출물 어디에도 포함하지 않는다(히어로 케이스 `JBG-104` 포함 전부 synthetic) |

## Rejection Rules

- 승인 없이 고객에게 직접 발송·접촉·계좌변경을 실행하는 출력은 거부한다.
- 근거(Evidence)나 신뢰도(confidence)를 숨기거나 생략한 출력은 거부한다.
- 법률·신용·계약상 확정 판단(전세사기 확정, 대출 승인/거절 확정, 금리·한도 확정)을 AI가 단정하는 출력은 거부한다.
- 원본 PII(주민번호·주소·계좌 등 재식별 가능 정보)가 외부 LLM 호출 페이로드에 포함된 경우 반출 스캔에서 즉시 차단·회수한다.
- 정책 규칙(policy rules)이 없는 상태에서 생성된 행동 초안(action draft)은 승인 대기열로 넘기지 않고 차단한다.
