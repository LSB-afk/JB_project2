# Core Bet — 핵심 베팅

> CPS(Context→Problem→Solution→Non-Solutions→Success Evidence) 노멀폼. 근거: `_canon.md`, [[08_본선/03_제품/00_결정-준비/키스톤-확정|키스톤-확정]], [[08_본선/03_제품/00_결정-준비/키스톤-역할축-검증|키스톤-역할축-검증]], [[08_본선/03_제품/00_결정-준비/결정-현황-종합|결정-현황-종합]], [[08_본선/03_제품/00_결정-준비/근거팩/ROI-근거팩|ROI-근거팩]].

## 우리가 베팅하는 것

- **씨앗**: 지역 금융 담당자는 하루에 수십 건의 이상 신호를 육안으로 검토한다 — 이 중 AI가 1차 분류·초안을 만들면 담당자는 최종 판단에만 집중할 수 있다
- **씨앗**: 기존 알림 시스템은 "보여준다"에 그치지만, LocalGuard는 "행동 초안까지 만든다"
- **씨앗**: 사람 승인 없이는 고객에게 아무것도 발송하지 않는다 — 이것이 금융권이 AI를 신뢰할 수 있는 유일한 조건

---

## Context

- JB금융그룹은 2026년을 **"AX(AI 전환) 원년"**으로 선언(김기홍 회장, canon §10)했고, 2025-12-26 **네이버클라우드 MOU**로 기업여신 상담→심사→**사후관리** 전반의 AI 적용을 단계적으로 검토 중이다. [검증됨]
- 콘솔의 조직 축(키스톤)은 이미 팀이 구현 중인 방향과 일치한다 — **계열사(전북은행/JB우리캐피탈) × 담당 직군**. 실측 직군은 RM·여신심사·사후관리·준법·AML 5개이고, 콘솔이 구현한 승인자는 **RM + 준법 2역할**([[08_본선/03_제품/00_결정-준비/키스톤-확정|키스톤-확정]], `app.js` `approvalLevelMatrix`, canon §2). [검증됨]
- 전북은행 RM 업무는 상담→서류징구/대조→신용평가·품의→심사 handoff→**사후관리/EWS/연체/FDS**→기록보관까지 여신 라이프사이클 전체를 포괄한다([[08_본선/03_제품/00_결정-준비/근거팩/ROI-근거팩|ROI-근거팩]] D16 baseline). [검증됨]
- 전북은행은 소상공인 대환대출 취급 **1위**(2023~24 누계 1,788억·점유율 44.7%, 2년 연속), FY2025 그룹 순익 **7,104억(사상 최대)**을 보유한다(canon §10). [검증됨]
- 규제 환경은 "외부 LLM 활용 + 원본 PII 비반출"을 이미 정책적으로 지지한다 — 신용정보법 §40조의2(가명처리·분리보관·재식별금지)·§36조의2, 전자금융감독규정 §15조(망분리), 금융위 2026 「금융분야 AI 가이드라인」(보조수단성·RACI: AI=권고/심사역=실무책임/센터장=최종책임), AI기본법(대출심사=고영향, §34 사람관리·감독)(canon §4). [검증됨]

## Problem

**히어로 실례 — 전주 중앙로 카페(코드 `JBG-104`, 전북은행, 개인사업자, riskScore 88, 상태 `Approval Pending`(urgent), 노출 신호: 운전자금 1.8억·카드매출 둔화, 핵심 pain: 금리 부담+매출 둔화, 데모 `?demo=sme`).**

- 이 케이스를 담당하는 RM은 계정 행동, 서류 상태, 정책 제약, 공개 시세 등 흩어진 신호를 여러 시스템에서 직접 모아야 하고, 반복 행정(대조·기록·handoff 재작업)에 케이스당 10~30분이 소모된다([[08_본선/03_제품/00_결정-준비/근거팩/ROI-근거팩|ROI-근거팩]] baseline, `[추정]`). 이 시간이 회수되지 않으면 RM은 정작 필요한 최종 판단(승인·조건 조정)에 쓸 여력을 잃는다.
- 놓치면 발생하는 비용: 취약 자영업자 42.7만명(13.7%), 2024년 폐업 100.8만명(첫 100만 돌파, 사업부진 사유 50.2%, canon §10) — 전주 카페 같은 케이스가 정책자금 매칭 기회를 놓치면 이 통계에 편입될 위험으로 이어진다. [검증됨]
- 같은 구조적 병목이 **사후관리**에서 이미 규제 리스크로 현실화됐다 — 2026-06-30 시행 "자금용도외 유용 사후점검준칙" 개정으로 의무 점검 대상이 하향(개인사업자 1억→5,000만원 등)되어 케이스가 폭증하는데, RM 인력은 오히려 축소 추세다([[08_본선/03_제품/00_결정-준비/키스톤-역할축-검증|키스톤-역할축-검증]] §"사후관리"). "인력은 줄고 의무 케이스는 폭증"이 이미 발생 중인 페인이다. [검증됨, 캐피탈 적용성은 [미결/7-4]]
- 일반 AI 챗봇으로는 이 문제를 풀 수 없다 — 근거 없는 블랙박스 위험점수, 승인 게이트·감사로그 부재는 금융권 실도입에서 규제 위반·책임소재 불명으로 직결된다(심사기준 5.5).

## Solution

- **운영 메커니즘**: `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`(canon §8). 콘솔 축 = 계열사×담당 직군, 도메인(여신·전세보호·피싱·사후관리)은 그 직군이 처리하는 **케이스**다([[08_본선/03_제품/00_결정-준비/키스톤-확정|키스톤-확정]]).
- 14개 전문 에이전트가 신호 수집→위험 분류→행동 초안 작성→검증을 수행하되, **고객 대상 행동은 승인 레벨 L0~L4를 통과하기 전까지 자동 실행되지 않는다**(RM 최종 승인자·준법 최종 승인자 2역할 + 승인 게이트, canon §2·§8). [검증됨]
- **PII 4중 방어**(데이터등급제·토큰화·모델 라우팅·반출 스캔·감사원장)로 원본 고객 PII를 외부 프런티어 LLM에 반출하지 않는다. 하이브리드 라우팅: **웹 = 정책/승인 UI + Claude/Codex API(비민감 요청)**, **로컬(EXAONE 3.5 7.8B 데모, Qwen2.5 실배포 권고) = PII 포함 업무**([[08_본선/03_제품/00_결정-준비/결정-현황-종합|결정-현황-종합]]). [검증됨]
- 계열사별로 직군·도메인 구성이 다르게 모듈화된다 — 전북은행=여신(RM)·전세보호·피싱, JB우리캐피탈=여신·사후관리(EWS). 공통 뼈대는 한 번만 만들고 직군별 에이전트셋·화면·데이터만 스왑한다([[08_본선/03_제품/00_결정-준비/키스톤-확정|키스톤-확정]] "하이브리드(옵션3)").

## Non-Solutions

- 승인 없는 자동 고객 접촉·발송 없음 (fraud 케이스는 전 구간 승인 필수).
- 원본 고객 PII를 외부 LLM에 반출하지 않음 — 데모·공개 자료에도 실제 고객 데이터를 쓰지 않는다.
- 신용평가·대출 승인·금리/한도 확정 판단을 AI가 대신하지 않는다 — AI는 권고, 심사역이 실무책임, 센터장이 최종책임(AI기본법 §34, 금융위 2026 가이드라인 RACI, canon §4).
- 법률·규정 확정 판단, 전세사기/피해자결정/보증가입 확정 등 확정적 단정을 AI가 내리지 않는다.
- 로컬모델 채택의 명분을 "비용 절감"으로 세우지 않는다 — TCO 모델링상 JB 규모에서는 API가 로컬보다 저렴할 수 있다(손익분기점 월 수십억~약 97억 토큰). 로컬모델의 실제 근거는 **원본 PII 비반출·규제 준수**다([[08_본선/03_제품/00_결정-준비/결정-현황-종합|결정-현황-종합]] "로컬모델 채택 명분").

## Success Evidence

- KPI(canon §3): Triage time **50% 단축**, Evidence traceability **100%**, Approval safety **100%**, 사후관리 누락 **0건** 목표, Jeonse safe-contract **100%**.
- 히어로 케이스(전주 중앙로 카페 `JBG-104`)가 실 LLM 에이전트로 판단→근거→승인→감사 전 과정을 동작시킨다 — 7/4 개발 목표, 나머지 골든패스(전세보호/피싱)는 실동작 지향 [조건부/7-4]([[08_본선/03_제품/00_결정-준비/키스톤-확정|키스톤-확정]] "정직한 전제").
- ROI 근거([[08_본선/03_제품/00_결정-준비/근거팩/ROI-근거팩|ROI-근거팩]], 심사용 보수 프레임 채택): JB RM 116석 기준 **연 0.83억원** 절감(보수, 기준 7.66억~공격 31.42억), 세대상 재작업·기록·handoff 반복 행정 회수분에 한정 — 확장 seat 적용은 [미검증].
- 규제 준수 실증: PII 4중 방어가 신용정보법 §40조의2·전자금융감독규정 §15조·금융위 2026 가이드라인 요구사항과 1:1로 매핑된다(canon §4).
- 확장성(5.3): 계열사별 모듈식 도메인팩(전북은행=여신·전세보호·피싱, JB우리캐피탈=여신·사후관리 EWS)이 동일 운영계약 위에서 재사용된다는 것을, 승보 프로토타입(JB우리캐피탈 13에이전트·전세보호 10에이전트, 이미 동작하는 코드)이 선행 실증한다([[08_본선/03_제품/00_결정-준비/설계/승보-프로토타입-반영|승보-프로토타입-반영]]).
