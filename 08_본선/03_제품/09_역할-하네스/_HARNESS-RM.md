# RM 역할 하네스 (LocalGuard OS RM 콘솔)

> ⚠️ 대외비 — 팀 내부 기획 자료. 외부 제출·공개·고객 대상 사용 전 사람 최종승인 필요.
> **핵심 원칙 — 역할 정본화(Role-canon-by-default)**: *콘솔에서 RM(기업여신 담당자) 역할이 겪는 화면·여정·키보드 규칙·에이전트 워크플로우·산출물 구조는 전부 이 하네스에서 출발한다.* [[_HARNESS-WOORICAP|계열사 도메인 하네스]]가 "타겟 계열사의 현실"을 정본화하듯, 이 하네스는 "역할이 콘솔을 쓰는 방식"을 정본화한다.
> **근거 SSOT**: 팀 Figma "망상궤도" 보드 실측(파일 `DOBJfqzxMZfVyq9UxnEm1r`, 보드 node `4924:2701`, 화면 3장) + 2026-07-04 팀 회의 확정 UX 스펙([[회의록-2026-07-04-스코프확정]]). **화면 단위 빌드 스펙(ASCII+컴포넌트트리+데이터바인딩+상태+키맵)은 [[_설계도-INDEX|설계도]] 5종이 정본** — 이 문서는 그 화면들을 가로지르는 RM 여정 전체의 상위 UX 스펙 + Figma 실측 재현이며, 설계도의 세부를 다시 만들지 않고 링크로 연결한다.

### 1.0 보드 플로우

```text
[프로세스 노트] "각각의 모든 케이스가 이렇게 업무를 처리하는 프로세스를 가진다" (+ 플로우차트, Figma 주석)
        │
        ▼
 화면 A(작업 전, node 4924:844)  ──▶  화면 B(작업 진행·md 통합 뷰어, node 4924:2522)  ──▶  화면 C(딤+md 플로팅 모달, node 4930:848)
 메인 업무보드                          진행 상태 배지 + 통합 뷰어                              전면 딤 + md 플로팅 모달
```

- 큐 순서·필터는 [[01-메인-업무보드]] `StageCountBar`/`KanbanBoard`(전/중/후 3분류 카운트 헤더)와 동일 개념의 역할별 재현이다.

│            │  [통합본] [문서희-MD] [상환위험-MD] [정책금융-MD]  ← 탭 전환      │
│            │  ── md 통합 뷰어 본문 (통합본 선택 시) ──────────────────────    │

│            │  ▾ 사용 출처 링크 (펼치기)                                     │

- 상태 배지는 진행 전/중/후 3분류를 그대로 쓰되 표현이 다르다 — **완료=파랑 배지**, **분석중=다크 배지**(어두운 색 = 완료가 아니라 "지금 진행중"을 가리키는 로컬 표기, §5 상태 규칙과 병기 확인 필요 [조건부]).
- 탭 = 통합본 + 개별 md ×3(케이스마다 개수는 실제 에이전트 체인 길이를 따름, §7).
- 하단 2단 접이식 섹션 = "사용 에이전트/스킬 표"(관여율 포함) + "사용 출처 링크" — [[02-케이스-상세]]의 `ResultMdViewer` → `SkillSourceFooter`와 같은 개념의 역할 레벨 재현.

### 1.5 실측 정합 참고 (Figma ↔ 다른 SSOT 대조 — 빌드 전 확인 필요)

이 문서는 Figma 실측을 있는 그대로 옮기되, 다른 정본 문서와 대조했을 때 나온 차이를 숨기지 않고 각주로 남긴다.

1. **[※1] Case 2 인물/계열사 표기差**: 현재 페르소나 SSOT [[전라도-실감-페르소나-확장]]의 JBG-206은 2026-07-03 데이터셋 정합 반영(Nemotron 실인물 매칭) 이후 **조동준·광주 광산구·광주은행**으로 갱신되었다(원안 강하준·병원 원무 → 실 데이터 각색, 전남/광주 소재는 광주은행으로 통일). Figma 화면은 그 갱신 이전 스냅숏이라 구(舊) 인물명·지역·계열사가 남아 있다. 케이스 서사(급여 정상화 전 상환일 겹침 → 급여흐름·DSR 조기경보)는 canon과 동일하므로, 빌드 시 **표시명/지역/계열사만 최신 페르소나 값으로 교체**하면 된다. [조건부]
2. **로딩 표기 差**: 화면 A 마지막 에이전트 카드는 "40%" 퍼센트를 보여주지만, 확정 설계도 [[03-에이전트-실행뷰]] §5는 "Loading(queued/running) = 스피너만(퍼센트 없음)"을 원칙으로 명시한다(LLM 지연은 추정 불가능하므로 가짜 정밀도를 보여주지 않는다는 취지). Figma는 초기 드래프트 표현으로 보이며, 빌드 시 **퍼센트 제거 + 스피너만 유지**가 확정 원칙과 맞다. [조건부]
3. **에이전트 배지 분해 差**: Case 2의 "Salary Flow / DSR Guard" 2개 배지는 [[전라도-실감-페르소나-확장]] JBG-206 서사에서는 "상환위험 분류 에이전트" **1개**가 급여 주기·DSR을 함께 판단하는 것으로 기술된다. Figma가 그 한 에이전트의 두 판단 축(급여흐름/DSR)을 별도 배지로 쪼갠 것인지, 실제로 두 개의 세부 에이전트로 분리할 계획인지는 디자이너·에이전트 설계 쪽 확인이 필요하다. [조건부]

## 2. 제품 한 줄 설명

RM(기업여신 담당자)이 콘솔에 들어오는 순간 근거 문장 기반으로 정렬된 오늘의 케이스를 보고, 배정된 에이전트를 Enter 한 번으로 승인·실행하며, 응답을 기다리는 동안에는 다른 케이스를 계속 처리하다가 완료 알림으로 돌아와 통합 MD 산출물을 확인하는 — 키보드만으로 완주하는 AI 협업 콘솔. "일은 에이전트가, 사람은 눈만"([[회의록-2026-07-04-스코프확정]] 확정 슬로건).

## 3. 핵심 사용자 여정

1. **케이스 선택** — 업무보드(화면 A) 큐에서 숫자키 또는 Enter로 케이스 진입. 정렬 기준은 점수가 아니라 근거 문장(FR-27, §9-1).
2. **에이전트 배정 확인** — 케이스 상세 하위에 배치된 에이전트 카드에서 "붙는 이유(한 줄)"·"기대값"·데이터 칩을 확인([[03-에이전트-실행뷰]] `AgentReasonLine`/`ExpectedValuePreview`).
3. **Enter 승인·실행** — 포커스된 에이전트 카드에서 Enter. 일반 실행은 단일 Enter, `riskLevel=high/critical` 산출물은 2단 확인(§6).
4. **로딩 흡수** — 실행 중에는 퍼센트 없는 스피너만 표시되고, RM은 `Esc`/`BackgroundNavHint`로 다른 케이스를 계속 본다(FR-22). 화면 A의 "다른 케이스 보기로 흡수"가 이 지점이다.
5. **완료 알림** — 백그라운드 완료 시 상단 고정 알림(🔔 띠링)이 뜨고, 클릭하면 해당 케이스로 복귀한다(`NotificationChime`).
6. **통합 MD 확인** — 케이스로 복귀하면 통합 MD(`ResultMdViewer`)를 확인하고, 필요하면 개별 MD로 옵시디언식 링크 이동, 최종 승인은 [[04-승인-대기·결정]]에서 마무리한다.

## 7. 에이전트 워크플로우

케이스 진입 → 도메인별 에이전트 체인이 순차 실행되며, 각 에이전트는 스킬/MCP/외부 공공 API/내부 규정 중 필요한 무기를 사용하고 결과를 개별 MD(`outputSummary`)로 남긴다. 체인이 끝나면 오케스트레이터가 통합 MD 1개로 취합하고, 고객 접점 행동(연락·조건변경·발송)은 전부 사람 승인 게이트를 거친다 — PII는 항상 콘솔 내부에만 남고 외부로 반출되지 않는다(전 케이스 공통 원칙, [[전라도-실감-페르소나-확장]]).

## 8. MD 결과물 구조

- **통합본**: 케이스별 1개, 메타데이터(케이스ID·작성 에이전트·시각) → 제목 → Summary → 근거 표 순으로 렌더(§1.3 화면 C). `[[개별MD]]` 링크·`@에이전트` 멘션을 포함한다.
- **개별 MD**: 체인의 각 에이전트가 남기는 산출물, 케이스당 3~5개(§7 표). 통합본에서 옵시디언식으로 클릭 진입(`MdLinkChip` → `FloatingMdModal`).
- **하단 공통**: 사용 스킬 배지 + 출처(sourceTag) 링크를 모든 MD 맨 아래 고정(`SkillSourceFooter`) — 모든 수치·판단에 근거를 연결해 환각을 억제한다(§9-4).
- **뷰 형태**: 사이드/탭/플로팅 3안 중 이 하네스 기준은 **탭(개요) + 플로팅(개별 상세)** 조합(화면 B/C 실측) — 최종 택1은 팀 확정 대기([[_디자이너-핸드오프]] "결과물 열람 3안" 미결 항목).

## 9. MVP 범위 / 고도화 범위

**MVP**: 통합 MD 1개 + 개별 MD 목록(케이스당 3~5개) 열람, 탭 전환(통합본|개별×N), 플로팅 모달, 스킬·출처 하단 고정 표시, 진행 전/중/후 상태 배지.

**고도화**:
- `[[개별MD]]` 실제 클릭 네비게이션 — 미구현 시 통합+개별 분리 표시로 폴백([[02-케이스-상세]] §5 "폴백(결과물)" 상태)
- `@에이전트` 멘션 → 에이전트 워크스페이스(`S-07-detail`) 실제 연결
- 알림 고도화 — 현재는 단일 띠링 배지, 향후 우선순위별 정렬·다중 알림 큐잉
- 카운트 헤더(전/중/후) 클릭 시 필터링 세부 UX

## 10. 발표 차별점

1. **근거 기반 우선순위** — 케이스 순서는 점수가 아니라 "왜 이 케이스가 먼저인가"를 문장으로 보여준다(FR-27). 가중치 산식은 있되(01-메인-업무보드 §4 `priorityScore`), 화면에는 항상 근거 문장이 먼저 보인다.
2. **키보드 퍼스트** — 숫자·Enter·Space·Tab만으로 케이스 선택부터 승인까지 완주(FR-20). "일은 에이전트가, 사람은 눈만"을 화면 조작 자체로 증명한다.
3. **지연 흡수 UX** — LLM 응답 지연을 숨은 로딩바가 아니라 "다른 케이스 검토"로 치환한다(FR-22). 기다림을 생산적 행동으로 바꾸는 것이 차별점이다.
4. **출처 연결 환각 억제** — 모든 MD 산출물 하단에 사용 스킬 + 출처를 고정 노출(`SkillSourceFooter`)하고, 준법 검토 에이전트가 확정 표현·근거 부족을 별도로 점검하는 생성-검증 분리 원칙을 따른다(다른 콘솔의 evaluator 패턴과 동일 철학, [[agent-roster|에이전트 로스터]] §2.3).
