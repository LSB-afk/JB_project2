# 04. 승인 대기함 · 결정 (S-04 / S-04-detail)

> SSOT: `06-정보체계-뷰-데이터바인딩-스펙.md` §3.6(승인대기함) · §3.7(승인/거부/수정후승인) · `기능요구사항-7-4회의.md` FR-04·06·09·20.

## 1. 목적 · 진입

- **목적**: 사람 결정이 필요한 `ApprovalRecord`를 위험도·SLA·역할·계열사 기준으로 놓치지 않게 정렬하고(`S-04`), 승인자가 초안·근거·규정검증·감사 영향을 한 화면에서 보고 승인/거부/수정후승인을 결정하게 한다(`S-04-detail`).
- **진입**: [[01-메인-업무보드]] 알림 badge → `S-04`. `Ctrl+2`(§2.3.1) 단축 이동. `ApprovalCard` Enter/클릭 → `S-04-detail`. [[02-케이스-상세]]의 `ApprovalStatusStrip`에서도 `S-04-detail`로 직행(결정은 이 화면에서만 일어난다).
- **역할 분기**:
  - `reviewer`: 본인이 요청한 승인 상태 추적 위주(읽기), 수정 요청 대응·resubmit.
  - `supervisor`/`준법`: approve/reject/modify CTA 표시, L3/L4는 상단 우선.
  - `관리자`/`감사자`: 읽기 전용(raw payload·감사만).

## 6. 키보드 · 인터랙션 맵

| 키 | 동작 | 2단 확인 | 전환 |
|---|---|---|---|
| `↓`/`Space`(S-04 큐) | 다음 승인 건으로 포커스 이동 | 없음 | 없음 |
| `Enter`(`ApprovalCard`) | `S-04-detail` 진입 | 없음 | 페이지 전환 |
| `Enter`(S-04-detail, `GateCheckList` 통과 후) | 승인 | **L3/L4는 마우스 명시 확인 또는 2단 확인 강제**(§2.3 예외) | `ConfirmDialog` → 결정 확정 → 감사 기록 |
| `Shift+Enter` | 거부/수정 요청 | **사유코드/메모 필수**, 없으면 제출 차단 | 상태 전이(`rejected`/`revisionRequested`) |
| `Tab` | ActionDraft → Evidence → Gate 순 포커스 이동(3분할 순회) | 위험 결정은 `Tab`으로 확인 버튼까지 이동 후 명시 활성화(a11y 필수) | — |
| 필터 항목에서 `Enter` | 필터 적용 | 없음 | 큐 재정렬 |

- `Ctrl+2`(또는 대안 바인딩 `[분기/미확정]`)로 이 화면 전체 진입 가능(§2.3.1 뷰 이동 축, Enter-first 케이스 진행 축과 분리).
- 고위험(L3/L4) 승인은 Enter 단독으로 종료되지 않는다 — "엔터에 과한 책임 금지" 원칙이 여기서 가장 엄격히 적용된다.
