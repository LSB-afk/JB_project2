# [Q5] 은행 DB 연결 방식 — 후보·추천

## 배경·현재가안

Q5의 핵심은 “은행 DB에 바로 붙일 것인가, 데모용 정적 데이터로 갈 것인가, 둘을 나눌 것인가”다. 현재 SSOT 커널은 `Case→AgentRun→Agent→Skill→Evidence→Approval→Audit`이고, 4함수계약은 정적 MVP에서 `computeRiskDecision`, `buildDashboardData`, `auditChainRecords`, `moveCaseToColumn`을 먼저 검증한 뒤 서버 API로 승격하는 방향이다.

기존 근거상 가장 강한 원칙은 세 가지다. 첫째, 에이전트 평가는 최종 문장보다 DB/상태가 목표 상태와 일치하는지 봐야 한다(D13). 둘째, 금융 AI는 자동판단 엔진보다 RM 승인·감사 루프가 결합된 보조수단이 법·감사 적합성이 높다(D14). 셋째, 외부 LLM에는 고객 PII·개인신용정보를 보내지 않는 구조가 망분리·SaaS 예외 정책과 가장 정합적이다(D5b).

## 후보 A·B·C

### 후보 A. 정적 데이터 연결

- 데이터 근거·출처: D13은 발표장 MVP에서 deterministic 골든패스, 정답셋, 정책룰, 감사로그로 검증 가능한 상태 변화를 보여줘야 한다고 정리했다. D13은 `Playwright e2e`와 `verify_static.py`를 검증 증거물로 격상하라고 제안했다.
- 방식: 은행 DB 연결 없이 `modules.js` 또는 JSON seed로 여신·전세·피싱 케이스를 고정하고, 4함수계약만 브라우저에서 실행한다.
- 장점: 7/5 제출 전 리스크가 가장 낮다. PII 반출·망분리·DB 접근권한 이슈가 없다. 같은 입력에서 같은 `riskScore`, 승인레벨, GENESIS 해시체인, 감사로그가 재현된다.
- 단점: “실제 은행 DB와 붙는가?” 질문에는 약하다. ROI와 배포 가능성은 데모 수준에 머문다. D13 기준의 goal-state 검증은 가능하지만, D-테스트베드식 실데이터 14주 검증으로 이어졌다는 증거는 없다.

### 후보 B. 서버 API 직접 연결

- 데이터 근거·출처: D13은 에이전트 평가가 대화 종료 후 DB 상태와 목표 상태를 비교해야 한다고 한다. D14는 소상공인 여신의 thin-file·분절 데이터, 전세 데이터 품질 편차, FDS의 class imbalance·concept drift를 지적한다. D5b는 내부망·외부망 분리 원칙과 고유식별정보·개인신용정보 외부 처리 제한을 제시한다.
- 방식: 실제 또는 유사 은행 DB를 서버 API/MCP 게이트웨이에 연결하고, 콘솔은 API를 통해 case/evidence/approval/audit를 읽고 쓴다.
- 장점: 실배포 설득력이 가장 높다. D13의 goal-state 평가와 운영 감사로그를 실제 시스템 구조로 구현할 수 있다. 향후 JB 전북은행·JB우리캐피탈 공통 운영계약으로 확장하기 좋다.
- 단점: 제출 전 구현·보안·권한·망분리 리스크가 크다. 현장 케이스 메모리 관리 방식이 미결이면 데이터 정합성, 캐시, 재평가, 이의제기 재산정 흐름을 확정하기 어렵다. 신용정보법 §40조의2 ①②⑥⑦⑧ 적용은 근거 확보됨 → 근거: [[적법성-근거팩]] (망분리 예외 항·호 번호만 제출 전 재확인 필요).

### 후보 C. 하이브리드 연결

- 데이터 근거·출처: D13은 정적 규칙 검증, 승인 전 실행 금지, 근거 추적, 감사로그를 KPI와 1:1 매핑하라고 한다. D14는 인간 승인·설명패킷·재심 파이프라인을 제품 중심에 두라고 한다. D5b는 외부 LLM은 SaaS/샌드박스 경로로 쓰되 고객 PII·개인신용정보는 외부 반출 금지하는 설계를 가장 보수적인 정책 적합 방향으로 본다.
- 방식: 발표 MVP는 정적 seed로 돌리되, 내부 아키텍처는 `LocalGuard API Gateway`를 전제로 둔다. 프런트는 `CaseRepository` 같은 포트를 호출하고, 현재 어댑터는 static fixture, 다음 어댑터는 read-only bank API, 최종 어댑터는 승인 토큰 기반 write API로 승격한다. 원본 PII·신용정보 처리는 로컬모델/내부망, 외부 LLM은 비식별 factsheet·정책룰 ID·근거요약만 받는다.
- 장점: 7/5에는 안정적인 실동작을 보이고, 심사위원 질문에는 “정적 데모가 아니라 API 승격 가능한 계약”으로 답할 수 있다. PII 4중방어와 승인레벨 L0~L4를 유지하면서도 서버 배포 경로를 설명할 수 있다.
- 단점: 실제 DB 연결 완료는 아니다. “은행 DB 연결 완료”라고 말하면 과장이다. 김주용·이승보와 API/MCP 게이트웨이, read-only 범위, 케이스 메모리 소유권, 승인 후 write-back 범위를 확정해야 한다.

## 추천안

추천은 **후보 C. 하이브리드 연결**이다. 발표 문구는 “현재 MVP는 정적 검증 데이터로 deterministic하게 실동작하고, 운영 전환 시 동일 4함수계약을 `LocalGuard API Gateway` 뒤의 은행 read-only API/MCP 어댑터로 교체한다”가 가장 안전하다.

① 페인포인트가 실재·정량인가: 소상공인 thin-file·분절 데이터는 D14, FDS 사기 거래 0.13%와 class imbalance·concept drift는 D14, D-테스트베드의 실데이터 14주 검증 선례는 D13에 있다. 다만 “5천만원 절감” 직접 근거는 없음(근거 필요).

② 적법성: PIPA 자동화된 결정 설명·거부·재검토 권리는 D14, 신용정보법 §36조의2(별개 조문)에 있다. 망분리 원칙, SaaS 예외, 고유식별정보·개인신용정보 외부 처리 제한은 D5b에 있다. 신용정보법 §40조의2 ①②⑥⑦⑧(가명처리·재식별금지, 보충 PIPA §28조의4·§28조의5)이 근거다 → 근거: [[적법성-근거팩]].

③ 신뢰성·정확성: D13 기준대로 정답셋, 상태 변화, 승인 전 실행 0건, 해시체인 감사로그로 검증한다. 하나 놓치면 위험한 전세·피싱은 accuracy보다 recall/FN 중심으로 말해야 한다(D13, D14).

④ ROI: 하이브리드는 정적 데모보다 파일럿 전환성이 높고, 서버 직결보다 제출 리스크가 낮다. Triage 시간 50%↓ 같은 측정 프로토콜은 D13에 있으나, 5천만원급 ROI 산식은 별도 산출 필요(근거 필요).

⑤ 실동작·배포 증명: 7/5에는 정적 MVP+검증 스크립트+e2e로 실동작을 증명하고, 서버 승격은 API 계약·어댑터 구조·read-only fallback으로 제시한다. D13의 “관측된 증거” 언어와 맞다.

⑥ 확장성: `Case→AgentRun→Evidence→Approval→Audit` 계약을 고정하면 전북은행 여신, JB우리캐피탈 심사, 전세·피싱 도메인을 같은 운영체계로 확장할 수 있다. 외부 LLM은 비식별 보조, 로컬모델은 원본 처리로 라우팅하면 D5b의 정책 방향과도 맞다.

## 리스크·의존성

- 김주용·이승보 논의 필요: API/MCP 중 어느 연결 방식을 공식 표현으로 쓸지, 7/5 발표에서 “연결 예정”과 “구현 완료”의 경계를 확정해야 한다.
- 현장 케이스 메모리 관리 방식 미결: case memory가 브라우저 localStorage, 서버 DB, 은행 원장, 별도 감사원장 중 어디에 남는지 정해야 한다.
- write-back 금지선 필요: L0 관찰·로그는 read-only 가능, L1/L2 이상은 RM 승인 토큰 없이는 고객 알림·약정·자금 release·상태 확정 write 금지로 잡는 편이 안전하다.
- 법무 근거 갭: 신용정보법 §40조의2는 해소됨(근거: [[적법성-근거팩]]); 은행 내부 DB 접근 정책, 계열사 간 데이터 공동활용 범위는 현재 제공 D코드만으로 단정 불가(근거 필요).
- ROI 갭: 5천만원급 효과는 시간절감, 사고예방, 감사비용 절감, 파일럿 표본 수로 별도 계산해야 한다(근거 필요).

## 팀 후속 딥리서치 프롬프트

JB LocalGuard OS의 은행 DB 연결 방식을 검증하라. 후보는 정적 seed, 서버 API/MCP 직접연결, 하이브리드 어댑터다. 전북은행·JB우리캐피탈 환경에서 read-only 조회, 승인 후 write-back, case memory 저장 위치, 감사원장 분리, PII·개인신용정보 외부 반출 금지, 망분리/SaaS 예외, 신용정보법 §40조의2·PIPA 적용을 확인하라. 각 후보별 법무 리스크, 보안통제, 7/5 발표 가능성, 4주/14주 파일럿 전환성, 5천만원급 ROI 산식까지 표로 비교하고, 공개 1차 근거가 없으면 반드시 “근거 필요”로 남겨라.

## 🔎 심사위원 반박 & 보강

### 반박 요지

**① 페인포인트 실재·정량**: D14의 thin-file·class imbalance·concept drift, D13의 D-테스트베드 14주 선례는 실재하는 근거지만, 이 문서가 실제로 인용하는 것은 "D13/D14가 그렇게 말했다"는 2차 서술뿐이다. 원문 수치(FDS 사기율 0.13% 등)를 이 문서 안에서 직접 검증할 수 없고, 문서 스스로도 "5천만원 절감 직접 근거는 없음"을 인정한다. 페인포인트의 존재 자체는 인정하되, 그 크기(=이 프로젝트가 해결해야 할 만큼 큰가)는 정량화되어 있지 않다.

**② 적법성**: ~~신용정보법 §40조의2 자체 근거 없음~~ → 오탐. §40조의2 ①②⑥⑦⑧(가명처리·재식별금지)이 근거이며 로컬처리·외부반출 금지 설계와 1:1 대응한다 → 근거: [[적법성-근거팩]]. 다만 read-only bank API 접속 자체의 은행 내부 접근 정책·계열사 데이터 공동활용 범위는 법무 검토가 별도로 필요하다(설계 확정보다 법무 확인이 선행돼야 한다는 원칙은 유효).

**③ 신뢰성·정확성 — 검증되지 않은 핵심 개념**: 후보 C가 제시하는 `LocalGuard API Gateway`, `CaseRepository` 포트/어댑터 구조는 이 문서에서 처음 등장하는 이름이다. 코드베이스(`02_제품/app/`)와 `08_본선/` 전체를 검색해도 `CaseRepository`·`API Gateway`·`adapter` 관련 기존 설계·코드가 전혀 없다(grep 결과 0건, 이 문서 자체 제외). 즉 "정적 fixture → read-only bank API → write API로 순차 승격 가능한 포트 구조"는 실제로 구현되거나 검증된 적 없는 **이 문서 안에서 새로 제안된 가정**이며, 4함수계약(`computeRiskDecision` 등)이 실제로 이런 포트 뒤에서 어댑터 교체만으로 동작하도록 설계돼 있는지 코드 근거가 없다. "동일 4함수계약을 어댑터로 교체한다"는 발표 문구(39행)는 현재 코드가 증명하지 못하는 약속이다.

**④ ROI**: 문서 스스로 "5천만원급 ROI 산식은 별도 산출 필요"라고 명시(47행). 추천 근거의 핵심 축 하나가 완전히 비어 있는 상태에서 "추천"을 확정하는 것은 시기상조다.

**⑤ 실동작·배포 증명**: 7/5 시점에 증명되는 것은 정적 MVP뿐이다. 하이브리드의 "서버 승격 가능성"은 코드가 아니라 발표 문구와 아키텍처 의도로만 증명된다. 심사위원이 "포트 구조 코드를 보여달라"고 물으면 현재 근거로는 답할 수 없다.

**⑥ 확장성**: 계약(`Case→AgentRun→Evidence→Approval→Audit`) 고정 자체는 타당한 원칙이나, 이것이 후보 A(정적)에서도 동일하게 성립하는 원칙이라 하이브리드만의 차별화 근거로 보기 약하다.

### 판정: **약함(조건부 유지)**

방향성(정적 데모 + 서버 승격 서사)은 7/5 리스크 관리 관점에서 합리적이나, ①법무 미확인 상태에서 read-only bank API 연결을 전제로 한 것, ②`LocalGuard API Gateway`/`CaseRepository`가 코드·기존 설계 근거 없이 이 문서에서 처음 제안된 것, ③ROI 산식 부재 — 이 세 가지가 해소되지 않으면 "추천안"이 아니라 "가설안"으로 격을 낮춰야 한다.

### 보강 필요 항목

- (해소됨 → 근거: [[적법성-근거팩]]) 신용정보법 §40조의2 ①②⑥⑦⑧이 원문 근거. 남은 것은 은행 내부 DB 접근 정책·계열사 데이터 공동활용 범위의 별도 법무 확인.
- `LocalGuard API Gateway`/`CaseRepository` 개념이 이 문서 이전에 팀 내 합의·설계 문서로 존재했는지 확인 — 없다면 "추천안의 핵심 근거"가 아니라 "새 제안"임을 문구에서 명시
- 5천만원급 ROI 산식(시간절감·사고예방·감사비용·표본수 기반 계산) 별도 산출
- 발표 문구에서 "동일 4함수계약을 어댑터로 교체"라는 표현은 현재 코드가 증명 못 하므로, 실제로 4함수가 데이터 소스 추상화 없이 하드코딩돼 있다면 발표 전 문구 수정 필요
