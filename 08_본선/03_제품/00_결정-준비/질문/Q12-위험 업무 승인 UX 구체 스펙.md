# [Q12] 위험 업무 승인 UX 구체 스펙 — 후보·추천

## 배경·현재가안

위험 업무 승인 UX는 아직 D28 미회수 갭이다. 현재 근거 있는 축은 “AI가 실행 초안을 만들고, 최종 판정은 사람”이라는 승인우선 패턴이다. D30은 은행연합회-업스테이지 AI 광고심의 사례를 “AI가 보고서, 최종 판정은 사람”으로 정리했고, 이를 제도권 조달 패턴으로 봤다. D15는 승인루프·감사증거 방어력을 핵심으로 회수됐다. B1은 StackStorm 승인게이트·실행이력, Temporal append-only 이벤트·Signal 승인 구조를 롤모델로 제시했다.

SSOT 기준 UX는 `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit` 순서를 화면에 그대로 드러내야 한다. 즉 “승인 버튼”이 아니라 “근거 묶음, 위험결정, 사람 승인, 컬럼 이동, 감사원장 기록”까지 한 번에 닫히는 운영계약 UX여야 한다.

권장 승인 레벨 초안:

| 레벨 | 의미 | 사람 UX | 실행/컬럼 연계 |
|---|---|---|---|
| L0 | 관찰·로그 | 승인 없음, 근거 자동 적재 | `moveCaseToColumn(caseId, "monitoring")` |
| L1 | RM 검토 | Evidence 전체 열람 후 “검토완료” | `review_queue → rm_reviewed` |
| L2 | RM 승인 | 위험요약·PII 반출스캔·대안안 확인 후 승인/반려 | `approval_queue → approved` 또는 `rejected` |
| L3 | 준법 승인 | 준법 체크리스트, 조항·근거 링크, 승인 사유 필수 | `compliance_queue → compliance_approved` |
| L4 | 차단·상위검토 | 기본 차단, 상위 승인 없이는 실행 불가 | `blocked → escalated` 또는 `blocked_closed` |

스페이스 두 번, 키보드 스크롤 후 승인 같은 마찰 장치는 현 가안 아이디어일 뿐, D코드 근거는 없다. 채택하려면 “오승인 감소율, 승인시간 증가폭, 사용자 피로도” 실험 근거가 필요하다. 현재 표기는 `(근거 필요)`이 맞다.

## 후보 A·B·C

### 후보 A. Evidence-First 2단 승인 UX

데이터 근거·출처: D15 승인우선·감사원장, D30 “AI 보고서+사람 최종판정”, B1 승인게이트·실행이력.

구성:
- L0: 자동 관찰·로그.
- L1: RM이 Evidence 묶음을 끝까지 열람해야 “검토완료” 활성화.
- L2: RM 승인자는 위험결정, 반출스캔 결과, 대안 조치안을 보고 승인/반려.
- L3 이상은 준법 큐로 자동 승격.
- 모든 전이는 `moveCaseToColumn` 호출과 `auditChainRecords` GENESIS 해시체인 기록을 동시에 남김.

장점:
- 시연에서 이해가 빠르다.
- D15·D30·B1 근거와 가장 잘 맞는다.
- 7/5 제출 전 구현 난도가 낮다.

단점:
- L3/L4의 준법·상위검토 세부 체크리스트는 추가 정의 필요.
- 스페이스 두 번 등 행동 마찰의 정량효과는 근거 부족.

### 후보 B. Friction-by-Risk UX

데이터 근거·출처: D21 관측-이해-가상검증-실행 운영레이어, D22 로컬 우선+통제 승격, D25 데이터레인 적법성. 단, “스페이스 두 번/키보드 스크롤” 자체는 `(근거 필요)`.

구성:
- L1: Evidence 스크롤 열람.
- L2: 승인 사유 20자 이상 또는 사유 템플릿 선택.
- L3: 준법 체크박스와 관련 조항 확인.
- L4: 버튼 비활성, 상위검토 요청만 가능.
- 고위험일수록 승인 전 “실행 결과 미리보기/가상검증”을 요구.

장점:
- “위험할수록 마찰이 커진다”는 메시지가 선명하다.
- 하나 놓치면 큰 금융업무에 맞는 UX 논리가 있다.
- L4 차단을 화면에서 강하게 보여줄 수 있다.

단점:
- 마찰 장치가 과하면 현업 생산성 저하로 보일 수 있다.
- 구체 장치별 효과 데이터가 없다.
- 잘못 만들면 데모가 장황해진다.

### 후보 C. Compliance-Led Gate UX

데이터 근거·출처: D25 신용정보법 §17·§32·§33·§40조의2, PIPA §3·§21, 금융지주회사법 §48조의2, 금융위 2026.4 SaaS 예외·2026.5 생성형AI. D5a 법조문, D5b 망분리·클라우드.

구성:
- 승인 화면의 중심을 “법적 허용 여부”로 둔다.
- 원본 PII·신용정보가 외부 프런티어 LLM으로 나가면 L4 차단.
- 로컬모델 처리, 토큰화, 반출스캔 통과 시에만 L1/L2 진행.
- L3는 준법 전용 큐에서 조항별 확인을 요구.

장점:
- 깐깐한 금융 심사위원에게 방어력이 높다.
- PII 4중방어와 직접 연결된다.
- 법·망분리 질문에 답하기 쉽다.

단점:
- 제품 UX보다 규제 체크리스트처럼 보일 수 있다.
- 5분 예선 발표에서는 복잡해질 수 있다.
- ROI·현업 페인포인트 전달력이 약해질 수 있다.

## 추천안

추천은 **후보 A를 기본으로, 후보 B의 위험별 마찰과 후보 C의 L4 차단 규칙을 얹은 “Evidence-First Risk Gate”**다.

핵심 스펙:
- `computeRiskDecision`이 L0~L4를 산출한다.
- `buildDashboardData`가 레벨별 큐와 승인 카드 데이터를 만든다.
- 사용자는 Evidence 패널, 위험사유, PII 반출스캔, 추천 조치를 본다.
- 승인/반려/상위검토 선택 시 `moveCaseToColumn`이 케이스 컬럼을 이동한다.
- 동시에 `Approval`과 `Audit`이 `auditChainRecords`로 해시체인에 남는다.
- L4는 승인 버튼을 없애고 “상위검토 요청”만 허용한다.

깐깐한 금융 심사위원 렌즈:

① 페인포인트가 실재·정량인가: D23은 위험조정 NPV 약 24.2억, ROI 약 471%, 회수 약 5.4개월을 제시했다. D30은 제품형 ARR 초기 SOM 약 4.6~13.5억을 제시했다. 단, 승인 UX 자체의 오승인 감소율은 `(근거 필요)`.

② 적법성: D25와 D5a/D5b 근거로 신용정보법 §40조의2, PIPA, 망분리·클라우드 이슈를 L4 차단 규칙에 직접 반영한다. 원본 PII·신용정보 외부 LLM 반출 금지는 UX에서 “차단 사유”로 표시한다.

③ 신뢰성·정확성: D14는 데이터·모델 신뢰성, D20은 LLM을 주모델이 아니라 오케스트레이터로 두는 방향을 뒷받침한다. 하나 놓치면 실행되지 않게 L3/L4를 차단·상위검토로 설계한다.

④ ROI가 5천만원급인가: D23의 NPV·ROI 수치가 5천만원급을 넘는 논거다. 다만 후보 UX 단독 ROI가 아니라 운영콘솔 전체 ROI라는 점은 명확히 말해야 한다.

⑤ 실동작·배포로 증명되나: 정적 MVP에서는 4함수계약으로 증명한다. `computeRiskDecision → buildDashboardData → moveCaseToColumn → auditChainRecords` 흐름이 한 케이스에서 실제로 작동해야 한다.

⑥ 확장성: B1의 StackStorm/Temporal 참조처럼 승인게이트와 append-only 이벤트는 서버 API·durable workflow로 승격 가능하다. D30의 White space도 “원본 PII 내부처리 + 승인우선 + 계열사 이식 + 감사증적 내장” 동시 포지션을 근거로 삼는다.

## 리스크·의존성

- D28 승인UX 리서치가 미회수라 세부 마찰 장치의 근거가 약하다.
- “스페이스 두 번/스크롤 후 승인”은 데모 아이디어로만 쓰고, 정량효과는 `(근거 필요)`로 표시해야 한다.
- L3 준법 체크리스트는 D25 조항을 UX 문구로 재정리해야 한다.
- L4 차단 조건은 PII 반출, 신용정보 외부 전송, 모델 라우팅 실패, 감사기록 실패 중 무엇을 필수 트리거로 볼지 확정 필요.
- `moveCaseToColumn`은 단순 UI 이동이 아니라 Approval·Audit 생성과 원자적으로 묶여야 한다.
- e2e 테스트는 L0 자동로그, L2 승인, L3 준법큐, L4 차단 최소 4개가 필요하다.

## 팀 후속 딥리서치 프롬프트

금융권 위험 업무 승인 UX의 구체 패턴을 조사해줘. 대상은 대출심사, 이상거래, 광고심의, 개인정보 반출, RPA/워크플로 승인이다. 승인 전 마찰 장치(스크롤 열람, 이중확인, 사유 입력, 키보드 확인, 4-eyes principle)가 오승인·처리시간·감사 대응에 미친 정량 근거를 찾아라. 국내 금융 규제, 망분리, 신용정보법 §40조의2, PIPA 관점에서 L0~L4 승인레벨별 권장 UX와 금지 UX를 표로 정리하고, JB LocalGuard의 `Case→Approval→Audit→moveCaseToColumn` 설계에 바로 넣을 스펙으로 제안하라.

## 🔎 심사위원 반박 & 보강

검증 방법: `02_제품/app/app.js`를 grep해 이 문서가 "실동작 증명"의 근거로 삼는 함수·상태값이 실제로 존재하는지 대조했다.

### 반박 요지

1. **L0~L4 네이밍 충돌 — 문서가 언급하지 않음.** `app.js` 703~707행에는 이미 L0~L4가 정의돼 있고, 의미가 이 문서의 제안과 다르다. 기존 것은 **위험점수 구간**(L0=0-39, L1=40-59, L2=60-79, L3=80-89, L4=90-100)에 `customerNotice`/`contract`/`fraud` 3열 대응표다. 이 문서가 새로 제안하는 L0~L4는 **승인 워크플로 단계**(관찰→RM검토→RM승인→준법승인→차단)다. 같은 레이블 L0~L4가 이미 심사·시연 화면에 노출된 "위험점수 구간" 의미로 굳어 있는데, 이 문서는 그 위에 다른 의미의 L0~L4를 덧씌우려 하면서 충돌 여부를 전혀 검토하지 않았다. 실제 구현 시 화면·발표 양쪽에서 "L2가 뭘 뜻하냐"는 질문에 두 가지 답이 나올 위험이 크다.

2. **"실동작 증명" 주장이 기존 코드와 신규 제안을 섞어 과장한다.** `computeRiskDecision`, `buildDashboardData`, `moveCaseToColumn`, `auditChainRecords`(GENESIS 해시체인 포함)는 실제로 `app.js`에 구현·호출돼 있어 이 부분은 사실이다. 그러나 이 문서가 스펙으로 제시한 상태값 `rm_reviewed`, `review_queue`, `approval_queue`, `compliance_queue`, `blocked_closed`는 `app.js` 전체에 **단 한 곳도 존재하지 않는다** (grep 결과 0건). "정적 MVP에서는 4함수계약으로 증명한다"는 문장은 이미 있는 함수 이름을 근거로 아직 없는 큐·상태 설계까지 "증명됐다"는 인상을 준다. ⑤ 항목 판정은 재작성이 필요하다: 함수 골격은 실동작하지만, 이 문서가 제안한 승인 단계·큐 자체는 미구현 설계도일 뿐이다.

3. **페인포인트의 정량 근거가 사실상 없다.** 본문이 스스로 "오승인 감소율·승인시간 증가폭·사용자 피로도는 (근거 필요)"라 인정했지만, 더 근본적으로 **현재 승인 프로세스의 베이스라인**(단계 수, 평균 소요시간, 실제 오승인 사례)조차 인용되지 않는다. "위험할수록 마찰이 커져야 한다"는 설계 원칙 자체가 정성적 직관이며, 4-eyes principle 등 인용된 개념도 실증 문헌 없이 이름만 언급됐다. 페인포인트가 아니라 설계 가설 단계다.

4. **적법성 반영이 미확정 상태와 모순된다.** 추천안은 "①②의 근거로 §40조의2·PIPA·망분리 이슈를 L4 차단 규칙에 직접 반영한다"고 단정하지만, 바로 아래 "리스크·의존성" 절에서 "L4 차단 조건은 PII 반출, 신용정보 외부 전송, 모델 라우팅 실패, 감사기록 실패 중 무엇을 필수 트리거로 볼지 확정 필요"라고 스스로 뒤집는다. 즉 적법성 방어는 아직 설계되지 않았고, "반영한다"는 현재형 서술이 과장이다.

5. **ROI는 승인 UX 단독 근거가 아니다.** 문서도 "후보 UX 단독 ROI가 아니라 운영콘솔 전체 ROI"라고 인정한다. D23 NPV·ROI(약 24.2억/471%/5.4개월)는 콘솔 전체 수치이고, 이 UX 스펙 자체가 5천만원급 심사를 통과할 별도 근거는 없다.

6. **오분류 시 이중 안전장치 부재.** L3/L4는 "하나 놓치면 실행되지 않게 차단·상위검토로 설계"한다고 하나, 이는 `computeRiskDecision`이 올바르게 L3/L4로 분류했을 때만 유효하다. 실제로 L4건이 L2로 오분류되는 경우(모델 오류·데이터 누락)에 대한 사후 감사 트리거나 재분류 안전장치는 스펙에 없다. Single point of failure다.

### 판정: **약함 (weak)**

방향성(Evidence-First + 위험별 마찰 + 준법 차단 결합)은 SSOT `Case→AgentRun→...→Audit` 계약과 부합해 완전 기각은 아니다. 그러나 ⑤ 실동작 증명이 과장돼 있고, ①③④의 핵심 수치·확정 사항이 다수 `(근거 필요)`/미확정이며, 기존 앱의 L0~L4와 이름 충돌을 검토하지 않은 채 채택 직전 상태로 서술된 점에서 "추천안"으로서는 근거가 약하다.

### 보강 필요항목 (구체)

- 기존 `app.js` L0~L4(위험점수 구간)와 이 문서 L0~L4(승인 단계)의 관계를 명시하는 매핑표 — 같은 레이블 재사용인지, 다른 축(예: 승인단계는 `A0~A4`)으로 분리할지 결정.
- `rm_reviewed`/`approval_queue`/`compliance_queue`/`blocked_closed` 등 신규 상태값의 구현 공수 산정 및 `moveCaseToColumn`/`auditChainRecords` 연동 설계 — 현재는 이름만 있고 코드 없음.
- 현재(as-is) 승인 프로세스 베이스라인 데이터: 단계 수, 평균 처리시간, 오승인 사례 건수 — 데이터 출처 명시.
- L4 필수 차단 트리거 확정(PII 반출/신용정보 외부전송/모델 라우팅 실패/감사기록 실패 중 무엇을 AND/OR로 묶을지).
- `computeRiskDecision` 오분류 시 사후 감사·재분류 절차 — 현재 스펙에 없음.
- 승인 UX 단독의 ROI 또는 효과 지표를 콘솔 전체 ROI와 분리해 별도 표기, 혹은 "단독 ROI 주장 안 함"으로 명문화.

### 통과하려면 추가로 필요한 데이터/실험

- D28 승인UX 리서치 회수(마찰 장치의 오승인·처리시간 정량 문헌, 국내외 사례).
- 현 JB(또는 유사 지방은행) 승인 프로세스의 실제 소요시간·단계 수 인터뷰/문서 근거.
- 신규 큐 상태값을 반영한 `app.js` 프로토타입 1건 + e2e 시나리오(L0 자동로그/L2 승인/L3 준법큐/L4 차단) 실제 실행 결과.
- L0~L4 네이밍 충돌 해소안에 대한 팀 합의(디자이너·빌더 포함) 기록.
