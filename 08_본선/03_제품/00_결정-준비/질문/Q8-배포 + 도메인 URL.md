# [Q8] 배포 + 도메인 URL — 후보·추천

## 배경·현재가안

7/1에는 “배포 여부 미정”, 7/3에는 “데모 URL 필요”만 재확인된 상태라, 심사위원에게는 **실동작 MVP를 어디서 어떻게 볼 수 있는지**가 아직 약하다. D13은 발표장에서 “좋은 답변”보다 **정답셋·정책룰·감사로그로 검증 가능한 상태 변화**를 보여야 한다고 정리한다. 특히 `입력 → 모델판단 → 근거 → 승인게이트 → 상태변화 → 감사로그` 프레임과 Playwright e2e, `verify_static.py`를 검증 증거물로 격상하라고 제안한다. (D13)

현재 SSOT 커널은 `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`이고, 승인레벨은 L0~L4, 4함수계약은 정적 MVP에서 서버 API 승격 목표다. PII 4중방어와 “고객 원본 PII·신용정보 외부 프런티어 LLM 반출 금지”가 핵심 제약이다. 금융권 AI 도입은 PoC→파일럿→내부 정식 적용→대고객 서비스화로 단계화되고, 대고객 노출 전환 시 법적 책임이 급격히 커진다. (D+b)

따라서 URL은 단순 호스팅 문제가 아니라 **“공개 데모 URL은 안전한 정적/합성 데이터만, 로컬모델 실동작은 통제된 하이브리드 데모로 증명”**하는 배포 전략 문제다.

## 후보 A·B·C

### 후보 A. 로컬 전용 데모: `localhost` 실행 + 발표장 화면 공유

- 데이터 근거: D13은 deterministic 골든패스, Before/After, 실패·차단 케이스 노출, e2e 검증을 요구한다. 로컬 전용은 PII 비반출과 로컬모델 실동작을 가장 강하게 통제할 수 있다. (D13)
- 출처 D코드: D13, D+b
- 장점: 원본 PII 외부 반출 금지 원칙과 가장 잘 맞는다. 로컬모델, 감사로그, 승인게이트, 차단 API 403 같은 “실동작”을 발표자가 안정적으로 통제할 수 있다. 대고객 서비스화가 아니라 내부 PoC/파일럿 단계로 보이기 때문에 법적 노출도 낮다. (D+b)
- 단점: 7/3 요구인 “데모 URL 필요”를 충족하지 못한다. 심사위원이 사전 접속하거나 제출물에서 직접 확인하기 어렵다. 배포·완성도 항목에서 감점 가능성이 있다. “실현가능성·완성도”가 공식 평가 요소라는 근거와 충돌한다. (D13)

### 후보 B. 공개 정적 URL: GitHub Pages/Netlify/Vercel류 정적 배포 + 합성 데이터

- 데이터 근거: 현재 MVP는 `02_제품/app`의 vanilla HTML/CSS/JS 정적 앱 구조이고, D13은 `verify_static.py`와 Playwright e2e를 검증 증거물로 쓰라고 제안한다. 정적 URL은 “실제 접속 가능한 MVP” 증명에 유리하다. (D13)
- 출처 D코드: D13, D+a
- 장점: 7/5 10:30 제출 전 가장 빠르게 URL을 만들 수 있다. 서버 API 없이도 `computeRiskDecision`, `buildDashboardData`, `auditChainRecords`, `moveCaseToColumn`의 정적 함수계약과 화면 상태 변화를 보여줄 수 있다. 고객 원본 PII 없이 합성·더미 데이터만 쓰면 외부 반출 리스크를 낮출 수 있다.
- 단점: 로컬모델 실동작, 서버 API 승격, 망분리·내부망 SaaS 적합성은 “진짜 운영 배포”로 증명되지 않는다. 공개 URL에 금융·신용정보처럼 보이는 데이터가 들어가면 오해 리스크가 있다. 실제 도메인 소유, 배포 플랫폼 보안 설정, 접근제어 여부는 현재 근거 없음. (근거 필요)

### 후보 C. 하이브리드 권장형: 공개 정적 데모 URL + 발표장 로컬모델 실동작 + 감사로그 증거 패키지

- 데이터 근거: D13은 배포 환경과 유사한 조건에서 성능·보증 기준을 입증하고, TEVV 기록·incident monitoring·human-AI outcome monitoring을 남기라고 정리한다. D+a는 망분리 완화와 SaaS/생성형 AI 활용이 열리는 과도기에서 PII 비반출 거버넌스가 핵심 보안 요건이라고 본다. D+b는 PoC→파일럿→내부 정식 적용의 단계별 게이트와 HITL 하드 임계값을 강조한다. (D13, D+a, D+b)
- 출처 D코드: D13, D+a, D+b
- 장점: 공개 URL 요구와 금융 보안 제약을 동시에 만족한다. 공개 URL은 합성 데이터 기반 “심사위원 접속용”, 발표장/녹화는 로컬모델·승인게이트·감사원장 “실동작 증명용”으로 역할을 분리한다. 정적 MVP의 한계를 숨기지 않고 “서버 API 승격 목표”와 PoC 단계로 정확히 포지셔닝할 수 있다.
- 단점: 운영 메시지가 복잡해질 수 있다. “왜 공개 URL에서 로컬모델까지 다 안 되나?”라는 질문에 대비해야 한다. 커스텀 도메인, 배포 플랫폼, 접근제어, URL 만료 정책은 별도 확인이 필요하다. (근거 필요)

## 추천안

**추천: 후보 C. 공개 정적 데모 URL + 발표장 로컬모델 실동작의 하이브리드 배포**

권장 URL 포지션은 `https://demo.jb-localguard-os.kr` 또는 `https://localguard-demo.[team-domain]` 형태다. 실제 도메인 구매 가능 여부, DNS 설정, SSL, 배포 플랫폼은 현재 근거 없음. (근거 필요) 시간이 촉박하면 우선 플랫폼 기본 URL을 제출하고, 커스텀 도메인은 후속으로 붙이는 것이 안전하다. 플랫폼명별 보안·약관 비교는 현재 입력 근거에 없음. (근거 필요)

깐깐한 금융 심사위원 렌즈 통과 이유:

1. **페인포인트가 실재·정량인가**: 전북은행 연체율 1.46%, NPL 1.12%, 지방은행 평균 대비 악화, RM 효율·대손 방어 ROI 모델이 제시되어 있다. 단, ROI는 명시적 가정 기반 추정치로 말해야 한다. (D+a)
2. **적법성**: 공개 URL에는 합성 데이터만 올리고, 원본 PII·신용정보는 로컬모델/내부 통제 경계 안에서만 처리한다. PII 4중방어, HITL, 감사원장은 금융 AI 가이드라인·AI RMF·망분리 완화 흐름과 맞닿는다. 신용정보법 §40조의2 직접 해석 근거는 입력에 부족하다. (D+b, 근거 필요)
3. **신뢰성·정확성**: 공개 데모는 `Case→Audit` 상태 변화와 GENESIS 해시체인을 보여주고, 발표장에서는 일부러 L3/L4 차단 케이스를 넣어 “하나 놓치면?”에 답한다. D13은 FN/FP, precision/recall, approval invariant, citation coverage를 함께 제시하라고 한다. (D13)
4. **ROI가 5천만원급인가**: D+a의 기준 시나리오는 초기 구축비 5억 원, 연간 순효과 113.5억 원, 보수 시나리오도 50.05억 원 순효과로 산정한다. 단, 실측값이 아니라 가정 기반 민감도 분석임을 명시해야 한다. (D+a)
5. **실동작·배포로 증명되나**: 공개 URL로 “접속 가능한 완성도”를, 로컬 데모로 “원본 PII 비반출·로컬모델·승인게이트 실동작”을 증명한다. D13의 “실제 동작 가능한 MVP” 요구와 공식 평가의 실현가능성·완성도 기준에 가장 잘 맞는다. (D13)
6. **확장성**: PoC→파일럿→내부 정식 적용→대고객 서비스화 단계로 승격 가능하고, 전북은행에서 JB우리캐피탈로 확산할 때도 공동 플랫폼+로컬 데이터 격리 구조를 설명할 수 있다. (D+b)

## 리스크·의존성

- 공개 URL에 실제 고객형 PII, 신용정보, 원본 문서가 들어가면 안 된다. 합성 PII/honeytoken만 사용하고 외부 반출 로그 0건을 증거로 남겨야 한다. (D13)
- “100% 보장” 표현은 금지에 가깝다. “사전 정의된 평가 범위에서 0 failures observed”로 표현해야 한다. (D13)
- 커스텀 도메인, SSL, 배포 플랫폼, 접근제어, URL 만료 정책은 현재 근거 없음. (근거 필요)
- 신용정보법 §40조의2, PIPA, 망분리 예외 적용 가능성은 법무·보안팀 확인이 필요하다. 입력 근거는 방향성은 있으나 조문별 적용 판단까지는 부족하다. (D+b, 근거 필요)
- 로컬모델 실동작이 발표장 네트워크·장비에 의존하면 장애 리스크가 있다. 정적 URL, 녹화본, Playwright/e2e 결과, 감사로그 샘플을 백업으로 준비해야 한다. (D13)
- 전세·피싱 중 1개 실동작 조건부 일정이 7/4라면, 7/5 제출본에는 “실동작 완료/미완료” 상태를 절대 과장하지 말아야 한다. (근거 필요)

## 팀 후속 딥리서치 프롬프트

JB LocalGuard OS의 7/5 제출용 배포 전략을 검증해줘. 후보는 ① 로컬 전용 데모 ② 공개 정적 URL ③ 공개 정적 URL+로컬모델 하이브리드다. GitHub Pages, Cloudflare Pages, Netlify, Vercel의 금융 PoC 적합성, 무료/유료 제한, 비공개 배포·접근제어, 커스텀 도메인·SSL, 로그/PII 리스크, 국내 금융권 망분리·PIPA·신용정보법 §40조의2 관점 리스크를 비교해줘. 실제 제출 전 24시간 내 가능한 실행 절차와 체크리스트도 포함해줘.

## 🔎 심사위원 반박 & 보강

**반박 요지**

1. **미실행 상태의 구체적 확정처럼 보인다.** `https://demo.jb-localguard-os.kr`이라는 특정 도메인이 본문·추천안에 등장하지만, 문서 스스로 "실제 도메인 구매 가능 여부, DNS 설정, SSL, 배포 플랫폼은 현재 근거 없음(근거 필요)"이라고 인정한다. 소유·구매 확인 없이 특정 문자열을 추천안 헤드라인에 박아두면, 검토자가 이미 확보된 자산으로 오독할 위험이 있다. 저장소 전체를 grep해도 실제 배포(GitHub Pages/Netlify/Vercel 등)나 도메인 확보 흔적은 전혀 없다 — C안은 현재 순수 계획일 뿐, 실행된 적이 없다.
2. **하이브리드의 "실동작" 다리가 다른 미결정 질문에 의존한다.** C안의 차별점은 "발표장 로컬모델 실동작"인데, 같은 폴더의 `Q7-로컬모델+API+OAuth 구동.md`는 로컬모델 후보(Gemma/EXAONE/Qwen)조차 확정하지 못한 상태다(딥리서치 프롬프트로 위임됨). 7/4~5 정읍 발표까지 로컬모델이 확정·구동되지 않으면 C안은 B안(공개 정적 URL만)으로 사실상 강등되는데, 이 의존성과 강등 시나리오가 본 문서에 명시되어 있지 않다.
3. **일정 리스크가 과소평가돼 있다.** 오늘(7/3) 기준 제출은 7/5 10:30, 발표는 7/4~5 정읍이다. 도메인 구매+DNS+SSL+정적 배포+합성데이터 생성·검증+로컬모델 확정·구동을 병행할 실행 오너·시간표가 이 문서에 없다. "시간이 촉박하면 플랫폼 기본 URL을 우선 제출"이라는 완화책은 있지만, 그 경우 추천안의 커스텀 도메인 문구 자체가 무의미해지므로 사실상 기본 추천을 B안 수준으로 되돌리는 셈인데 이 모순이 정리되지 않았다.
4. **합성 데이터의 안전성이 검증되지 않은 주장이다.** "합성 PII/honeytoken만 사용", "외부 반출 로그 0건을 증거로" 라고 쓰여 있지만, 합성 데이터 생성 방법·실제 PII 미포함 검증 절차(예: 정규식/PII 탐지기 스캔, 실명·주민번호 패턴 대조)가 전혀 제시되지 않았다. "적법성"과 "신뢰성" 두 심사 축 모두 이 지점에서 근거가 비어 있다.
5. **적법성 결론이 유보된 채 실행 계획에 흡수돼 있다.** 신용정보법 §40조의2, PIPA, 망분리 예외 적용 판단은 "법무·보안팀 확인 필요"로 남겨져 있는데, 정작 추천안 채택 자체는 이 확인 없이 이미 확정형 문장("추천: 후보 C")으로 제시된다. 법적 판단이 나오기 전에 배포 방식을 확정하는 순서가 거꾸로다.
6. **공개 URL의 가용성·상태 관리 리스크가 빠졌다.** MVP는 `localStorage` 기반 정적 상태(키 `jb-localguard-os-state-v2`)로 동작한다. 심사위원이 제출 후 임의 시점에 URL을 재방문하거나 새로고침하면 데모 시나리오 상태가 리셋되거나 어긋날 수 있다는 점, 그리고 발표장 네트워크가 아닌 심사위원 자택/사무실 접속 시 URL 가용성(무료 티어 다운타임 등)에 대한 리스크가 문서에 없다.

**판정: 약함(weak)**

하이브리드 배포 전략의 논리(공개 URL=합성데이터·완성도 증명, 로컬 데모=PII 비반출·실동작 증명)와 PII 비반출 원칙과의 정합성은 타당하다. 그러나 ① 실제 실행(도메인·배포·합성데이터 검증)이 전혀 없고 ② 핵심 의존성(Q7 로컬모델 확정)이 미결이며 ③ 법적 판단 전에 결론을 확정한 순서 오류가 있어, 현재 상태로는 "추천안"이 아니라 "가설"에 가깝다. 통과하려면 아래 보강이 필요하다.

**보강 필요항목**

- 도메인 실제 구매/보유 여부 확인 — 없다면 추천 문구에서 특정 도메인명을 제거하고 "플랫폼 기본 URL"로 낮춰 표현
- Q7(로컬모델 확정)과의 의존 관계 명시 및 "Q7 미확정 시 C→B 강등" 폴백 조건을 본 문서에 교차 기재
- 도메인·배포·합성데이터·로컬모델 확정을 누가 언제까지 하는지 오너·마감이 담긴 실행 체크리스트(시간 단위)
- 합성 데이터 생성 방법과 PII 미포함 검증 절차(스캔 도구/방법) 구체화
- 신용정보법 §40조의2·PIPA·망분리 예외 적용에 대한 법무/보안 확인 결과 — 확인 전까지는 추천안을 "조건부"로 하향 표기
- 공개 URL의 상태 관리(새로고침 시 데모 리셋 여부)·가용성(무료 티어 다운타임) 리스크와 대응책

**통과하려면 추가로 필요한 데이터/실험**

- 실제 정적 배포 1회 시연(Netlify/Vercel/Pages 중 택1) 스크린샷 또는 URL — "실동작·배포로 증명되나" 항목을 근거 있는 사실로 전환
- Q7 로컬모델 확정 결과 및 7/4 발표장 구동 성공/실패 여부 기록
- 합성 데이터 PII 스캔 결과(0건 근거)
- 법무/보안 담당자의 §40조의2·PIPA 적용 가능성 서면 확인(또는 최소 팀 내 리스크 수용 결정 기록)
