# [Q7] 로컬모델+API+OAuth 구동 — 후보·추천

## 배경·현재가안

Q7의 핵심은 “로컬모델이 원본 PII·신용정보를 내부에서 처리하고, 외부 API/프런티어 LLM은 Zero-PII 초안 보조로만 쓰며, OAuth/OIDC 기반 권한·승인·감사를 묶어 실제 구동을 증명할 수 있는가”다.

현재 가안은 `Claude/Codex + Ollama` 하이브리드, 1차 로컬모델은 `Gemma` 시도다. `EXAONE/Qwen` 최종 비교, 로컬모델 구동 PC 사양, OAuth/OIDC 연동 방식은 회의록상 확정 근거가 없으므로 `(근거 필요)`로 둔다.

SSOT는 `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`이고, 이는 금융위 AI 가이드라인의 “AI는 보조수단, 최종 책임은 임직원” 원칙 및 승인책임자/사람 통제 요구와 정합적이다. (D16, D17, D5b)

## 후보 A·B·C

### 후보 A. 로컬 우선: `Ollama + Gemma + 내부 API + OAuth 승인 게이트`

- **데이터 근거·출처 D코드**
  - 원본 PII·개인신용정보 외부 반출 금지는 금융권 SaaS/생성형 AI 예외의 핵심 조건이다. 고유식별정보·개인신용정보 처리에는 일반 SaaS 망분리 예외가 적용되지 않는다. (D5b)
  - 그룹 공통 레이어에는 원본 고객 원장, 주민번호, 계좌번호, 상담 전문이 아니라 `subject_token`, `risk_code`, `score_band`, `evidence_pointer`, `approval_state` 같은 최소 필드만 올려야 한다. (D17)
  - RM 업무에서 AI가 줄일 수 있는 영역은 초안 작성·증빙 대조·누락 탐지·기록 작성이며, 최종 승인·예외 판단은 사람이 남아야 한다. (D16)
  - `Gemma` 선택의 성능·정확도·금융 한국어 적합성 비교 근거는 제공 리서치에 없음. `(근거 필요)`
  - OAuth/OIDC 구체 구현 근거는 제공 리서치에 없음. `(근거 필요)`

- **장점**
  - 심사위원에게 가장 방어적인 구조: “원본은 로컬, 외부는 비식별”을 실제 시연으로 보여줄 수 있다.
  - PII 4중방어와 Case→Approval→Audit 계약을 제품 구조로 직접 증명한다.
  - 7/5 제출 전 MVP 범위에 맞다. Ollama 로컬 호출, API 래퍼, 승인 로그만 붙여도 실동작 메시지가 선명하다.

- **단점**
  - 로컬 PC/GPU 성능이 병목이다. 승권님 액션아이템 미완이면 데모 리스크가 크다.
  - Gemma가 금융 문서 요약·한국어 판단에서 충분한지 아직 근거가 없다.
  - OAuth까지 완전 구현하려면 시간이 빠듯하다. 데모에서는 mock OAuth 또는 local session + role claim으로 축소할 가능성이 있다.

### 후보 B. 외부 API 우선: `Claude/Codex/프런티어 LLM API + Prompt Proxy + Zero-PII`

- **데이터 근거·출처 D코드**
  - 외부 생성형 AI·SaaS는 샌드박스와 내부업무망 SaaS 예외로 조건부 활용 경로가 열렸다. (D5b)
  - 외부 LLM에는 고객명, 주민번호, 계좌번호, 개인신용정보를 보내면 안 되며, 가명정보도 별도 혁신금융서비스 경로가 더 안전하다. (D5b)
  - 프롬프트 프록시는 PII 탐지, 토큰 치환, 반출 스캔, 차단 로그를 수행해야 한다. (D17)

- **장점**
  - 추론 품질과 데모 안정성은 가장 높을 가능성이 크다.
  - Codex/Claude로 초안·설명·리스크 문안 생성 품질을 빠르게 확보할 수 있다.
  - “외부 API를 쓰되 PII는 안 나간다”는 정책 정합성이 D5b와 잘 맞는다.

- **단점**
  - 로컬모델 실동작이라는 Q7의 핵심 차별성이 약해진다.
  - 프롬프트에 PII가 섞이는 순간 심사위원 질문에 취약하다.
  - 금융권 실제 배포 관점에서는 SaaS 분류, 위탁계약, 샌드박스/보안평가가 추가 검토 대상이다. (D5b)

### 후보 C. 모델 비교형: `Gemma vs EXAONE vs Qwen` 로컬 스냅샷 벤치마크 후 고정

- **데이터 근거·출처 D코드**
  - 로컬 원본 처리와 외부 비식별 보조 구조 자체는 D17/D5b로 방어 가능하다.
  - 그러나 `EXAONE`, `Qwen`, `Gemma`의 금융 한국어 성능, PII 추출률, 지연시간, 하드웨어 요구량 비교표는 제공 리서치에 없다. `(근거 필요)`
  - 로컬모델 구동용 컴퓨터 사양도 확정 근거가 없다. `(근거 필요)`

- **장점**
  - 기술 심사위원에게 “모델 선택을 감으로 하지 않았다”는 인상을 준다.
  - 한국어·금융문서·지연시간 기준으로 최종 모델을 고정하면 배포 설득력이 오른다.
  - 추후 본선/PoC 확장에는 가장 좋은 방식이다.

- **단점**
  - 7/5 10:30 제출 전에는 과하다.
  - 벤치마크 데이터셋, 평가 기준, GPU 환경이 없으면 오히려 미완성으로 보인다.
  - 회의록에 후보군 비교가 없으므로 지금 발표자료에 확정처럼 쓰면 근거 취약하다.

## 추천안

**추천: 후보 A를 메인으로 하고, 후보 B를 보조 API 경로로 숨긴 하이브리드.**  
즉, `로컬 Ollama/Gemma = 원본 PII 처리·요약·리스크 초안 1차`, `외부 Claude/Codex API = 비식별 템플릿·문안 보강`, `OAuth/OIDC 또는 mock role = RM/준법/상위승인자 권한`, `Audit = GENESIS 해시체인`으로 시연한다.

깐깐한 금융 심사위원 렌즈로 보면:

① **페인포인트가 실재·정량인가**  
기업여신 심사 리드타임은 평균 15일, 품의/심사의견 작성은 30분→10초 AI 단축 사례, 서류 입력은 400분→50분 또는 20분→6분 단축 사례가 있다. RM 시간당 원가도 전북은행 평균 근로소득 9,619만~9,966만 원, 기준 시간당 7.5만~8.25만 원 수준으로 방어 가능하다. (D16)

② **적법성**  
원본 PII·개인신용정보는 로컬모델이 처리하고, 외부 API에는 비식별 요약만 보낸다. 이는 신용정보법·PIPA·금융지주회사법상 원장 미제공, 암호화/변환, 목적 제한, 내부 경영관리 목적 통제와 맞다. (D17) 망분리 측면에서도 외부 SaaS/생성형 AI는 조건부 경로가 있지만 고유식별정보·개인신용정보 반출은 금지에 가깝다. (D5b)

③ **신뢰성·정확성**  
하나 놓치면 금융사고가 된다. 그래서 AI는 `computeRiskDecision`으로 L0~L4 제안만 하고, L1/L2는 RM 검토·승인, L3는 준법, L4는 차단·상위검토로 멈춘다. 최종 책임은 임직원이라는 금융 AI 원칙과 맞다. (D16, D5b)

④ **ROI가 5천만원급인가**  
RM 10명 파일럿만 잡아도 보수~기준 시나리오에서 연 9,900만~1.8억 원 절감 추정이 가능하고, 50명 기준은 4.95억~9억 원까지 제시된 바 있다. 단, 케이스 수·실현율은 내부 실측 전까지 `[추정]`으로 표시해야 한다. (D16)

⑤ **실동작·배포로 증명되나**  
데모는 “전주 카페 대출심사” 케이스에서 원본문서 입력 → 로컬모델 처리 → 외부 API 전송 전 PII 차단 로그 → Evidence 생성 → RM 승인 → Audit 해시체인까지 이어져야 한다. 로컬모델 실동작 PC가 미확정이면 가장 큰 의존성이다. `(근거 필요)`

⑥ **확장성**  
같은 커널이 SME 여신, 전세 위험, 보이스피싱/FDS에 공통 적용된다. 세 도메인은 모두 “증빙 교차검증 → 리스크 등급 → 사람 승인 → 감사 증적” 구조라 Case/AgentRun/Evidence/Approval/Audit로 확장 가능하다. (D16, D17)

## 리스크·의존성

- 로컬모델 구동 PC/GPU 사양 미확정. 데모 실패 리스크 1순위. `(근거 필요)`
- `Gemma` 우선 선택의 정량 비교 근거 부재. EXAONE/Qwen 대비 성능·속도·라이선스 비교 필요. `(근거 필요)`
- OAuth/OIDC 실제 연동 범위 미확정. 제출 전에는 `role-based mock auth`라도 감사로그에 `operator_id`, `approval_role`, `action_type`을 남겨야 한다.
- 외부 API 사용 시 PII 반출 스캔이 반드시 보여야 한다. 차단 로그 없이 “안 보낸다”고 말하면 설득력이 약하다. (D5b, D17)
- ROI 숫자는 강하지만 케이스/인, 실현율, handoff 재작업 시간은 공개 실측이 부족하다. `[추정]` 라벨 유지 필요. (D16)
- L3/L4 승인·차단 정책은 시연용 룰과 실제 금융회사 내규가 다를 수 있다. 내부 법무·준법 검토 필요. (D17)

## 팀 후속 딥리서치 프롬프트

JB LocalGuard OS의 로컬모델 후보를 Gemma, EXAONE, Qwen 중심으로 비교하라. 한국어 금융문서 요약, PII 탐지/마스킹, 여신·전세·FDS 리스크 사유 생성, 8GB/16GB/32GB GPU 또는 CPU 구동 지연시간, Ollama/llama.cpp 지원성, 라이선스·상업 이용 가능성, 오프라인 배포 난이도, OAuth/OIDC 연계 시 감사로그 설계까지 표로 정리하라. 근거 없는 항목은 반드시 `(근거 필요)`로 표시하고, 7/5 MVP용 1순위 모델과 본선용 교체 후보를 분리 추천하라.

## 🔎 심사위원 반박 & 보강

**반박 요지**

1. **"실동작 증명"의 핵심 전제 3개가 모두 `(근거 필요)`다.** 로컬모델 구동 PC/GPU 사양, `Gemma` 채택의 정량 비교, OAuth/OIDC 연동 방식 — 이 셋은 후보 A의 방어 논리("원본은 로컬, 외부는 비식별을 실제로 보여준다")를 성립시키는 전제 그 자체다. 전제가 셋 다 미확정이면 "실동작 증명"이 아니라 "실동작 계획"이다. 심사위원이 "그래서 지금 로컬모델이 도는 걸 보여줄 수 있느냐"고 물으면, 이 문서 자체가 "아니다, PC 사양부터 정해야 한다"고 답하게 되어 있다.
2. **ROI 근거(④, D16)를 인용하면서 SSOT의 자체 적대검증 결과를 반영하지 않았다.** `08_본선/05_제출/리서치-딥프롬프트/_결과/_적대검증-리포트.md`는 D16(RM 원가·절감시간)을 "고(高)" 심각도 불일치로 분류한다 — 시간당 비용이 `5.5만원/7.5만원/8.25만원/6.8~12.9만원`으로, 서류처리 단축이 `400분→50분` 대 `20분→6분`으로 회수본마다 갈리며, 결론은 "단일 고정값을 버리고 범위형 시나리오로만 사용. 실제 제출 수치화는 웹 재검증 필요"다. 이 문서 ④는 "9,619만~9,966만 원, 7.5만~8.25만 원 수준으로 방어 가능하다"고 서술하는데, 9,619만(D16-geminipro, 전북은행)과 9,966만(D16-geminiflash, 전북은행)은 서로 다른 LLM 회수본의 수치를 하나의 범위처럼 이어붙인 것이고, 그 회수본 간 불일치가 이미 팀 자체 적대검증에서 "고" 등급으로 지적된 상태다. 이 사실을 언급하지 않고 "방어 가능하다"로 마무리한 것은 근거의 강도를 실제보다 세게 보이게 한다.
3. **OAuth/OIDC는 사실상 이름만 걸려 있다.** 리스크 섹션은 이미 "제출 전에는 `role-based mock auth`라도"라고 스스로 인정한다. 그렇다면 소제목·추천안 문구의 "OAuth/OIDC"는 7/5 제출 시점 기준으로 오해를 부르는 표현이다. mock auth로 갈 것이 거의 확실하다면 문서 제목·추천안에서부터 "OAuth/OIDC(또는 role-based mock)"로 조건부 표기해야, 나중에 발표자료에 "OAuth 구동"이라는 문구가 그대로 옮겨가는 사고를 막는다.
4. **후보 C(모델 비교형)를 "과하다"며 기각했지만, 후보 A 채택의 전제인 `Gemma` 우위는 후보 C가 만들 근거 없이는 방어되지 않는다.** 즉 문서는 후보 C의 비교 작업 자체는 필요 없다고 하면서, 그 비교가 주는 결론(Gemma가 맞는 선택)은 이미 전제로 쓰고 있다. 최소한 "왜 EXAONE·Qwen이 아니라 Gemma로 가안을 잡았는가"에 대한 1줄 근거(설치 용이성, 팀 기존 경험, 라이선스 등 무엇이든)가 없으면, 심사위원에게는 "임의 선택"으로 보인다.
5. **적법성(D17) 인용 자체는 정확하다.** `D17-결과-gpt55high.md`를 대조한 결과 "원장 미제공", "고유식별정보 암호화/변환", `subject_token/risk_code/score_band/evidence_pointer/approval_state` 최소 필드 구성은 원문과 일치한다. 이 부분은 왜곡이 없다.

**판정: 약함 (weak)**
구조(로컬 원본 처리 + 외부 비식별 보조 + Approval/Audit)와 적법성 인용(D17, D5b)은 유지할 만하다. 그러나 실동작 증명·ROI 방어 강도가 스스로 인정한 것보다 더 셀 미확정 항목(PC 사양, 모델 비교, OAuth) 위에 서 있고, ROI 수치는 팀 자체 적대검증에서 이미 "고" 위험으로 지정된 범위를 재검증 없이 "방어 가능"으로 서술했다. 추천안(후보 A 메인 + B 보조 하이브리드) 방향 자체를 기각할 근거는 없지만, "실동작·배포 증명"이라는 핵심 주장은 현재 근거로는 완전하지 않다.

**보강 필요 항목**
- 로컬모델 구동 PC/GPU 실측 사양(승권님 액션아이템) 확정 — 없으면 데모 리스크 1순위 항목이 그대로 심사 리스크가 됨.
- `Gemma` 채택 사유 1줄이라도 명문화(비교표가 없다면 최소 "왜 지금은 Gemma인가"의 임시 근거).
- ROI 문구에 "회수본 간 시간당 비용·처리시간 불일치(고 위험, 웹 재검증 필요)" 캐비엇을 명시적으로 추가 — 현재는 "[추정]" 라벨이 케이스 수·실현율에만 걸려 있고, 근본 수치 자체의 회수본 간 불일치는 언급되지 않음.
- 문서 제목·추천안의 "OAuth/OIDC" 표기를 실제 7/5 구현 범위(role-based mock)와 일치시킬지 여부 결정.

**통과하려면 추가로 필요한 데이터/실험**
- 로컬 PC/GPU에서 `Gemma`(혹은 대안) 1회 이상 실추론 지연시간 측정값(예: 전주 카페 케이스 1건 처리 시간).
- D16 세 회수본(geminipro/geminiflash/기타)의 시간당 원가·처리시간 차이를 웹 재검증으로 좁힌 단일 범위(적대검증 리포트가 요구한 작업).
- OAuth/OIDC 실제 구현 여부에 대한 팀 결정(진짜 구현 vs mock) — 결정 없으면 발표자료·질의응답에서 문구 불일치 위험.
