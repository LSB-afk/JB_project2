# [Q6] 보안 연결 아키텍처 — 후보·추천

## 배경·현재가안

7/1 회의에서 "보안을 어떻게 할지, 로컬/외부 모델 미정"으로만 논의되었고, 7/3 회의에서 "전세 실거래가·건물값 외부 API 연결 시도"가 액션아이템으로만 잡혀, 보안 연결 아키텍처 자체는 아직 미확정 상태다. (근거: 회의록 원문 — 리서치 D코드 없음, 회의 결정사항 자체)

현 가안(SSOT):
- 운영계약 Case→AgentRun→Agent→Skill→Evidence→Approval→Audit, 승인레벨 L0(관찰/로그)–L1/L2(RM 검토/승인)–L3(준법)–L4(차단·상위검토)
- PII 4중방어: 데이터 등급제·토큰화·모델 라우팅·반출 스캔 + 감사원장
- "고객 원본 PII·신용정보는 외부 프런티어 LLM에 반출하지 않고 로컬모델이 원본을 처리한다"는 원칙만 있고, 이를 구현하는 **연결 아키텍처(로컬-외부 경계를 어디서 어떻게 긋는가, 외부 API를 어떤 형태로 붙이는가)**는 결정되지 않았다.

## 후보 A·B·C

### 후보 A — 망분리 유지 + SaaS 예외 경로 (원본 내부, 가명·비식별만 외부)

내부업무망은 전자금융감독규정 제15조제1항제3호에 따라 원칙적으로 외부통신망과 분리·차단하되, 2026-04-20 시행세칙 개정으로 정식화된 **SaaS 예외**를 활용한다. 외부 프런티어 LLM을 SaaS 형태로 이용하고, 단말 보호대책·FSEC 등 평가·반기 1회 평가·정보보호위원회 보고를 전제로 한다. 단, 이 경로는 **고유식별정보·개인신용정보를 처리하는 경우 예외가 허용되지 않는다** — 즉 원본 PII/신용정보는 절대 이 경로로 못 나간다. 가명정보를 보내려면 SaaS 일반예외가 아니라 별도의 **혁신금융서비스 지정**이 필요하다. (D5b, 허용금지매트릭스)

- 장점: 정책적으로 가장 보수적이고 방어 가능한 경로. 금융위·금감원이 이미 M365·Copilot·HyperCLOVA X·AWS Bedrock·Azure OpenAI 등을 "내부업무용 단말기에서 SaaS 및 생성형 AI 이용" 혁신금융서비스로 지정한 선례가 있어 심사위원에게 "제도화된 경로를 따랐다"고 설명 가능. (D5b)
- 단점: SaaS 예외는 개인신용정보·고유식별정보를 원천 배제하므로, 전세 실거래가·건물값 API처럼 **주소·물건 식별정보가 포함된 외부 조회**는 이 카테고리로 단순 정리되지 않는다 — 별도 검토 필요(근거 필요, D5b는 SaaS AI 이용 매트릭스이지 부동산 공공API 연동 자체를 다루지 않음).

### 후보 B — 데이터 등급제 + 토큰화 분리보관 + 모델 라우팅 게이트 (제품 핵심 4중방어의 구현체)

신용정보법 §40조의2①②⑧과 PIPA §28조의4·시행령 제29조의5에 근거해 **토큰↔원본 키를 분리보관**하고, 외부 LLM 라우트에는 원본 개인식별자·계좌식별자·주민번호·연락처·재결합 키를 절대 포함하지 않는다. 데이터 분류는 최소 "원본 개인신용정보 / 가명정보 / 익명정보 / AI 입력 불가 / AI 입력 가능"으로 나누고, 라우팅 게이트가 이 등급에 따라 로컬모델(원본 처리) vs 외부 프런티어 LLM(가명·비식별만) 경로를 강제한다. 외부 전송 전후로 반출 스캔(주민번호 패턴, 주소 조합, 준식별자 조합 재식별 위험 탐지)을 수행하고, 재식별 위험 발견 시 신용정보법 §40조의2⑦·PIPA §28조의5②에 따라 즉시 처리중지·회수·삭제한다. 감사원장은 "누가 어떤 프롬프트를 만들었나"가 아니라 "어떤 데이터가 어떤 가명처리 규칙·어떤 외부모델 라우트·어떤 승인·어떤 스캔 결과를 거쳐 나갔는지"를 3년 이상 보관되는 법적 증빙 형식으로 남긴다. (D5a — 데이터 등급제·토큰화·모델 라우팅·반출 스캔·감사원장 전 항목)

- 장점: 이미 콘솔 스펙의 "PII 4중방어" 원칙과 1:1로 대응되어 **추가 설계 없이 기존 4함수계약(auditChainRecords 등)에 바로 얹을 수 있다.** 법조문(§40조의2 각 항)을 직접 인용해 심사위원에게 "엔지니어링 편의가 아니라 법이 요구하는 통제"라고 말할 수 있는 문장까지 이미 확보됨(D5a "제품 차별점 문장" 섹션). 데이터가 실재하는 정량 근거(제재·과징금 사례표, D5a)로 페인포인트를 뒷받침 가능.
- 단점: 라우팅 게이트·반출 스캔의 **오탐/미탐 정확도**가 검증되지 않으면 "형식만 통제, 실질은 구멍"이 될 위험(D15 — rubber-stamping·과잉차단 실패모드). 전세·건물값 외부 API처럼 **인바운드(외부→내부)** 연결에는 이 프레임(주로 아웃바운드 PII 반출 통제)이 그대로 적용되지 않아 별도 설계가 필요.

### 후보 C — 하이브리드: 로컬모델이 원본 오케스트레이션, 외부 API/LLM은 "읽기전용 데이터 소스"로만 편입

전세 실거래가·건물값 조회처럼 **외부에서 데이터를 받아오는(inbound) API**는 PII 반출 문제가 아니라 별개 트랙으로 분리한다. 로컬모델이 Case의 원본 데이터(고객 PII 포함)를 들고 있는 상태에서, 필요한 시점에만 주소·물건 식별자 등 **비-PII 조회 파라미터만** 외부 공공/시세 API로 보내고, 응답(시세·등기 정보)을 다시 로컬 파이프라인에 결합한다. 이는 후보 B의 모델 라우팅 게이트를 "LLM 프롬프트 반출"뿐 아니라 "구조화 API 호출 파라미터 반출"까지 확장한 형태다. 다만 국토연구원·국토부 자료(D15)는 실거래가·공시가격 데이터 자체의 품질 편차(시세-공시 괴리, 수기입력 오류)를 지적하고 있어, 외부 데이터를 그대로 신뢰하면 안 되고 **검증 절차가 필요**하다고 명시한다.

- 장점: 전세 시나리오의 "외부 API 연결 시도"라는 실제 액션아이템과 가장 직접적으로 맞물린다. PII는 로컬에 고정, 외부는 순수 조회 소스로 최소화해 반출 표면적을 줄인다.
- 단점: 외부 실거래가/시세 API의 **데이터 신뢰성 자체가 낮다**는 것이 정책기관(국토연구원·국토부)의 공식 인정 사항이므로(D15), "외부 API로 실시간성/정확성을 확보했다"는 주장은 검증 없이 하면 반박 소재가 된다. 데모 시연에서 이 API가 죽거나 느리면 하이브리드 전체가 흔들리는 단일 장애점(SPOF) 위험(근거 필요 — 해당 공공API의 가용성 SLA는 리서치 없음).

## 추천안

**후보 B(4중방어 라우팅 게이트)를 뼈대로 하고, 전세 도메인의 외부 조회만 후보 C의 "비-PII 파라미터 한정 인바운드" 규칙으로 감싸는 조합**을 추천한다. 후보 A(SaaS 예외)는 정책 정합성 설명 문구로만 인용하고, 실제 아키텍처 결정으로 채택하지 않는다 — 이유는 아래 심사위원 렌즈로 설명한다.

① **페인포인트 실재·정량성**: 원본 PII 외부 반출 금지 요구는 회의록(7/1)에서 이미 확인된 팀 자체 요구이고, 법적 근거(신용정보법 §40조의2, PIPA §28조의4)도 D5a에 조문 단위로 확보되어 있다. 정량 근거(제재·과징금 사례표)도 D5a에 있어 "왜 이게 중요한가"를 숫자로 말할 수 있다.
② **적법성**: 후보 B는 신용정보법 §40조의2 각 항, PIPA §28조의4·시행령 29조의5를 직접 인용해 통제 항목(분리보관·모델라우팅·반출스캔·3년 보존 감사원장)을 조문에 1:1 대응시킨다. 후보 A(SaaS 예외)만으로는 개인신용정보 자체가 원천 배제되어, 여신 심사처럼 신용정보를 다루는 히어로 시나리오(전주 카페 대출심사)에는 적용되지 않는다 — 그래서 A를 단독 채택안이 아니라 보조 설명으로만 쓴다.
③ **신뢰성·정확성**: D15가 지적하는 실패모드(rubber-stamping, 과잉차단, kill switch 미연결)를 승인레벨 L1~L4 + 감사원장 append-only 구조로 흡수한다고 명시하면, "하나 놓치면?"에 대한 답(즉시 처리중지·회수·삭제 트리거, 반출 스캔)이 이미 설계에 내장된다.
④ **ROI**: 4중방어는 이미 콘솔 스펙 문구로 존재해 신규 개발 비용이 "게이트 로직 + 스캔 룰" 정도로 제한적이다. 5천만원급 임팩트는 "형사·행정 제재 리스크 회피"라는 정량 근거(D5a 제재표)로 뒷받침 가능.
⑤ **실동작·배포 증명**: 전세 도메인이 [조건부] 실동작 후보이므로, 후보 C의 "비-PII 파라미터만 외부 API로" 규칙을 실제 코드(예: moveCaseToColumn/computeRiskDecision 호출 전 파라미터 마스킹)로 시연 가능.
⑥ **확장성**: 라우팅 게이트는 도메인(여신/전세/피싱)과 계열사(전북은행/JB우리캐피탈)에 무관하게 동일 구조로 재사용된다.

## 리스크·의존성

- 전세 실거래가/건물값 공공API의 **데이터 품질 자체가 정책기관에 의해 편차·오류가 있다고 공식 인정**된 상태(D15, 국토연구원·국토부) — 시연에서 "실거래가 정확도"를 과신하는 발언은 피해야 함.
- 반출 스캔·모델 라우팅 게이트의 **오탐/미탐 정확도는 이번 리서치로 검증되지 않음**(근거 필요) — 데모에서는 룰 기반 최소 구현으로 시연 범위를 한정할 것.
- 후보 A(SaaS 예외)와 후보 B(가명처리 경로)는 법적으로 **서로 다른 트랙**(SaaS 일반예외 vs 혁신금융서비스 지정)이라 혼용해서 "SaaS니까 가명정보도 보낼 수 있다"고 말하면 오류(D5b, 허용금지매트릭스에서 명시적으로 구분).
- 외부 API 연동은 아직 "액션아이템"일 뿐 실제 연결 코드/계약 여부 미확인 — 7/4 조건부 실동작 목표와의 일정 의존성 있음.

## 팀 후속 딥리서치 프롬프트

"한국 금융권이 부동산 시세·실거래가 공공API(국토부 실거래가 공개시스템, 건축물대장 등)를 내부업무망과 연동할 때, 이를 전자금융감독규정상 '외부통신망 연결'로 보아 망분리 예외절차(SaaS 예외 또는 R&D망 예외)를 거쳐야 하는지, 아니면 단순 공공데이터 조회로 예외 적용 대상이 아닌지 확인해줘. 특히 (1) 국토부 실거래가 API가 개인신용정보나 고유식별정보를 포함하지 않는 순수 시세조회로 분류되는 근거, (2) 해당 API의 가용성 SLA·응답지연 시 대체 데이터소스 정책, (3) 2026-04-20 SaaS 예외 시행세칙 개정 원문에서 '데이터 조회형 API'가 'SaaS 이용'과 동일하게 취급되는지 여부를 법령·감독규정·금융위 보도자료 원문으로 확인해줘. 결과는 조문·보도자료 링크와 함께 정리."

## 🔎 심사위원 반박 & 보강

### 반박 요지

1. **④ ROI 근거의 인과관계 오류**: 추천안 ④는 "5천만원급 임팩트는 '형사·행정 제재 리스크 회피'라는 정량 근거(D5a 제재표)로 뒷받침 가능"이라고 썼다. 그런데 D5a 원문 스스로 "신용정보법 §40조의2 가명정보 재식별 위반에 대한 공개 제재 사례는 이번 조사에서 찾지 못했다"고 명시한다(D5a-결과.md L141). D5a 제재표에 실린 롯데카드 96.2억·카카오페이 83.75억·다이렉트보험 92.08억은 모두 **PIPA상 주민등록번호 처리·안전조치·동의·국외이전** 위반 사례이지, 이 문서가 채택한 §40조의2(가명처리·재식별) 위반 사례가 아니다(D5a-결과.md L32, L73). 즉 "우리가 만드는 통제(§40조의2 준수)를 안 하면 저 금액대 제재를 받는다"는 인과관계는 D5a 자신이 부인한 것을 Q6가 뭉뚱그려 재활용한 것이다. 정량 ROI 근거로 쓰려면 §40조의2 자체의 집행 사례나, 최소한 "유사 규모 미비 시 예상 제재 범주(3천만원 과태료/3% 과징금 조문 자체)"로 재서술해야 한다.

2. **⑤ 실동작·배포 증명이 아직 미래형 서술**: 추천안 ⑤는 "후보 C의 규칙을 실제 코드로 시연 가능"이라고 가능성만 말한다. 반면 `08_본선/_분석/JB-콘솔-프로토타입-스펙.md`(L83)에는 "PII거버넌스(4중방어)·감사체인(auditChainRecords)"가 이미 **실동작·페이지에러 0, 커밋 2975443**으로 완료됐다고 기록되어 있다. Q6는 이 기존 구현을 전혀 인용하지 않는다 — 그 결과 (a) 이미 있는 실동작 증거를 스스로 누락해 근거를 약화시켰거나, (b) 그 기존 구현이 정말 이 문서가 말하는 "라우팅 게이트+반출 스캔"과 1:1로 대응하는지 확인되지 않은 채 별개로 "시연 가능"이라 주장하는 것이다. 어느 쪽이든 이 문서만 읽으면 실동작 여부를 판단할 수 없다 — 근거 연결 누락.

3. **미검증 항목이 결정을 흔들 만큼 많다**: 문서 자체가 인정한 미검증이 최소 4개다 — (a) 반출 스캔·라우팅 게이트의 오탐/미탐 정확도, (b) 전세 실거래가 공공API의 가용성 SLA, (c) MLS(다층보안) 용어의 1차 문서 확인 여부(D5b L30, "이번 조사 기준 미확인"), (d) 인바운드 API 결합 시의 법적 프레임 자체가 D5b·D5a 어디에도 다뤄지지 않음(후보 B/C 모두 "별도 검토 필요"로만 적음). 각각이 단독으로는 치명적이지 않지만, 데모 시연(전세 도메인)이 이 네 가지 미검증 지점을 모두 통과해야 하는 경로에 있다는 점에서 리스크가 누적된다.

4. **"5천만원 벌금" 프레이밍의 오독 위험**: 배경에서 인용된 신용정보법 형사처벌 "5년 이하 징역 또는 5천만원 이하 벌금"(D5a L28)은 재식별을 **고의로 시도한 자**에 대한 처벌이지, 회사가 반출 스캔을 안 갖췄을 때 자동으로 물리는 액수가 아니다(그 경우는 3천만원 과태료 또는 매출액 3% 과징금, D5a L28). 발표 자리에서 "5천만원 벌금 리스크를 막는다"고 잘못 말하면 조문을 부정확하게 인용한 것이 되어 오히려 신뢰도 감점 요인이 된다.

### 판정: **약함(weak)**

적법성 축(②)은 D5a·D5b 조문 인용이 정확하고 SSOT 정합적이라 탄탄하다. 그러나 ROI(④) 근거는 D5a가 스스로 부인한 인과관계를 재사용한 오류이고, 실동작 증명(⑤)은 이미 있을 수 있는 증거를 인용하지 않은 채 미래형으로만 서술되어 있어 "증명됐다"고 보기 어렵다. 기각까지는 아니지만, 이 두 축을 보강하지 않으면 심사에서 "법 조문은 많이 인용했는데 실제로 뭘 만들었고 그게 정말 그 돈을 아끼는지는 불명확하다"는 반박에 답할 수 없다.

### 보강 필요 항목

- ④ ROI 문장에서 D5a 제재표(PIPA 사례)를 §40조의2 리스크 근거로 쓰는 것을 중단하고, §40조의2 자체의 법정형(3년/3천만원 과태료, 매출액 3% 과징금)으로 교체해 인과관계를 정확히 할 것.
- ⑤ `JB-콘솔-프로토타입-스펙.md`의 기존 "PII거버넌스(4중방어)/감사체인(auditChainRecords) 실동작·커밋 2975443"이 이 문서가 말하는 후보 B 라우팅 게이트와 동일 구현인지 대조하고, 동일하면 그 커밋·페이지를 이 문서에 직접 인용해 실동작 근거로 승격할 것. 다르면 "무엇이 이미 있고 무엇이 없는지"를 표로 분리할 것.
- 반출 스캔·라우팅 게이트의 오탐/미탐율을 최소 1회 이상 룰 기반 최소 구현으로 실측(예: 준식별자 조합 N개 샘플 통과율)해 "근거 필요"를 해소할 것.
- 후보 A/B/C 어디에도 없는 "인바운드 API(전세 실거래가) 자체가 망분리 예외 대상인가"를 팀 후속 리서치 프롬프트(문서 하단에 이미 초안 있음)로 실제 발주해 회수할 것 — 현재는 질문만 있고 답이 없다.
