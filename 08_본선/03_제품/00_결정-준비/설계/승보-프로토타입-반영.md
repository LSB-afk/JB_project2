## 3. 하네스 시스템 계약 (`harnessCore.js` / `harnessRegistry.js` / `harnessVerification.js`)

- **harnessCore.js** — business 로직 0줄. manifest 등록소(`registerHarness`/`getHarness`/`listHarnesses`), hook 실행기(`harnessRunHooks`, 위반 로그 최근 200건 보관), 공통 guardrail 검사 유틸 4종(`harnessGuardCheckPII`/`harnessGuardCheckAssertions`/`harnessGuardCheckScope`/`harnessGuardCheckAutoClose`/`harnessGuardCheckApprovalRequired`), 신규 역할 스캐폴드 기술자 `describeNewHarnessScaffold(roleKey)`.
- **harnessRegistry.js** — 두 하네스(jeonse-protection / jb-woori-capital)를 표준 manifest로 등록만 한다. manifest는 `id·kind·scopeKey·scopeValue·displayName·routeBase·sidebarConfig·countService·searchService·caseCreationFlow·agents·skills·commands·hooks·rules·guardrails·verification`(16키). **main/default/safety 이름의 alias 하네스는 존재하지 않음** — 만들면 verify 실패.
  - jeonse-protection: `enforceHooks: true`, requiredHooks 3종(beforeCaseCreate/beforeAgentRun/beforeCustomerMessage), requiredAgents 10, requiredCommands 6.
  - jb-woori-capital: `enforceHooks: false`(**TODO 명시** — "hook 연결은 다음 단계"), requiredAgents 13, requiredCommands 0. → JBWC는 전세보호보다 하네스 성숙도가 한 단계 낮다(hook 미배선).
- **harnessVerification.js** — ECC의 Smoke Test/AgentShield 관점을 6개 검증기로 구현: `verifyHarnessIntegrity`(manifest 필수키+count/search 함수+전용wizard+skills≥1) · `verifyRoleHarnessScope`(scopeProbe: 미지정 조회 시 예외 발생 확인) · `verifyNoForbiddenRoleResurface`(삭제 역할/광주은행 레일/label-only 심볼 재유입 검사) · `verifyNoPIILeakage`(저장소 직렬화+DOM innerText 스캔) · `verifyAgentRegistryCompleteness`(agent 수 + 11개 필드 + 공통 blockedActions 전원 포함) · `verifyHookCoverage`(enforceHooks 시 hook/command 연결 확인). `runHarnessSelfTest(harnessId)`가 6개를 집계해 `window.__lastHarnessSelfTest`에 저장 — 브라우저 콘솔/명령(`/jeonse-run-smoke-test`)에서 실행 가능한 **런타임 자체 검증 루프**.

## 4. ROLE_HARNESS_CONTRACT / SECURITY_GUARDRAILS 핵심 규칙

### 4.1 ROLE_HARNESS_CONTRACT.md
- 기준 구현은 `jeonse-protection`. **템플릿 복사-붙여넣기 alias 금지** — 신규 역할은 `describeNewHarnessScaffold`가 기술한 9종 파일을 각자 새로 작성.
- 신규 역할 추가 8단계: scaffold 체크 → 전용 파일세트(config→agents+skills→rules+hooks→scope강제db→services→helpers/views→commands→sidebar→router) → index.html 로드순서(core→config→registry→rules→db→services→views→commands→sidebar→router) → harnessRegistry 등록(enforceHooks:true, requiredHooks≥3) → app.js 배선 → verify_static.py 갱신 → e2e+smoke 추가 → 문서 갱신.
- 금지 alias 패턴: 타 하네스 business 함수/상수 참조(`jbwcTable(`/`JBWC_*`/`jpoTable(`/`JPO_*`(자기 외)/메인 `agents` 배열/`roleDashboardPage(`) · `mainHarness`/`defaultHarness`/`safetyHarness` alias · scope 없는 전체조회 · label-only 대시보드 회귀 · 삭제 축 재유입(`consumer-protection-dashboard`/`compliance-dashboard`/광주은행).
- **다음 역할 확장 후보로 문서가 이미 제안**: RM(scope `rm-officer`, 히어로 전주카페 seed 승격), 기업여신 담당자(전결 L1~L4), 보이스피싱/FDS 담당자(기존 label형 fds-dashboard를 이 계약으로 승격), JB우리캐피탈(hook 배선 남음).

원칙: "차단 가능 지점(생성·발송·반출)은 차단+감사기록, 이미 발생한 지점은 안전강등+reviewRequired 감사기록." 모든 훅 실행은 `harnessHookLog()`에 남는다.

## 5. DB/데모데이터 구조·시드값

### 5.1 JB우리캐피탈 (`wooricap-db.js`, `JBWC_DB_KEY = "jbwc-ops-db-v3"`)
- `jbwcRepository`: `table(name, affiliateId)`(affiliateId 필수, 미지정 시 예외) / `insert` / `nextId` / `reset` / `snapshot`.
- affiliates 2건(jb-woori-capital + jeonbuk-bank 검증용), users 12명("김OO" 익명 표기, role=ops/review/compliance/risk/support), ops_cases 32건(11도메인 커버, `caseSpecs` 배열에 도메인/상품/제목/상태/priority/riskLevel 명시) + **`CASE-OTHER-0001`(affiliateId=jeonbuk-bank, tags:["exclude"])** — count/search 어디에도 노출되면 안 되는 스코프 격리 검증용 seed.
- 파생 테이블(docs/03 §7): ops_tasks 18·approvals 3·audit_logs 5·privacy_permission_checks 3·external_connectors 5·ai_analysis_requests 3·ai_recommendations 5·business_capabilities 11·role_assignments 12·inspection_schedules 3·customer_support_cases 4·consumer_protection_reviews 3·fds_alerts 3·document_cases 4·vehicle_lifecycle_tasks 4·agent_runs 8·agent_handoffs 3·kpi_snapshots 10·harness_agents 13(레지스트리 동기화).
- ID 규약 `<TYPE>-JBWC-<seq>`(`jbwcNextId`).

### 5.2 전세보호 (`jeonseProtection-db.js`, `JPO_DB_KEY = "jpo-ops-db-v1"`)
- `jpoRepository` 동일 인터페이스, scope 키는 `roleKey`.
- role_workspaces 2건(jeonse-protection=active + fds=roadmap 예고), users 8명, jeonse_cases 20건(업무유형10 커버, 보증금 band 4단계) + **`JEONSE-OTHER-0001`(roleKey=fds-officer)** 격리 검증용 seed. 모든 개체는 익명 Ref(TENANT-REF-*/CONTRACT-REF-*/PROPERTY-REF-*/LANDLORD-REF-*/ADDRESS-REF-*)만 저장 — 실명/주민번호/전화/계좌/등기원문 필드 자체가 없음.
- 파생: jeonse_tasks 12·risk_assessments 7(preContract4+vulnerable3)·price 4·registry 4·guarantee 3·victim 3·referrals 6(auction2·legal2·care2)·alerts 3(high2·critical1)·approvals pending3(+승인1)·audit reviewRequired5·connectors5(degraded1·down1)·analysis3·recommendations5·runs6·handoffs3·capabilities10·role_assignments8·inspections3.

두 DB 모두 **localStorage 손상/버전 불일치 시 자동 재시드**, 저장소 유일 계약(operator interface)이 명문화되어 있어 운영 DB 전환 시 이 인터페이스만 서버 API로 교체하면 되는 구조.

## 6. 우리 볼트 반영 시사점 (+ `_canon.md`/키스톤 정합·충돌)

### 6.1 정합(그대로 재사용 가능)
- **운영 계약 동형**: Case→AgentRun→Agent→Skill→Evidence→Approval→Audit 계약이 우리 canon과 완전히 동일한 패턴으로 JBWC·전세보호 양쪽에 재구현되어 있음 — [[키스톤-확정]]의 "공통 뼈대는 한 번, 직군별 에이전트셋·화면·데이터만 스왑"이라는 **모듈식 확장성** 논거를 승보 코드가 이미 실증. `07_계열사-하네스/_HARNESS-WOORICAP.md`에 이 실증 사례를 "이미 동작하는 코드 근거"로 인용 가능.
- **PII 4중 방어 계층**(훅/검색차단/런타임스캔/정적게이트)이 우리가 발표에서 주장하는 "PII 비연결·감사우선" 원칙과 1:1 대응 — `04_tech`/보안 문서에 그대로 인용 가능한 구체 사례.
- **harnessCore/Registry/Verification의 3분리**(엔진/등록/검증)는 `02_agent-design`에 "신규 역할 확장 시 반드시 지킬 아키텍처 원칙"으로 채택할 가치가 큼. 특히 `describeNewHarnessScaffold`가 "복붙 alias 금지"를 코드로 강제하는 방식은 우리 에이전트 설계 문서에 없는 규율 — 편입 권고.
- **JBWC 하네스 = 캐피탈 콘솔의 실질 완성본**: [[이승보-구현-설계-정리]]가 회의록 기준 "일정 때문에 잠시 중단"이라고 말한 것과 달리, 코드는 13에이전트·24뷰·32케이스가 상당히 완결된 상태. `01_prd/mvp-scope.md`가 "캐피탈 콘솔 미완성 리스크"로 기술했다면 갱신 필요.
