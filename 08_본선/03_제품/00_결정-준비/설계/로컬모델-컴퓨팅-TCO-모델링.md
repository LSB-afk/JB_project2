## 2. TCO 수학 모델

은행 측 실제 GPU 보유량, 서버 가동률, 내부망 정책, 대상 RM 수를 알 수 없으므로 본 모델은 **저/중/고 3개 시나리오**로 둔다. 공개/입력 근거가 있는 값은 그대로 쓰고, 없는 값은 [추정]으로 분리한다.

### 2.1 업무량 시나리오

D16은 대상 RM 인원수를 보수 10명, 기준 50명, 공격 100명으로 제시했다(D16). 또 RM 1인당 연간 적용 케이스는 180/360/600건 범위로 제시되며(D16), 다른 D16 결과는 기준 시나리오에서 600건/년, 케이스당 0.5시간 절감, 실현율 60~90%를 사용했다(D16). 본 문서는 컴퓨팅 TCO용 호출량 산식이 목적이므로 다음처럼 둔다.

| 시나리오 | RM 수 $R$ | RM당 연 케이스 $V_y$ | 월 케이스 $N_m$ | 일 케이스 $N_d$ | 근거 |
|---|---:|---:|---:|---:|---|
| 저 | 10명 | 180건/년 | 150건/월 | 약 7.5건/영업일 [추정: 월 20영업일] | RM 10명, 180건/년(D16) |
| 중 | 50명 | 360건/년 | 1,500건/월 | 약 75건/영업일 [추정] | RM 50명, 360건/년(D16) |
| 고 | 100명 | 600건/년 | 5,000건/월 | 약 250건/영업일 [추정] | RM 100명, 600건/년(D16) |

토큰/건은 두 레벨로 둔다.

- **일반 문서 보조형:** 입력 3,000토큰, 출력 700토큰, 합계 3,700토큰/건 [추정]이며 D23-gpt55pro의 worked example 값이다(D23).
- **대형 RAG 심사형:** 입력 100,000토큰, 출력 5,000토큰, 합계 105,000토큰/건 [추정]이며 D23-gemini의 worked example 값이다(D23).

### 2.2 추론량 산식

월간 케이스 수는 다음과 같다.

$$
N_m = \frac{R \cdot V_y}{12}
$$

케이스당 입력/출력 토큰을 $t_{in}, t_{out}$이라 하면 일일·월간 토큰량은 다음과 같다.

$$
T_{m,in}=N_m \cdot t_{in}, \quad T_{m,out}=N_m \cdot t_{out}
$$

$$
T_m=T_{m,in}+T_{m,out}=N_m(t_{in}+t_{out})
$$

$$
T_d=\frac{T_m}{D_m}
$$

여기서 $D_m=20$영업일/월은 [추정]이다.

| 시나리오 | 일반 문서 보조형 월 토큰 | 대형 RAG 심사형 월 토큰 |
|---|---:|---:|
| 저 | 55.5만 tokens/월 [추정] | 1,575만 tokens/월 [추정] |
| 중 | 555만 tokens/월 [추정] | 1.575억 tokens/월 [추정] |
| 고 | 1,850만 tokens/월 [추정] | 5.25억 tokens/월 [추정] |

### 2.3 로컬 GPU 서버 TCO 함수

로컬 서버 월 TCO는 다음과 같이 둔다.

$$
C_{local,m}=
\frac{Capex_{sys}}{L_m}
+
PUE \cdot \frac{P_{IT}\cdot u \cdot h_m}{1000}\cdot p_{kWh}
+
OPEX_{ops,m}
+
C_{maint,m}
+
C_{quant,m}
$$

- $Capex_{sys}$: GPU 서버·CPU·RAM·NVMe·네트워크·랙 부대비 포함 도입가 [추정].
- $L_m$: 감가상각 개월 수. 36개월 [추정: D22/D23의 36개월 상각 예시와 정합].
- $PUE$: 1.3~1.4 [추정]. D22는 1.3, D23은 1.4 예시를 사용(D22/D23).
- $P_{IT}$: 서버 IT 부하 kW [추정].
- $u$: 평균 이용률 [추정].
- $h_m$: 월 가동시간. 720시간 [추정: 24×30].
- $p_{kWh}$: 전력단가. D22는 KEPCO 산업용(갑)Ⅱ 고압A 선택Ⅱ 전력량요금 53.70~107.70원/kWh 및 기본요금 7,140원/kW를 인용했고, 계산 예시에는 80원/kWh [추정]을 사용했다(D22). D23은 173원/kWh 또는 185.5원/kWh 시나리오를 사용했다(D23). 본 문서는 2026-07-03 기준 **보수 80원/kWh, 기준 173원/kWh, 고 185.5원/kWh [추정]**으로 민감도 처리한다.
- $C_{quant,m}$: 양자화 품질 저하 대응 비용. D23은 $C_{quant}=N\cdot \Delta err\cdot t_{correct}\cdot C_i$를 제시했다(D23). 실제 파일럿 전에는 [추정]으로만 둔다.

기존 서버 활용 시나리오와 신규 GPU 구비 시나리오는 다음처럼 분리한다.

$$
C_{reuse,m}=C_{power,m}+C_{ops,m}+C_{quant,m}
$$

$$
C_{new,m}=C_{reuse,m}+\frac{Capex_{sys}}{L_m}+C_{maint,m}
$$

따라서 신규 구비의 추가 비용은 다음과 같다.

$$
\Delta C_{new-reuse,m}=\frac{Capex_{sys}}{L_m}+C_{maint,m}
$$

### 2.4 API 비용 함수

프런티어 API 비용은 입력/출력 단가를 분리한다. D23도 단순 $T_{month}\cdot p_{token}$ 대신 입력·출력·캐시를 분리해야 한다고 정정했다(D23).

$$
C_{api,m}^{USD}
=
\frac{T_{m,in}}{10^6}p_{in}
+
\frac{T_{m,out}}{10^6}p_{out}
+
F_{api,m}
$$

프롬프트 캐싱을 반영하면 다음과 같다.

$$
C_{api,m}^{USD}
=
\frac{T_m}{10^6}
\left[
\alpha_{in}\{(1-\beta)p_{in}+\beta p_{cache}\}
+
(1-\alpha_{in})p_{out}
\right]
+
F_{api,m}
$$

$$
C_{api,m}^{KRW}=FX\cdot C_{api,m}^{USD}
$$

D22는 GPT-4.1 mini 입력 $0.40/1M, 출력 $1.60/1M, GPT-5.5 입력 $5.00/1M, 출력 $30.00/1M을 2026-07-01 접속 기준으로 인용했다(D22). D23-gpt55pro는 GPT-5 mini 입력 $0.25/1M, 출력 $2.00/1M, GPT-5 입력 $1.25/1M, 출력 $10.00/1M, Claude Sonnet 4 입력 $3.00/1M, 출력 $15.00/1M을 2026-07-01 접속 기준으로 인용했다(D23). 환율은 D22에서 1 USD ≈ 1,350 KRW [추정]을 사용했고(D22), D23-gpt55pro는 1,533~1,549원을 인용했다(D23). 본 문서의 산식 예시는 2026-07-03 기준 **1,350원 [추정: D22와 비교 가능성 유지]**을 사용한다.

### 2.5 로컬 vs API 손익분기점

로컬 월 고정비를 $C_{local,m}$, API의 1M tokens당 가중 원화 단가를 $p_{api,KRW}^{1M}$라 하면 손익분기 월 토큰량은 다음과 같다.

$$
T_m^*
=
\frac{C_{local,m}}{p_{api,KRW}^{1M}}
\cdot 10^6
$$

케이스 기준 손익분기점은 다음과 같다.

$$
N_m^*
=
\frac{T_m^*}{t_{in}+t_{out}}
$$

D22는 4xH100 70B 시나리오에서 GPT-4.1 mini와 비교 시 월 약 97억 토큰이 필요하고, GPT-5.5와 비교 시 70/30 입력·출력 가정에서 월 약 5.9억 토큰, 50/50 가정에서 약 4.2억 토큰으로 내려간다고 계산했다(D22). D23-gemini도 8xH100 자체 구축 손익분기점을 월 97.1억 토큰, 케이스당 10.5만 토큰 기준 월 92,538건으로 산출했다(D23). D23-gpt55pro는 Sonnet 4 기준 월 8.17억 토큰, GPT-5 mini 기준 월 74.1억 토큰을 제시했다(D23).

따라서 본선 메시지는 다음이 방어적이다.

| 비교 기준 | 손익분기 요지 | 해석 |
|---|---|---|
| 저가 API 대비 | 월 수십억~약 97억 tokens 필요(D22/D23) | 지역은행 부서 단위에서는 순수 비용 기준 로컬이 불리 |
| 고가 프런티어 API 대비 | 월 약 4.2억~5.9억 tokens(D22) 또는 Sonnet 4 기준 약 8.17억 tokens(D23) | 고 시나리오의 대형 RAG 5.25억 tokens/월 [추정]부터 검토 가능 |
| 기존 서버 활용 | $Capex/L_m$ 제거 | 은행 내부 GPU가 이미 있고 보안상 로컬 필수라면 경제성 급개선 |
| 신규 GPU 구비 | $Capex/L_m+C_{maint}$ 추가 | 대량·반복·민감 업무가 없으면 과투자 위험 |

### 2.6 GPU 권장 사양과 VRAM 산정

모델 가중치 메모리는 단순 근사로 계산한다.

$$
M_{weights}\approx P_{model}\cdot b_q
$$

- BF16: $b_q=2$ bytes [추정].
- FP8: $b_q=1$ byte [추정].
- INT4/Q4: $b_q=0.5$ byte [추정].
- 실제 필요 메모리:

$$
M_{total}=M_{weights}+M_{KV}+M_{runtime}+M_{batch}
$$

$M_{KV}$는 컨텍스트 길이·동시성·배치에 따라 변동하므로 제출 전 실측 필요 [추정].

| 배포 규모 | 권장 GPU [추정] | 근거 |
|---|---|---|
| 7B~8B 단일 모델 | L40S 48GB, A6000 48GB, 또는 은행 보유 동급 GPU [추정] | INT4 가중치 3.5~4GB 수준 [추정], 긴 컨텍스트·동시성 여유까지 고려 |
| 14B~32B | L40S/A6000 1장 이상, 필요 시 2장 [추정] | 32B INT4 약 16GB, FP8 약 32GB(D22의 VRAM 근사) |
| 70B | 2~4 GPU 서버, 예: 4xH100 또는 2xH200 [추정] | D22는 Llama 3.1 70B 4xH100, Llama 3.3 70B 2xH200 벤치마크를 인용(D22) |
| 고처리량/전사 공용 | H100/L40S 다중 서버 [추정] | 활용률과 동시성 목표가 핵심. D22는 전기료보다 GPU 활용률과 모델 크기가 TCO 핵심이라고 정리(D22) |

## 3. 웹 배포로 로컬모델을 구동할 수 있는가

결론: **정적 웹사이트 배포(GitHub Pages 등)로는 브라우저에서 로컬 LLM을 구동할 수 없다.**

WebGPU/WASM 기반 브라우저 내 추론 기술은 이론적으로 존재하지만, 본 프로젝트의 정적 배포 방식, 7B~14B급 모델 크기, 고객 PII 비반출 요건, 브라우저 메모리·성능 편차를 고려하면 본선 제출용 아키텍처로 비현실적이다 [추정]. 특히 정적 페이지는 서버 프로세스, 모델 파일 관리, GPU 드라이버, 내부망 접근통제, 감사로그, 모델 라우팅 정책을 안정적으로 제공하지 못한다.

```text
        v
[API Gateway / Policy Router]
  - PII 탐지
  - 호출 로그
  - 모델 라우팅
  - 외부 전송 차단/허용 판단
        |------------------------------|
        |                              |
        v                              v
[Claude API / Codex API]          [Local LLM Runtime]
- PII 없는 일반 추론          - EXAONE/Qwen2.5
- 문서 초안/검증              - 팀원 MacBook Pro M4 Pro 데모
- 코드 보조                   - 실 배포 시 은행 내부망 GPU 서버
- 공개정보 요약               - PII 포함 업무 전용
```

실 배포형 내부망 구조는 다음과 같다.

```text
[RM Browser / Internal Web App]
        |
        v
[Bank Internal API Gateway]
        |
        +--> [PII Detector + Policy Engine]
        |          |
        |          +-- PII detected --> [On-prem Local LLM Server]
        |          |                    [Evidence Store]
        |          |                    [Approval Gate]
        |          |                    [Audit Ledger]
        |          |
        |          +-- No PII --------> [Approved External API Proxy]
        |
        v
[Human Approval]
        |
        v
[Core Banking / CRM / Case System]
```

즉 웹은 “모델을 직접 돌리는 장소”가 아니라 **정책·승인·증적 UI**이고, 로컬모델은 데모에서는 팀원 MacBook Pro M4 Pro에서, 실 배포에서는 은행 내부 서버에서 구동해야 한다.

## 4. M4 Pro 데모 실행 권고

decision-log는 “Ollama+한국어 로컬모델”을 확정 후보로 기록했고, 히어로를 EXAONE 3.5 7.8B, 상업 백업을 Qwen2.5 7B로 남겼다(decision-log). 2026-07-02 로그도 “데모 구동 머신 별도. 추천은 EXAONE 3.5 7.8B/Qwen2.5-14B(M4 Pro 기준)”이라고 정리했다(decision-log).

### 권장 실행 경로

1. **Ollama 경로:** 설치와 실행이 단순하고, 본선 시연에서 “실제 로컬모델 응답”을 보여주기 쉽다(decision-log).
2. **MLX 경로:** Apple Silicon Metal 가속에 더 최적화될 수 있으나, 준비 시간이 부족하면 Ollama를 우선한다 [추정].
3. **데모 모델 순서:** EXAONE 3.5 7.8B → Qwen2.5 7B → Qwen2.5 14B 순으로 리스크를 낮춘다 [추정].

### 권장 양자화

| 모델 | 권장 양자화 | 예상 메모리 | 예상 속도 | 비고 |
|---|---|---:|---:|---|
| EXAONE 3.5 7.8B | GGUF Q4_K_M 또는 MLX 4-bit [추정] | 가중치 약 3.9GB + KV/runtime 포함 8~16GB [추정] | 10~30 tok/s [추정: M4 Pro급 로컬 7B 경험칙, 입력 파일 내 실측 없음] | 데모 히어로. 라이선스상 실 배포 비권고 |
| Qwen2.5 7B | GGUF Q4_K_M/Q5_K_M [추정] | Q4 약 3.5GB, Q5 약 4.4GB + runtime [추정] | 10~35 tok/s [추정] | 권고 배포 메시지와 연결 |
| Qwen2.5 14B | GGUF Q4_K_M [추정] | 가중치 약 7GB + KV/runtime 포함 16~24GB [추정] | 5~18 tok/s [추정] | 품질 백업. M4 Pro 메모리 용량 확인 필요 |
| Gemma 2/3 소형 | 4-bit [추정] | 모델 크기에 따라 2~8GB대 [추정] | 15~40 tok/s [추정] | 보조 백업 |

M4 Pro 데모에서는 긴 RAG 컨텍스트를 무리하게 넣지 말고, **PII 감지 → 로컬모델 라우팅 → 근거 요약 → 사람 승인 대기**의 짧은 경로를 보여주는 것이 안정적이다. 로컬 추론 속도 자체보다 “외부 API로 민감정보를 보내지 않는다”는 정책 실행이 시연의 핵심이다.

## 5. Claude/Codex 폴백 라우팅

원칙: **PII가 포함된 요청은 반드시 로컬모델로만 처리한다. PII가 없는 일반 추론·검증·문서 작업은 Claude API/Codex API로 폴백할 수 있다.**

D22는 금융권의 현실적 해법을 “대량·반복·민감 업무는 로컬, 희소·고난도는 외부 premium”으로 정리했다(D22). 또한 OpenAI API가 기본적으로 학습에 쓰이지 않더라도 최대 30일 보관 가능성이 있어 은행 기준에서는 여전히 외부 전송이라는 점을 지적했다(D22). 따라서 외부 API는 **비식별·비민감·승인된 작업**에 한정한다.

### PII 탐지 기준 [추정]

정규식·룰 기반 1차 탐지 후, 불확실하면 로컬로 보낸다.

| 유형 | 탐지 예시 [추정] | 라우팅 |
|---|---|---|
| 주민등록번호/외국인등록번호 | `\d{6}-[1-8]\d{6}` | Local only |
| 휴대폰/전화번호 | `01[016789]-?\d{3,4}-?\d{4}` | Local only |
| 계좌번호 후보 | 은행명 + 8~16자리 숫자 패턴 [추정] | Local only |
| 카드번호 후보 | 13~19자리 숫자 + Luhn 검증 [추정] | Local only |
| 사업자등록번호 | `\d{3}-\d{2}-\d{5}` | 업무 맥락상 민감 가능성이 높아 Local 우선 |
| 고객명+주소 | 한국 주소 키워드와 이름/상호 결합 [추정] | Local only |
| 내부 문서/대출/연체/심사 키워드 | “여신”, “연체”, “신용등급”, “보이스피싱 신고”, “전세계약서” 등 [추정] | 원문 포함 시 Local only |
| 공개 통계/코드/문서 초안 | 고객 식별자 없음 | Claude/Codex 가능 |

### 라우팅 흐름

```text
[User Request]
      |
      v
[1차 PII Regex Scan]
      |
      +-- match --> [Local LLM: EXAONE/Qwen2.5]
      |
      v
[2차 업무 민감도 Classifier]
      |
      +-- sensitive or uncertain --> [Local LLM]
      |
      v
[Redaction Check]
      |
      +-- redaction failed --> [Local LLM]
      |
      v
[External Fallback Allowed]
      |
      +-- code/document/general reasoning --> [Codex API / Claude API]
      |
      v
[Response + Audit Log + Human Approval if action]
```

정책은 다음처럼 단순하게 운영한다.

$$
route(x)=
\begin{cases}
Local, & PII(x)=1 \\
Local, & Sensitive(x)=1 \\
Local, & Uncertain(x)=1 \\
External, & PII(x)=0 \land Sensitive(x)=0 \land Approved(x)=1
\end{cases}
$$

외부 호출 시에도 원문 대신 비식별 요약을 보내는 것을 기본값으로 둔다.

$$
x_{external}=Redact(x)
$$

$$
allowExternal(x)=\neg PII(x)\land \neg Sensitive(x)\land AuditEnabled
$$

감사로그에는 원문 PII를 남기지 않고, 라우팅 결과·탐지 룰 ID·모델명·승인자·타임스탬프만 남긴다 [추정]. 사람 승인 원칙은 D16의 Approval-First 및 금융위 AI 원칙과 정합적이다(D16/D22).

## 결론 요약

히어로는 EXAONE 3.5 7.8B, 실 배포 권고는 라이선스가 명확한 Qwen2.5 7B/14B이며 EXAONE은 데모 전용이다.  
정적 웹배포만으로는 로컬 LLM을 구동할 수 없고, 웹은 UI·정책 라우터, 로컬모델은 M4 Pro 또는 은행 내부망 서버에서 실행해야 한다.  
TCO는 저가 API 대비 로컬 손익분기점이 매우 늦고, 기존 GPU 활용 또는 월 수억~수십억 토큰의 민감 반복업무가 있을 때만 로컬 신규 투자가 경제적으로 방어된다.
