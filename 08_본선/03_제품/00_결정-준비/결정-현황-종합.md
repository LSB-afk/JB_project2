## ❓ 미결·열린질문

**Q1 도메인 개수·조합** — 현 가안 여신·전세사기·보이스피싱(3~4). 더 뾰족한 조합 또는 우리가 못 본 도메인? 도메인을 작게 잡지 말되(개발·배포 여력 있음) 뾰족한 이유 필수. RM 업무 적합성(Q2)과 연결. 7/2엔 4개(여신+사후관리+피싱+전세), 7/3엔 "RM·전세·피싱" 3개안 → "여신·전세·피싱" 안으로 이동, 김주용은 "네 개 다 할지 핵심만 줄일지" 7/3 시점까지도 미결로 명시.

**Q2 RM 직책 포괄 범위** — 콘솔이 RM(관계형 매니저) 직책을 완전히 포괄하나? RM 실업무 범위·타 직무와의 차이 → 도메인 가감 판단. RM 업무는 다른 직무와 동일하지 않다. 7/3에 이승보는 "RM·전세사기·보이스피싱"을 핵심 도메인으로 제안했으나 곧이어 김민주가 "RM이 포괄적이므로 RM 대신 여신을 넣자"고 반박, 팀 내 미합의 상태로 남음.

**Q3 에이전트 수·규모·구성** — 적으면 안 됨(복잡한 금융=하나 놓치면 큰 문제, 정확성·신뢰 최우선). 모듈식(담당자별 활동 에이전트 상이). 이름=한 가지 잘하는 직원처럼, 상호 협업. 에이전트 자체 신뢰도 평가(모델 평가하듯) 필요 여부. 7/1엔 "기존 14개 에이전트는 리서치 기반으로 역할에 맞게 다시 생각해야 한다"(미결), 7/3엔 "조직도상 14개와 추천 4~5개 사이에서 판단이 어려워 일단 AI에 설계를 맡기기로" 함 — 즉 7/3 회의 시점까지 팀 합의로 확정된 숫자는 없음.

**Q4 5천만원 페인포인트·ROI** — 이 아이디어가 5천만원 상금급 문제/페인포인트를 건드리나? 실제 손실·비용 정량 + 에이전트 도입 시 비용절감·생산성향상 정량. 깐깐한 금융 심사위원이 데이터로 납득할 근거. 회의록에는 "업무 에이전트 자동화가 실제 은행 업무에 적합한지 확신이 부족하다"(7/2, 김주용)는 자기 의심만 있고 정량 근거는 아직 없음.

**Q5 은행 DB 연결 방식** — 은행 업무는 문서 기반·상당 자동화. 전산 입력정보를 에이전트가 함께 읽어 고객 메모리/근거문서로 활용·작성 가능해야. 연결 가능성을 어떻게 시연할지, 빠른 실현가능성 테스트 방법. 7/1엔 "데이터 API나 MCP"로 방향만 제시, 7/2엔 "실제 은행 업무 방식과 고객/케이스 메모리 관리가 현장에서 어떻게 되는지 알 수 없다"고 명시적 미결, 7/3엔 "DB 연동을 정적으로 할지, 실제 서버를 만들지, 하이브리드로 할지" 김주용·이승보 논의가 필요한 채로 남음.

**Q6 보안 연결 아키텍처** — 스킬·플러그인·MCP·금융 플러그인·외부DB·정책·금융뉴스·법령·실거래가·전세가격·국토부 오픈데이터·공개 신뢰데이터를 안전하게 연결하고 실제로 보여줘야. 모든 연결이 보안상 안전해야 한다. 7/1엔 "보안을 어떻게 할지, 모델을 로컬로 쓸지 외부 Claude 등을 쓸지 미정"이었고, 7/3엔 "전세 실거래가, 건물값 등 외부 API 연결 가능하게 시도"가 액션아이템으로 잡혔을 뿐 아키텍처 자체는 미확정.

**Q7 로컬모델+API+OAuth 구동** — AI 에이전트 모델이 실제 콘솔에서 구동 + 결과물 산출. 로컬모델(EXAONE/Qwen 후보)·프런티어 API·OAuth 모델 조합. 어느 모델·구성이 최적인가. 7/3엔 "Claude/Codex 연결 + Ollama 로컬 모델 하이브리드"가 유력 방향으로 제시되고 "Gemma 등으로 먼저 시도"가 액션아이템으로 잡혔으나, EXAONE/Qwen 등 확정 스냅샷이 언급하는 후보군과의 최종 비교는 회의록에 없음. 컴퓨팅 자원(로컬 모델 구동용 컴퓨터)은 승권님 액션아이템으로만 남아 있음.

**Q8 배포 + 도메인 URL** — 시험용 배포 + 제출용 도메인 배포 URL 필요. 정적/서버/보안 고려한 배포 방식 후보. 7/1엔 "시연은 배포할지, 로컬에서 돌아가게 할지 아직 정하지 않았다", 7/3엔 "데모 URL도 필요하다"고 재확인만 되었을 뿐 배포 방식 후보 비교는 없음. (리서치가 얇으면 500자 프롬프트를 강화해 팀이 결정하게.)

**Q9 스토리·공감(체스판)** — 공감을 이끌 우리 스토리·고객 스토리를 체스판에 올린다. 발표 전략·전술. 히어로=전주 카페 대출심사, 전세/피싱 피해 서사. 심사위원·청중 공감 포인트. 7/2에 "전주 카페, 김민주 사례 등 팀원의 개인적 스토리 활용" 방향은 나왔으나 구체적 전술(어느 시점에 어떤 스토리를 배치할지)은 미결.

**Q10 리더보드/순위 산정 기준** — 리더보드·순위가 제출 순서 기준인지, 점수 반영인지, 무작위인지 팀이 정확히 모르겠다고 명시(7/2, 이승보·김민주). 대회 안내문 재확인이 필요한 외부 규정 이슈.

**Q11 시스템 이름·브랜딩** — LocalGuard 명칭을 그대로 가져갈지, 슬로건·브랜딩을 새로 만들지 미정(7/3, 김주용). 김민주가 후보를 가져오기로 했으나 최종 결정은 미결.

**Q12 위험 업무 승인 UX 구체 스펙** — "스페이스 두 번" 또는 "키보드 스크롤 후 승인" 등 금융 위험 업무의 승인 마찰 장치가 필요하다는 방향(7/2, 김주용)만 있고, L1~L4 승인 레벨별로 어떤 UX가 붙는지 구체 스펙은 미결. 4함수계약의 `moveCaseToColumn`·승인 게이트 설계와 직결.

## 확정 스냅샷과의 상충 메모
- Q3(에이전트 수)은 확정 스냅샷의 "에이전트 14 운영 + 사람 2 승인자"가 회의록상 팀 합의가 아니라 "AI에 설계를 맡기기로" 한 잠정 위임 상태에서 나온 숫자일 가능성이 있어 **재확인 필요**.
- 그 외 항목(계열사, 하이브리드, 도메인 방향, 히어로, 제출 마감)은 회의록 흐름과 정합.
