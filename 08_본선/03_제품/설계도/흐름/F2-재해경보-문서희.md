# F2 재해경보 실데이터 — 문서희(JBG-204) 흐름

> **[대외비]** 인물·거래는 전부 합성(PII無, [[전라도-실감-페르소나-확장]] JBG-204 시드). 외부 공개 전 사람 최종승인 필요. 이 문서는 화면 자체를 다시 그리지 않고 [[01-메인-업무보드]]·[[02-케이스-상세]]·[[03-에이전트-실행뷰]]·[[04-승인-대기·결정]]의 S-ID를 그대로 참조한다.

## 1. 한 줄 시나리오 + 페르소나

문서희(45세, 전남 완도군, 전복양식 중규모 어가, 광주은행, JBG-204, Risk 86)는 고수온·태풍 경보가 겹치는 시기에 사료·산소공급 지출은 늘고 출하 정산 입금은 늦어지는 운전자금 케이스다. LocalGuard는 연체가 발생하기 전에 **실제 공공 API에서 방금 조회한 해상 특보**를 근거로 케이스를 신규 등록하고, 상환유예·정책자금 후보를 사람 승인 앞에 올려놓는다.

이 케이스가 "지역을 아는 금융 AI"를 가장 선명하게 보여주는 이유는 세 가지다. 첫째, 위험신호의 출처가 **내부 추정이 아니라 정부기관이 실시간 발표하는 공식 특보**라서 환각 방어 서사가 데모 화면에서 직접 증명된다. 둘째, 완도의 전복양식은 전남 지역성이 강해 "날씨 → 어가 현금흐름 → 여신 리스크"라는 인과가 발표에서 한 문장으로 설명된다. 셋째, F1(오태림, 실 LLM)이 텍스트 생성 신뢰를 보여준다면 F2는 **데이터 그라운딩 신뢰**를 보여줘 두 흐름이 "신뢰"라는 같은 심사축을 다른 각도에서 채운다.

## 3. 실 API 포인트 상세

### 3-1. 후보 공공 API와 검증 상태

| 후보 | 제공기관 | 확인 상태 | 근거 |
|---|---|---|---|
| **기상청_기상특보 조회서비스** (`apis.data.go.kr`, data.go.kr 카탈로그 15000415) | 기상청 | **확인됨** — data.go.kr에 등록된 실 Open API. 태풍·풍랑·강풍·폭풍해일 등 12개 현상을 178개 시군+44개 해역 단위로 특보/통보문 제공. `DATA_GO_KR_KEY` 1개로 전세 API(`RTMSDataSvcRHTrade`)와 동일 인증체계(Decoding 키, apis.data.go.kr 게이트웨이) 재사용 가능 | https://www.data.go.kr/data/15000415/openapi.do |
| 기상청_특보구역정보 조회서비스 (15126651) | 기상청 | **확인됨** — 완도 인근 해역코드 매핑용 보조 API. 해역코드→완도 권역 매핑표는 사전에 1회 구축(런타임 조회 불필요) | https://www.data.go.kr/data/15126651/openapi.do |
| 국립수산과학원 고수온특보/실시간 어장정보(RISA) | 해양수산부 국립수산과학원 | **[확인필요]** — `nifs.go.kr/risa` 웹 서비스는 실재하나, data.go.kr에 동일 데이터를 제공하는 정식 Open API 엔드포인트를 이번 조사에서 확정하지 못함. 팀 확정 전에는 F2 핵심 경로에 넣지 않는다 | https://www.nifs.go.kr/risa/ (웹, API 아님 가능성) |

**F2 핵심 경로는 확인된 기상청 해상특보 API 1개만 쓴다.** 고수온·파고 API는 스트레치 골(확보되면 "복수 기관 근거" 서사를 더 보강)로만 다루고, 확정되지 않은 API를 있는 것처럼 데모 대사에 넣지 않는다.

### 3-2. 조회 흐름

```
브라우저(?live=1) → 로컬 proxy(02_제품/scripts/api-proxy.mjs, 8020) → apis.data.go.kr (기상특보 조회서비스)
                                                                    ↳ 키 없음/장애 시 → 사전 캡처 고정 특보 JSON 폴백
```

- 기존 `/jeonse` 라우트와 동일하게 proxy에 `/marine-alert?region=완도` 라우트를 추가하는 형태로 확장한다(신규 서비스 도입 아님 — 같은 `api-proxy.mjs`, 같은 `DATA_GO_KR_KEY` 환경변수, 같은 CORS 헤더 셋업 재사용).
- data.go.kr 오픈API는 브라우저에서 직접 fetch 시 CORS가 열려 있지 않은 경우가 대부분이라(전세 API 프록시를 이미 둔 이유와 동일) 반드시 proxy를 경유한다.
- `?live=1`이 없으면 앱은 사전 캡처한 고정 특보 응답(발표일시·특보종류·해역명 하드코딩 JSON)을 그대로 렌더 — 오프라인·키 없음 상황에서도 데모가 끊기지 않는다.

### 3-3. 케이스 위험신호·EvidencePack 바인딩

1. proxy가 파싱한 특보 필드(특보종류, 해역명, 발표시각, 유효시간, 원문통보문 요약)를 그대로 `Signal` 레코드에 매핑한다 — 별도 LLM 가공 없이 **원문 그대로**가 근거의 핵심.
2. `Signal.source`에 `"기상청 기상특보 조회서비스(apis.data.go.kr)"` 문자열과 조회 타임스탬프를 고정 표기 — 심사위원이 화면에서 "이건 지어낸 문장이 아니라 API 응답"임을 즉시 알 수 있게 한다.
3. `EvidencePack`은 이 Signal을 근거탭에 그대로 인용하고, Verification Agent가 "특보 유효시간이 만기일과 겹치는가"만 재확인 — LLM이 특보 내용 자체를 재작성하지 않는다(환각 방지의 핵심 설계).

### 3-4. 환각 방어 서사

전세 슬라이스(`?live=1` → data.go.kr 실거래가)가 이미 증명한 패턴을 재사용한다: **LLM은 판단·초안 문구만 생성하고, 위험신호의 사실관계(특보 발표 여부·시각·해역)는 100% 외부 API 원문에서 가져온다.** 그래서 "AI가 재해를 예측했다"가 아니라 "AI가 정부 공식 경보를 놓치지 않고 즉시 케이스에 연결했다"는 정직한 주장만 한다.

## 4. PII·승인·감사

- 해상특보 데이터는 **지역·해역 단위 공개 데이터**이며 고객 개인정보가 아니다 — 조회·저장·화면 표시에 PII 비반출 원칙이 걸리지 않는다.
- 문서희 개인 식별정보(성명·계좌·연락처)는 어떤 단계에서도 외부 API·외부 LLM으로 나가지 않는다 — 특보 조회는 지역코드만 사용, 에이전트 판단 프롬프트는 Zero-PII 필드(케이스ID·리스크코드·만기일)만 사용.
- 상환유예·정책자금 조건변경은 **사람 최종승인**([[04-승인-대기·결정]] Decision 패널)을 거쳐야 실행되며, AI는 후보 초안까지만 만든다.
- 감사 봉인은 특보 API 호출 시각·응답 요약·승인자·승인시각을 체인 기록에 남긴다 — 단, base app만 해시체인이 실재하므로([[_설계도-INDEX]] 실측 정합 주의) F2 데모의 감사 단계는 base app 경로로만 시연한다.

## 5. 데모 스크립트(구두, 30~60초)

1. (0~10초) "완도 전복양식 어가 문서희 대표 케이스입니다. 화면에 방금 뜬 이 경보, 지금 이 순간 기상청 공공 API에서 실제로 가져온 실데이터입니다." — `?live=1` 토글 후 케이스 신규 등록 카드가 뜨는 순간을 보여준다.
2. (10~25초) 케이스 상세 위험신호 탭을 열어 "특보종류·해역·발표시각"이 그대로 찍힌 화면을 가리키며 "AI가 지어낸 문장이 아니라 정부기관 발표 원문입니다"라고 말한다.
3. (25~40초) 에이전트 실행뷰로 이동 — Marine Risk Agent → Credit Care Agent 카드가 슬라이드되며 상환유예·정책자금 후보가 만들어지는 과정을 짧게 보여준다.
4. (40~55초) 승인대기함으로 이동 — "조건 변경은 AI가 실행하지 않습니다. 사람이 최종 승인합니다"라며 3분할 결정 화면에서 승인 버튼을 누른다.
5. (55~60초) 감사기록 화면으로 컷 — "이 판단의 근거, API 호출 시각까지 전부 감사 로그에 봉인됩니다."

## 6. 수용 기준

- [ ] `?live=1` 지정 시 완도 해역 대상 기상청 해상특보 API가 실제로 호출되고, 응답(또는 무특보 상태)이 화면에 반영된다.
- [ ] `?live=1` 없이도(또는 API 키 미설정 시) 사전 캡처 고정 특보 데이터로 동일 흐름이 끊김 없이 재생된다.
- [ ] 위험신호 탭에 `Signal.source`로 API 출처명·조회 타임스탬프가 표기된다.
- [ ] 상환유예/정책자금 조건변경은 승인대기함을 거치지 않고는 "실행됨" 상태가 될 수 없다.
- [ ] 감사기록에 이 케이스의 특보 조회~승인 체인이 남는다(base app 경로 기준).
- [ ] 문서희 개인정보(성명·계좌·연락처 등)가 프롬프트 로그·네트워크 요청 어디에도 원문으로 나타나지 않는다.
- [ ] 데모 스크립트 30~60초를 실제로 리허설해 끊김 없이 재생된다.

## 7. 의존성·리스크

- **API 실시간 도달성**: 기상청 해상특보는 발표 시점이 불규칙(무특보 기간이 김) — 데모 당일 완도 해역에 활성 특보가 없을 수 있다. 대응: 사전 캡처한 "특보 발효 중" 응답을 오프라인 폴백으로 반드시 준비하고, 라이브 조회는 "지금 이 순간에도 실제로 조회한다"는 보조 증거로만 쓴다(주 서사는 폴백 화면으로 리허설).
- **CORS**: `apis.data.go.kr`는 브라우저 직접 호출이 막힐 가능성이 높다 — 반드시 `api-proxy.mjs` 경유. 신규 라우트(`/marine-alert`) 추가는 기존 `/jeonse` 핸들러 패턴을 그대로 복제하면 된다(별도 설계 불필요).
- **키 필요 여부**: `DATA_GO_KR_KEY` 1개로 기상특보 API도 신청 가능할 가능성이 높으나 **[확인필요]** — 활용신청 승인 소요시간(통상 1~2일)을 고려해 데모 전 조기 신청 필요.
- **demo:proxy 설정**: `npm run demo:proxy`가 이미 `DATA_GO_KR_KEY` 환경변수로 기동하는 구조라 F2 전용 추가 설정은 신규 라우트 코드 외에 없다.
- **오프라인 폴백**: 발표 현장 네트워크 불안정 대비, 기상특보 API 응답 샘플(특보 발효 케이스 1건 + 무특보 케이스 1건)을 사전 캡처해 로컬 고정 JSON으로 보관 — 6번 수용 기준의 "끊김 없이 재생" 항목과 동일 자산.
- **고수온·파고 API 미확정 리스크**: 3-1의 [확인필요] 2건이 확정되지 않으면 F2는 기상청 해상특보 단일 API로만 진행 — 이는 감점 요인이 아니라 "확인된 것만 정직하게 쓴다"는 원칙의 증거로 발표에 활용 가능.
