# 11 — 메모리 3계층·자동진화 설계도 (고객/에이전트/직원 + Hermes式 증류)

> **상태**: CaseOps 분기 설계안 [분기/미확정]. [[01-메모리-거버넌스]]의 8계층 중 **사용자가 지목한 핵심 3계층(Customer/Agent/Staff)**의 실장 스펙 + 증류 자동화 + RAG/그래프 연동을 담는다. 8계층 거버넌스(권한·보존·MemoryMutationLog)는 01이 SSOT — 이 문서는 그 위의 실행 설계다.
> **핵심 주장**: 에이전트 폴더의 파일(SOUL·tools·memory)은 **사용자가 수정하지 않는다. 시스템이 축적으로 진화시킨다.** 세션·프롬프트를 전부 기록하는 것이 아니라 **의미 있는 것만 카드로 증류**하고(Hermes式), 축적될수록 똑똑해지는 것은 카드 수가 아니라 **검증된 카드의 재사용률**이다.

## 1. 3계층 메모리 — 무엇을 기억하는가

| 계층 | 주체 | 기억하는 것 (증류된 사실만) | 절대 기억하지 않는 것 |
|---|---|---|---|
| **Customer Memory** | 고객(케이스 단위 아님 — 고객 단위 종단) | 조치 이력·승인 결과·위험 패턴 변화·선호 채널·상환 행태 요약 (`subject_token` 기준) | 원본 PII·계좌·연락처·상담 전문 — 포인터(`evidence_pointer`)만 |
| **Agent Memory** | AI 에이전트·스킬 | 작업유형별 성능(반려율·티어 적합성)·불안정 패턴·승인된 정책 변경과 효과 | 프롬프트/응답 원문, 단발 관측(3회 미만) |
| **Staff Memory** | 담당 직원 (업무 한정) | 업무 패턴(담당 도메인·처리량·자주 쓰는 스킬)·반려 사유 스타일·업무 인수인계 컨텍스트 | 인사평가성 정보, 그리고 **고객 위험판단 입력으로의 유출(교차 오염) — 구조적 차단** |

## 2. 메모리 카드 — 공통 스키마 (DB SSOT)

```
MemoryCard {
  id, layer(customer|agent|staff), subject_key,   // 예: subject_token / agent_id / staff_id
  fact,                // 증류된 한 문장 (원문 복사 금지)
  provenance[],        // 근거 포인터: runId·evidenceId·auditSeq — Ledger Curator가 재검증
  observed_count,      // 관측 횟수 — 3회 미만은 후보(candidate) 상태
  confidence(candidate|confirmed|deprecated),
  pii_grade,           // 01 문서의 등급 태깅 준용, customer 계층은 Zero-PII 강제
  created_at, updated_at, ttl,                    // 보존·파기는 01 거버넌스 정책 따름
  embedding_ref, graph_edges[]                    // 파생 인덱스 포인터 (§4)
}
```

저장 3형태의 역할 분담 — **DB가 SSOT, 나머지는 파생(재생성 가능)**:

| 형태 | 역할 | 데모 구현 | 운영 승격 |
|---|---|---|---|
| DB 테이블 | 카드 원본·정합성·MemoryMutationLog | `localStorage`/JSON [E4 하네스 준용] | PostgreSQL append-mutation 테이블 |
| md 카드 | 사람 가독 뷰 — 콘솔 에이전트 폴더의 `memory.md`가 이것 | 파일 렌더 | DB→md 자동 생성(읽기 전용 뷰) |
| 벡터 인덱스 | 유사 사례 검색(RAG) | 키워드 매칭 폴백 [설계] | 로컬 임베딩 + sqlite-vec/Chroma — [[08_본선/03_제품/docs/07_architecture|07 §6]] 벡터스토어 후보와 동일, 외부 임베딩 API 미사용 |
| 그래프 | 관계 추적 `Case—Agent—Evidence—Approval—Staff` | 기존 엔티티 참조로 1-hop 재현 [설계] | 그래프 뷰 또는 edge 테이블 |

## 3. Memory Distiller — 증류 프롬프트 (이 원문을 그대로 사용)

증류는 별도 상주 에이전트가 아니라 **각 에이전트 하트비트 종료 직전에 실행되는 공통 단계**다(paperclip promptTemplate 말미에 삽입). 승격 심사는 Ledger Curator 관할([[10-Ledger-Curator-에이전트-설계도]]).

```markdown
# Memory Distiller — 세션 증류 단계

방금 끝난 작업에서 기억할 가치가 있는 것만 골라라. 기본값은 "기억하지 않는다"이다.

## 승격 기준 (하나라도 충족해야 후보 카드 생성)
1. 반복 관측: 같은 패턴을 이전 카드와 합쳐 3회 이상 관측 (observed_count 증가로 처리)
2. 사람 결정: 승인·반려·정책 변경 등 사람이 개입한 결과와 그 사유
3. 예외 사건: 폴백·사고·가드 위반 등 다음에 먼저 검색해야 할 일
4. 반증: 기존 confirmed 카드와 모순되는 관측 (해당 카드 deprecated 후보로 표시)

## 카드 작성 규칙
- fact는 한 문장, 원문 복사 금지, 주어는 subject_key로.
- provenance 없는 카드는 쓰지 마라 — runId·evidenceId를 반드시 붙인다.
- layer 판정: 고객에 관한 사실→customer(Zero-PII 검사 통과 필수),
  에이전트·스킬 성능→agent, 직원 업무 패턴→staff.
  staff 카드는 고객 위험판단 문맥에서 절대 인용되지 않도록 cross_ban=true 고정.
- 세션 전체 요약·감상·계획은 카드가 아니다. 버려라.

## 출력
JSON 배열(MemoryCard 후보). 없으면 빈 배열 — 빈 배열이 정상 케이스다.
```

## 4. RAG/그래프 연동 — 검색이 컨텍스트를 만든다

에이전트 하트비트 시작 시 시스템이 컨텍스트를 조립해 promptTemplate 변수로 주입한다. 에이전트가 직접 저장소를 뒤지지 않는다(권한 경계 유지).

```
wakeup(case) →
  1. scope 필터: 호출 에이전트의 layer 접근권한 + roleKey (01 거버넌스 ACL)
  2. 그래프 1-hop: 해당 Case—Customer—담당 Staff—참여 Agent 연결 카드
  3. 벡터 top-k(k=5): 현재 케이스 요약과 유사한 confirmed 카드 (deprecated 제외)
  4. 조립: {{memory.cards}} 변수로 프롬프트 주입 — 카드 fact+provenance만, 원문 없음
```

- 검색 결과에 staff 카드가 섞여도 **customer 판단 프롬프트에는 주입 차단**(cross_ban 필터) — 격리를 검색 계층에서 한 번 더 집행.
- 데모 구현: 그래프 1-hop은 기존 엔티티 참조(`case.agents`·`evidenceIds`)로 재현 가능, 벡터는 키워드 폴백 [설계].

## 5. 자동진화 루프 — "축적될수록 똑똑해진다"의 실체

```
카드 축적 → 주기 consolidation → SOUL 개정 제안 → 사람 승인 → 버전업
```

| 단계 | 주기 | 내용 | 권한 |
|---|---|---|---|
| 증류 | 매 하트비트 | §3 Distiller 단계 — 후보 카드 생성 | 자율 |
| 승격 심사 | 일 1회 | Ledger Curator가 provenance 재검증·layer 오배치·교차 오염 검사 → confirmed/기각 | 자율(심사)·기각 사유 기록 |
| Consolidation | 주 1회 | 중복 카드 병합(observed_count 합산), 반증 누적 카드 deprecated, TTL 만료 파기 제안 | 병합=자율 / 파기=**사람 승인** |
| **SOUL 개정 제안** | confirmed 카드 임계(예: 동일 주제 5장) 도달 시 | "이 지침을 SOUL에 반영하자"는 **diff 형태 제안** 생성 — 예: Cost Sentinel 카드 5장 "여신 요약은 로컬 충분" → SOUL 라우팅 지침에 1줄 추가 제안 | **사람 승인 후에만 SOUL 변경**, 버전 태그·이전 버전 보존 |

- **사용자 비수정·열람 원칙**: 담당자(AI 비전문가)는 콘솔에서 SOUL·memory.md를 **읽고**, SOUL 개정 제안을 **승인/반려**할 뿐, 파일을 직접 편집하지 않는다. 편집 UI 자체를 제공하지 않는다.
- 진화 지표(콘솔 노출): confirmed 카드 재사용률(검색되어 실제 컨텍스트에 주입된 비율) · SOUL 개정 반영 수 · deprecated 비율. 카드 총량은 지표가 아니다 — 쌓이기만 하는 메모리는 창고다.

## 6. 승인 게이트 요약

- **자율**: 증류·후보 생성·검색 주입·병합.
- **제안→승인**: SOUL 개정, 카드 파기, cross_ban 예외(원칙적으로 불허).
- **사람만**: customer 카드의 외부 반출(원칙 금지), 보존정책 변경.

## 8. 경계 정의 — 지식베이스 ≠ 스킬 ≠ 메모리 (혼용 금지 근거)

세 개념은 저장소·갱신 주체·검증 방법이 전부 달라서 섞으면 거버넌스가 무너진다. 판별 한 줄: **실행되면 스킬, 인용되면 지식, 관측이면 메모리.**

| 축 | 스킬(Skill) | 지식베이스(KB) | 메모리(Memory) |
|---|---|---|---|
| 본질 | 절차 지식 — "어떻게 한다" (동사) | 선언 지식 — "무엇이 사실이다" (명사) | 경험 증류 — "해보니 이랬다" (관측) |
| 우리 계약에서 | `Skill` 엔티티 — AgentRun이 **실행** | `Evidence`의 원천 — 판단이 **인용**(RAG 검색) | `MemoryCard` — 컨텍스트에 **주입** |
| 저장·형태 | 코드/정의 파일(tools.json·스킬 스크립트), 버전드 | 문서 스토어(규정·상품·정책 원문+출처), 04_tech/rag-rule-engine 소관 | 카드(DB SSOT + md 뷰), §2 |
| 갱신 주체·주기 | 개발/승인 배포 단위 — 임의 변경 금지 | 원천 기관 갱신 시 동기화(출처·발효일 관리) | 매 하트비트 증류, Curator 심사 |
| 틀렸을 때 | 버그 — 롤백·핫픽스(119) | 원천 오류/구버전 — 재동기화·출처 교체 | deprecated 강등 (§5) — 지우지 않고 반증 기록 |
| 검증 | eval/e2e 테스트 통과 | 출처 대조(canon 인용 규율과 동일) | provenance(runId) 재확인 |

혼용 시 실제 사고 유형: ① KB 문서를 스킬처럼 "실행"(규정 원문이 행동을 트리거) → 승인 게이트 우회 ② 메모리 카드를 KB처럼 확정 사실로 인용 → 관측 3회를 규정처럼 발표 ③ 스킬 안에 지식 하드코딩 → 규정 개정 시 스킬 전체 재배포. 세 사고 모두 **저장소를 물리적으로 분리**하면 구조적으로 차단된다 — 이것이 분리의 근거다.

**paperclip 실증** [E2, 2026-07-04 탐사]: paperclip도 정확히 이 분리를 코드로 강제한다 — 스킬은 `company_skills` DB 테이블(버전 이력·trustLevel·"설치≠배정" 2단계), 지식/정체성은 에이전트별 **파일 번들**(AGENTS.md·SOUL.md·HEARTBEAT.md·TOOLS.md, 매 실행 `--append-system-prompt-file` 주입). 스킬 문서 내부조차 SKILL.md(상시 로드) vs references/(작업 매칭 시만 로드)로 계층화해 토큰을 아낀다.

KB 구축 자체는 시스템 골격 완성 후 착수(사용자 결정, 2026-07-04). 그 전까지 KB 역할은 기존 근거팩·canon 문서가 수동으로 대신한다.

## 9. paperclip 실증 매핑 — 베낄 패턴 8 (탐사 2026-07-04, `_vendor/paperclip-master` 스키마·spec·캡처 기준)

| # | paperclip 패턴 (근거) | 우리 적용처 |
|---|---|---|
| 1 | 실행설정(adapterConfig) vs 운영정책(runtimeConfig) jsonb 분리 (`agents.ts`) | agent.yaml에서 adapter 절/heartbeat 절 분리 유지 근거 |
| 2 | 세션을 태스크 단위로 스코프(`agent_task_sessions` unique키, CLI --resume 재사용) | Case 재오픈 시에만 컨텍스트 재개 — Case-scoped 단기기억 스키마 |
| 3 | 누적 카운터 1-row 테이블(`agent_runtime_state`) — 원장과 롤업 분리(CQRS식) | `auditChainRecords`(원장) vs `buildDashboardData`(롤업) 분리 유지 근거 + Cost Sentinel 집계 테이블 |
| 4 | 풀로그/발췌/이벤트 3단 분리(logRef 외부 + 발췌 상한 + append 이벤트) | Evidence 저장: 원본 PII 문서는 경계 안 스토어, DB에는 발췌·포인터만 — PII 역설 방어와 동일 구조 |
| 5 | 설정 변경 diff 스냅샷(`agent_config_revisions`: before/after+changedKeys+롤백 체인) | §5 SOUL 개정 제안의 저장 형식 그대로 채택 |
| 6 | fingerprint 중복 억제(`issue_recovery_actions` partial unique) | 119 반복 사고·에스컬레이션 알림 피로 억제 ([[09-119-라우팅관측-확장-설계도]] 임계표와 결합) |
| 7 | 웨이크업 단일 진입점+코얼레싱(`enqueueWakeup`, coalescedCount) | AgentRun 트리거 경로 단일화 — 직접 어댑터 호출 금지를 아키텍처로 강제 |
| 8 | 지침 번들 역할별 파일 분리(SOUL=정체성/HEARTBEAT=반복절차/TOOLS=도구/AGENTS=entry) + Managed/External 모드 | 우리 파일셋에 **HEARTBEAT.md 분리 채택 검토** — 현재 SOUL에 순찰 절차가 섞여 있음 [개선 제안] |

**핵심 관찰**: paperclip **코어에는 자동 증류·장기기억 파이프라인이 없다**(compact context API·contextSnapshot 필드 수준, 증류는 동봉 예시 스킬 para-memory-files의 주간 합성+Hot/Warm/Cold 감쇠뿐). 즉 §3 Distiller + §5 자동진화 루프는 paperclip 모방이 아니라 **우리의 차별 확장**이다 — 발표에서 "paperclip을 베꼈다"가 아니라 "paperclip이 비워둔 기억 계층을 채웠다"로 포지셔닝. para-memory-files의 access_count 감쇠(Hot 7일/Warm 30일/Cold)는 §5 consolidation의 TTL 세부안으로 차용 [설계].
