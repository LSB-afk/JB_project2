# harness-audit

> 2026-07-04 세션에서 사용자가 "하네스 시스템이 파이프라인이 적합한지, 사용하지 않는 파일이 맥락 혼동을 주는지, 층위가 어긋나는지, 레퍼런스와 정본을 혼용하고 있지 않은지, 커밋/푸시/PR이 꼬이고 있지 않은지, 에이전트 활용은 어떤지"를 병렬 에이전트로 점검·개선해달라고 요청한 과정을 재사용 가능한 스킬로 정착시킨 것.

## 다른 스킬과의 차이

이 하네스의 다른 12종 스킬(`canon-moc-sync`·`harness-sync` 등)은 **결정론적 node 스크립트**다. `harness-audit`는 다르다 — Claude 서브에이전트를 병렬로 여러 개 띄워 판단·grep·git 조회를 시키고 교차검증하는 **Workflow 스킬**이라, `node ...`로 단독 실행할 수 없고 **Claude Code 세션 안에서 Workflow 툴로만** 호출 가능하다. 비용(토큰·시간)이 다른 스킬보다 훨씬 크므로 **자동 시행 대상이 아니다** — 사용자가 명시적으로 점검을 요청할 때만 쓴다.

## 언제 쓰는가

- 사용자가 "하네스 점검"/"전체 감사"/"시스템 점검"/"파이프라인 적합한지 봐줘" 등을 요청할 때
- 제출·발표 마일스톤 D-1 등, 큰 변경 전에 구조적 건강도를 확인하고 싶을 때
- 정기 점검(예: 매 스프린트 말)으로 팀이 직접 요청할 때

## 실행

```
Workflow({ scriptPath: '08_본선/_system/skills/harness-audit/workflow.js' })
```

Claude Code 세션에서 위처럼 `Workflow` 툴을 호출한다(스킬 자체가 스크립트 실행형이 아니라 워크플로 호출형). 완료까지 보통 10분 내외, 서브에이전트 20개 안팎·토큰 100만+ 소모— 사용자에게 규모를 먼저 알리고 진행할 것.

## 감사 차원 (9개, `workflow.js`의 `DIMENSIONS` 배열)

| 차원 | 확인 대상 |
|---|---|
| `pipeline` | 스킬 폴더 vs AGENTS.md §4-A 트리거표 vs registry-skills.md 1:1 정합, 오케스트레이터가 참조하는 하위 스크립트 실재 여부 |
| `deadfiles` | OS/동기화 노이즈, 0바이트 파일, canon-moc-sync 도달성/죽은링크 결과, 문서 중복 |
| `layering` | 파일명 중복(진짜 충돌 vs 인덱스 관례), 예선/본선 넘버링 불일치, frontmatter 준수율 |
| `canon_vs_reference` | gitignore된 `_vendor/` 등을 정본처럼 인용하는지, canon 정정이 다른 문서에 전파됐는지 |
| `git_health` | 브랜치 ahead/behind, 미커밋 분포·staged 비율, PR 상태, 대외비 파일 노출 여부 |
| `open_questions` | Capture-by-default 준수율, 레지스트리 드리프트, 텔레메트리 이상치 등 그때그때 새로 떠올리는 질문 |
| `agent_roles` | 본선 역할(role) 페르소나 registry vs 실사용(팀배정·로그빈도) |
| `cc_agents` | Claude Code 서브에이전트/스킬 정의 경로 중복·고아 여부, 전역 규칙 파일과 실제 상태 괴리 |
| `agent_registry` | `_agent-registry.md` 등 "운영 에이전트" 토큰 추적 문서들 간 정합성·방치 여부 |

각 차원은 **find(조사) → verify(독립 재검증)** 두 단계를 거친다(`pipeline()`으로 차원별 병렬 실행, 한 차원의 verify가 다른 차원의 find를 기다리지 않음). verify 단계는 find의 주장을 곧이곧대로 believe하지 않고 처음부터 다시 실측한다 — 실제로 2026-07-04 실행에서 verify가 find의 과장/오류(예: "산술 오류로 41개라 했는데 실제 49개", "fork 대비 29커밋 밀렸다 했는데 실제로는 0")를 다수 잡아냈다.

## 결과 사용법

`workflow.js`는 `[{dimension, key, find, verify}, ...]` 배열을 반환한다. 이후 Claude가:
1. 각 차원의 verify 결과를 기준으로 **정정된** 수치만 사용해 리포트 작성(find의 원 주장이 아니라 verify가 확인한 값).
2. 발견 사항을 심각도(제출물 영향 > 하네스 정합성 > 하우스키핑)로 정리.
3. 개선 조치를 3단계로 나눠 사용자에게 제시:
   - **즉시 안전 적용**(문서/코드 정정, 파괴적이지 않음)
   - **사실 판단 필요**(수치 충돌, SSOT 갱신 — 팀/사람 승인)
   - **git 쓰기 작업**(push 등 — 별도 승인, 명시경로 커밋만)
4. 승인된 조치만 적용하고, git 커밋은 **항상 `git commit -- <명시경로>`**로 한다([[병렬세션-커밋-트랩]] 참고 — 이 저장소는 병렬 세션이 흔해 인덱스를 자주 건드린다. `git rm --cached` 등으로 스테이징한 뒤 곧바로 커밋이 안 되면 다른 세션이 그 사이 인덱스를 되돌렸을 가능성이 크다 — 무한 재시도하지 말고 사용자에게 상황을 보고).

## 주의사항

- 읽기 위주 감사다. git은 조회 명령(`status`/`log`/`rev-list`/`gh pr view` 등)만 쓰고, 쓰기(`add`/`commit`/`push`)는 절대 이 워크플로 안에서 하지 않는다 — 감사 결과를 본 뒤 별도 승인을 받아 진행.
- `DIMENSIONS`의 각 프롬프트는 오늘 날짜의 구체적 숫자를 하드코딩하지 않는다 — 매 실행마다 실제 상태를 새로 조사하도록 절차만 지시한다. 재실행할 때 프롬프트를 굳이 갱신할 필요 없음.
- 차원을 늘리거나 줄이려면 `workflow.js`의 `DIMENSIONS` 배열만 수정 — `pipeline()`/스키마는 그대로 재사용된다.
- 결과가 크므로(서브에이전트 9~18개) 최종 리포트는 Claude가 직접 종합해서 압축 요약해야 한다 — 원문 JSON을 그대로 사용자에게 보여주지 말 것.
