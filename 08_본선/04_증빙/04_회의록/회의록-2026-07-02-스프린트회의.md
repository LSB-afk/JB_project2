## 공유·결정 (구조화 요약)
1. **디자인·UX 1차 완료(김민주)**: JB 웹 분석 → 디자인 시스템(컬러 토큰·폰트·페이싱·레디우스·마진) + 컴포넌트 + 상태값 컬러·콘솔 레이아웃. **UX 라이팅** 전면 교체("강소 금융그룹"·고객 업무 지원 톤). 완성본은 개발(이승보)에게 전달.
2. **개발 진척(이승보)**: 어제 UI 흐름 다듬기 + **비은행(JB우리캐피탈) 대시보드 병행** 분석·설계. 아직 Claude API는 호출하지 않은 상태였고, 오늘 API 연결 후 작업을 돌리면 **Fable 5**로 화면 퀄리티가 더 올라갈 것으로 기대.
3. **키보드-퍼스트 UX(확정 방향)**: 업무보드 = **원 슬라이드 페이지**. 스페이스/방향키로 순차 검토·수락 → 마우스 최소화(업무량 많은 담당자 손목 부담↓ = **차별화 포인트**, 발상은 "코덱스가 하는 말을 안 읽는다"는 주용의 습관에서 착안). **오승인 방지 = 스페이스 2회**(1회=내용 열람 후 키보드 스크롤, 2회=승인). "원클릭"보다 신선한 슬로건/키워드로 형상화하자는 아이디어.
4. **역할별 대시보드 + 로그인(개발설계, 이승보)**: RM·기업여신 담당자·전세보호 담당자 등 역할별 대시보드. 배포 시 **로그인 → 역할군 아이디 → 본인 역할 대시보드만 표시**. 처음엔 역할별로 메인화면 자체를 분리 설계했으나 "확 와닿지 않을 것 같다"는 판단으로 **좌측 사이드바에서 담당자별 대시보드 전환** 방식으로 변경. 계열사(전북/광주/캐피탈) 각각 내부에서 동일 구조.
5. **4대 핵심 업무 도메인 + 로스터(에이전트 목록)**: 계열사 공통으로 다루는 도메인은 **여신, 여신 사후관리, 보이스피싱, 전세사기 관련** 네 가지. "로스터"(에이전트 명단)에는 14개 이름이 붙어 있으나, 실제 몇 개 에이전트가 필요할지는 이 시점엔 유동적이라고 언급.
6. **에이전트 구조(제품 방향, 김주용)**: 담당자 R&R 중 에이전트가 할 수 있는 업무를 세팅. 구성 = 담당자 **노하우 스킬(암묵지)** + 기업 업무 스킬 + **MCP**(팀 일정·금융 MCP·정책 문서) + **은행 내부 DB** + **국가 오픈 API**(국토부 실거래가·전세·정책 규정) + **LLM**. 위험성 분석 반복. **최종 승인·책임 = 사람(담당자)**. **고객 개인정보는 에이전트가 직접 다루지 않음**(근거: SNS 계정 해킹 후 챗봇으로 정보가 유출된 사례를 우려 사례로 언급). "아는 것 많은 RM의 비서."
7. **케이스/고객 메모리 제안**: 오픈AI/헬멧류 서비스의 개인 메모리처럼, 케이스·고객별 메모리(통화내역·쪽지·시계열 일정·상담 데이터) 축적 → 담당자 확인·판단 지원. 본인도 "은행에서 실제 이렇게 안 할 수도 있는 망상 아니냐"고 자문했으나 이승보는 "괜찮을 것 같다"고 동의.
8. **신뢰·판단 해법**: 금융권 신뢰 장벽(티빙·쿠팡 정보유출 사례 언급) → "데이터 제공 시 기존 담당자 노하우·판단을 AI 학습 + 실시간 데이터 + 등록 스킬 결합 → 판단 신뢰도↑" 화법으로 대응. 위험성 분석은 반복적으로 수행.
9. **AX 피칭 내러티브(초안, 김주용)**: 심사위원 설득용 화법으로 "데이터 많고 담당자 업무 많은 거 알겠으니 이걸 에이전트화 해보자, AX 시도해보자 → 지방에서 내로라하는 은행이면 혁신해야 한다"는 도입부를 구상. 차별성은 "한 담당자의 업무를 면밀히 캐치해 자동화 요소를 UX까지 고려해 하네스로 만들어주는 것"이며, 단순 챗봇이 아니라 노하우·업무스킬·MCP·플러그인·은행 데이터 접근을 결합한 것이라는 점을 강조. 팀원 전원 공감.
10. **발표 전략**: 프로토타입 시연 = 팀원 개인정보로 연기(전주/광주 연고). 히어로 = **전주 카페 창업 대출 심사**(담당자→심사위원→고객 시선 연결). 발표자 = **김민주**(언니가 광주 거주, 지역 공감대 — 경상도 출신 주용은 공감대 부족을 스스로 인정). 전세 위험관리 케이스는 "서울 신축빌라 전세 5억"(고액)과 "전주 중앙로 카페"(자금 부족) 사례를 대비시켜 설명.
11. **보이스피싱 시연 후보 + 자기비판**: 노인 대상 송금 시 설문조사로 위험점수를 매겨 송금을 차단하는 시나리오를 제시했으나, 곧이어 "고객도 번거로워 짜증 내고 담당자도 번거로워서 실효성이 떨어질 수 있다"는 자기비판이 나옴 — 시연 후보로만 남기고 UX 재검토가 필요한 상태.
12. **평가·범위 인식**: 평가 각 20점(혁신성·완결성·MVP·사업화, 세부 조항은 위 발췌 "보강" 참조). "**MVP 실동작 + 산출물 신뢰 → 3등 이상 가능**"(라이브코딩 paperclip 기반 설계 = 경쟁 우위 자평). 도메인 지식 부족 인정, 데이터·인터뷰·멘토링 부재를 근거로 "이 대회는 아이디어·실행력을 보는 대회"로 재해석 → 백엔드 전체·시장분석 과열 경계, **MVP+발표 집중**. 자유주제 안에 지정주제 요소도 복합 포함해 설계했다는 점, 금융그룹 단위 자유주제 참가팀이 0팀이라는 점에서 그룹 확장성을 강조하기로 함.
13. **API 사용 비용 우려(구두 추정, 미검증)**: Claude API를 계속 쓸 경우 비용이 얼마나 될지 이야기하다 "450달러 정도"(약 70만 원 환산 추정), "10팀만 해도 700만 원"이라며 외화 유출 우려를 나눔 — 실제 청구액이 확인된 수치는 아니다. 같은 흐름에서 "AI 사용량(텔레메트리)이 평가 요소일 수 있다"는 문제의식이 나와 **사용량 전면 가시화**로 합의.
