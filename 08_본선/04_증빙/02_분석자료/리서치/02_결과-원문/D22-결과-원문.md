# 금융권 AI 모델 도입 전략과 로컬 우선 인프라 경제성

## 핵심 발견 요약

**A. 외부 상용 LLM 거부감과 해소 스펙트럼**\
한국 금융당국은 2024년 하반기부터 생성형 AI 활용을 제도적으로 열어 주었지만, 그 방식은 “무제한 허용”이 아니라 **상용 AI 허용 + 내부망 오픈소스 AI 지원**의 **이원화 체계**입니다. 금융위원회는 2024년 12월 발표에서 금융회사의 내부망에 직접 설치하는 오픈소스 AI 활용을 별도로 지원한다고 명시했고, 그 배경으로 망분리, 자체 보안규정, 검증된 모델·데이터 부족, 거버넌스 불명확성을 직접 적었습니다. 2025년 2월 전자금융감독규정 개정 역시 규칙 중심에서 원칙 중심으로 전환하면서, 보안 위험을 금융사가 스스로 진단하고 이사회 보고·제3자 리스크 관리까지 강화하도록 했습니다. 즉 은행이 외부 API를 꺼리는 것은 단순한 “보수성”이 아니라, **고객정보 반출, 제3자 IT 리스크, 감사 가능성, 모델/데이터 검증 책임**이 여전히 은행 자신에게 남기 때문입니다. 반대로 해소 스펙트럼은 존재합니다. 완전 온프레, 전용/지역 제한 클라우드, 게이트웨이·프록시, 비전송 설계는 각각 보안·운영·비용이 다르며, 한국 제도는 이 스펙트럼을 실무적으로 허용하는 방향으로 움직이고 있습니다. [\[1\]](https://www.fsc.go.kr/no010101/83594?curPage=&srchBeginDt=&srchCtgry=&srchEndDt=&srchKey=&srchText=)

**B. 모델 선택과 주권**\
은행의 합리적 모델 선택은 “벤치마크 1등”이 아니라 **상업 라이선스의 명확성, 전체 로컬 호스팅 가능성, 공급망/주권 리스크, 한국어·국내 지원 가능성, 추후 감사 가능한 운영 구조**를 함께 보는 문제입니다. 미국계 오픈웨이트의 대표인 Llama 3.1은 호스팅과 파생 개발은 넓게 허용하지만 Meta 커뮤니티 라이선스의 추가 조항을 따라야 합니다. 중국계 Qwen3는 Apache 2.0으로 매우 유연하고, DeepSeek-R1 계열은 MIT로 상업·증류에까지 열려 있어 기술·비용 면에서는 강합니다. 그러나 한국에서는 2025년 개인정보보호위원회가 **DeepSeek 앱 서비스를 잠정 중단**시켰고, 국가정보원도 **업무 활용시 보안 유의**를 공문으로 배포했습니다. 이 사실은 “중국계 모델 자체가 곧 위험하다”는 증거는 아니지만, 적어도 **중국계 호스티드 서비스는 금융권 신뢰·감사·대외 설명 부담이 매우 크다**는 신호입니다. 국산 계열에서는 NAVER HyperCLOVA X SEED와 Upstage Solar Open이 생산 후보이며, EXAONE 3.5는 2024년 공개본 기준 연구용 라이선스라 은행 실서비스에는 바로 쓰기 어렵습니다. 결과적으로 한국 은행의 철학·신뢰·국산화 관점에서 가장 자연스러운 선택지는 **국산 우선, 그다음은 한국 내 로컬 운영 가능한 개방형 외산 오픈웨이트, 외부 강력 API는 예외적 승격 라우팅**입니다. [\[2\]](https://github.com/meta-llama/llama-models/blob/main/models/llama3_1/LICENSE)

**C. 컴퓨팅·전력·TCO**\
금융권 로컬 LLM 경제성의 핵심 변수는 의외로 **전기료보다 GPU 활용률과 모델 크기**입니다. 한국전력의 산업용 전력요금은 kWh당 대략 50원대 후반에서 100원대 초반 수준이며 기본요금도 존재하지만, H100/H200급 추론 서버의 월 전력비는 총 TCO에서 보통 **작은 항목**입니다. 더 큰 비용은 GPU 자본비용, 감가상각, 유지보수, 상면, 운영 인력입니다. NVIDIA 공개 벤치마크 기준으로 Llama 3.1 8B는 H100 1장에서도 200 input / 200 output 워크로드에서 약 **214.56 tokens/s**, Llama 3.1 70B는 H100 4장 구성에서 같은 워크로드·동시성 5에서 약 **387.55 tokens/s**, Llama 3.3 70B는 H200 2장 구성에서 500 input / 2000 output 워크로드·동시성 5에서 약 **261.83 tokens/s**를 냅니다. 이 수치를 월간 처리량으로 바꾸면, 50% 활용률 기준 각각 **약 2.78억 / 5.02억 / 3.39억 토큰/월** 규모입니다. 따라서 일정 수준 이상의 볼륨이 있는 내부 지식검색·문서요약·상담보조·사내 코파일럿은 로컬 주력 모델이 경제적으로 설 자리가 분명합니다. 다만 **값싼 API**와 비교하면 break-even이 매우 늦고, **비싼 frontier API**와 비교할 때만 비교적 빨라집니다. 즉 “모든 것을 로컬로”가 아니라, **대량·반복·민감한 일은 로컬**, **희소하고 고난도인 일만 외부 premium**이 경제적으로 맞습니다. [\[3\]](https://home.kepco.co.kr/kepco/front/html/CY/E/E/CYEEHP00103.html)

**D. 하이브리드와 로컬 충분성 입증**\
“좋은 하네스 + 작은 로컬 모델”이 실용 성능을 낼 수 있다는 근거는 점점 강해지고 있습니다. OpenAI는 에이전트 구축을 모델·도구·상태·오케스트레이션의 조합으로 설명하고 있고, LangChain의 Deep Agents는 **frontier·open-weight·local 모델을 같은 하네스에 꽂을 수 있다**고 명시합니다. Microsoft는 Phi-4-mini 같은 3.8B급 소형 공개모델에도 **function calling**을 넣어 작은 모델이 외부 도구와 구조화된 인터페이스를 통해 한계를 보완하도록 설계했습니다. 또 Hugging Face는 코드 에이전트 중심 설계와 안전한 코드 실행을 전면에 내세우고 있습니다. 더 중요한 점은 벤치마크입니다. GAIA는 사람은 92%인데 GPT-4+plugins baseline은 15%에 불과하다고 보고했고, SWE-agent는 동일한 하네스 안에서 여러 모델을 꽂아 평가하며 mini-SWE-agent가 SWE-bench Verified 65%를 달성했다고 공개했습니다. 이건 “모델만 바꾸면 끝”이 아니라 **도구, 분해, 검증 루프, 사람 승인, 실행 하네스**가 실제 성능을 크게 좌우한다는 직접 증거입니다. 금융권 데모에서는 이 메시지가 특히 중요합니다. **민감정보 비반출을 지키는 로컬 모델이 1차 처리**, **외부 모델은 공개정보·비식별·고난도 작업에만 승격**되어도, 사용자 체감 성능은 충분히 높일 수 있습니다. [\[4\]](https://github.com/langchain-ai/deepagents)

**E. 추가로 중요한 요소**\
금융권 AI는 단순 모델 선택보다 운영 거버넌스가 더 중요합니다. 2025~2026 한국 금융당국안은 이사회·경영진 책임, AI 위험관리체계, 제3자 IT 리스크, 외부 모델 및 데이터 검증, 설명가능성, 보안성 점검을 모두 명시하고 있습니다. 따라서 은행이 준비해야 하는 것은 “모델 1개”가 아니라 **모델 등록부, 버전·EOL 관리, 평가 세트, 데이터 분류 정책, 외부 호출 승인 정책, 재현 가능한 로그와 감사 추적**입니다. 또한 전력·탄소는 ESG와 직결되지만, 실제 비용에서는 전기보다 GPU 유휴율이 더 큰 변수이므로, 그린 AI의 실무적 핵심도 **과대형 모델 상시 운영을 피하고 적정 크기 모델을 워크로드별로 라우팅하는 것**입니다. [\[5\]](https://www.fsc.go.kr/no010101/85908?curPage=&srchBeginDt=&srchCtgry=&srchEndDt=&srchKey=&srchText=)

## 외부 LLM 거부감과 해소 스펙트럼

은행이 외부 상용 LLM을 싫어하는 이유는 크게 다섯 가지로 수렴합니다. 첫째, **고객정보와 내부문서가 제3자 서비스로 이동한다는 사실 자체**입니다. OpenAI는 API 데이터를 기본적으로 학습에 쓰지 않지만, 기본 남용 모니터링 로그를 **최대 30일 보관**할 수 있다고 밝히고 있습니다. 이 문장 하나만으로도 “학습에는 안 써도 은행 기준으론 여전히 외부 전송”이라는 결론이 나옵니다. 둘째, **제3자 IT 리스크와 감사 책임**입니다. 금융위원회는 망분리 로드맵과 2025년 감독규정 개정에서 모두 제3자 리스크 관리와 자율보안 책임을 강조했습니다. 셋째, **거버넌스와 설명 책임**입니다. 2025년 금융분야 통합 AI 가이드라인(안)은 외부모델 및 데이터 검증, 보안성 검증·운영 관리, 감독당국 보고절차까지 넣고 있습니다. 넷째, **데이터 주권과 지역 처리** 문제입니다. Azure, Google은 지역 처리와 데이터 존을 제공하지만, 전역형 배포는 다른 지역에서 처리될 수 있습니다. 다섯째, **벤더 종속**입니다. API 가격, 모델 EOL, 지역 가용성, 정책 변경이 은행 내부 통제 밖에서 바뀔 수 있습니다. [\[6\]](https://developers.openai.com/api/docs/guides/your-data)

이 때문에 금융권의 현실적 해법은 “예/아니오”가 아니라 **해소 스펙트럼**입니다. 가장 왼쪽은 완전 온프레 자가호스팅입니다. 데이터 비반출과 감사가능성이 가장 높고, 한국 금융당국도 이를 제도적으로 지원하고 있습니다. 중간에는 전용·지역 제한 클라우드가 있습니다. Azure의 `Models sold by Azure`는 OpenAI 같은 원 공급자 서비스와 상호작용하지 않으며, 데이터 존/리전 배포를 명시합니다. Google의 Gemini Enterprise Agent Platform도 요청이 들어간 특정 지역·멀티리전 내 ML 처리, 요청/응답 로깅 기본 비활성화, 별도 설정을 통한 zero retention 설계를 제공합니다. 오른쪽에는 **게이트웨이·프록시**가 있습니다. 여기서는 PII 마스킹, 정책 엔진, 호출 승인, 로그 적재를 통해 외부 모델을 제한적으로 쓰되, 민감 원문은 안 보내거나 최소화합니다. 완전 오른쪽은 **비전송 설계**입니다. 질의 분류, 내부 검색, 문서 요약, 템플릿 초안, 규정 탐색 등 대부분을 로컬 모델이 처리하고, 외부 강력 모델에는 공개 문서·비식별 요약·합성 테스트셋만 보내는 방식입니다. [\[7\]](https://www.fsc.go.kr/no010101/83594?curPage=&srchBeginDt=&srchCtgry=&srchEndDt=&srchKey=&srchText=)

## 모델 선택지와 주권

모델 선택은 세 층으로 보는 것이 가장 실무적입니다. **생산 라이선스**, **주권/신뢰**, **로컬 적합성**입니다. 라이선스부터 보면, Qwen3는 Apache 2.0, DeepSeek-R1은 MIT여서 상업적 활용과 파생 개발이 가장 단순합니다. Llama 3.1은 널리 쓰이지만 Meta 고유 커뮤니티 라이선스를 따르며, 초대형 플랫폼에 대한 별도 조항과 브랜드·표시 의무가 있습니다. NAVER HyperCLOVA X SEED는 상업 활용이 가능하되 1천만 MAU 초과 또는 NAVER와 직접 경쟁하는 경우 별도 라이선스를 요구합니다. Upstage Solar Open 100B는 Apache 2.0 기반 변형에 가깝지만 Solar 상표 표시 의무가 추가된 자체 라이선스입니다. EXAONE 3.5는 공개본 기준 연구용에 가깝고 상업 활용은 별도 합의가 필요합니다. 이 차이 하나만으로도 “은행 프로덕션에 넣을 수 있는 후보”가 크게 갈립니다. [\[13\]](https://github.com/meta-llama/llama-models/blob/main/models/llama3_1/LICENSE)

주권과 신뢰 측면에서는, **호스티드 서비스와 셀프호스팅 오픈웨이트를 분리해서 보아야** 합니다. 예를 들어 한국 개인정보위와 국정원이 문제 삼은 것은 DeepSeek **앱/서비스 활용**이었습니다. 이는 서비스 운영·개인정보 처리·업무 활용 가이드 문제이지, 동일한 계열 오픈웨이트를 한국 은행 내부망에 내려받아 독립적으로 돌리는 행위와 동일하지 않습니다. 반면 미국계 API 역시 “무학습”만으로 문제가 해결되지는 않습니다. OpenAI는 기본적으로 API 입력·출력을 최대 30일 보관할 수 있고, Azure·Google도 지역 설정과 배포 유형에 따라 처리 위치가 달라집니다. 그러므로 주권·신뢰 기준은 “어느 나라 모델이냐”보다, **가중치와 서빙 경로를 은행이 통제하느냐, 계약·감사·삭제 보장이 명확하냐, 정책 변화가 은행 내부통제를 우회하지 않느냐**가 더 핵심입니다. 다만 외부 커뮤니케이션과 대외 신뢰까지 고려하면, 한국 은행이 **국산 또는 국내 사업자와 계약 가능한 모델을 우선**하는 건 충분히 합리적입니다. [\[14\]](https://www.pipc.go.kr/np/cop/bbs/selectBoardArticle.do?bbsId=BS074&mCode=C020010000&nttId=11007)

실행 관점의 추천은 다음과 같습니다. **기본 내재화 레이어**에는 8B~32B급 개방형 모델을 둡니다. 이 레이어는 내부 문서 탐색, 규정 QA, 콜센터 지식보조, 문안 초안, 분류·추출·요약 같은 대량 반복 업무를 맡습니다. **한국어/국산 우선 레이어**에는 HyperCLOVA X SEED, Solar Open 같은 국산 계열을 평가합니다. **외산 개방형 범용 레이어**에는 Llama 3.1, Qwen3, DeepSeek distill을 둡니다. **승격 레이어**로만 외부 premium API를 둡니다. 이때 승격 기준은 단순히 “어려운 질문”이 아니라 **공개정보만으로 풀 수 있는지, 민감정보를 제거했는지, 인간 승인 필요성이 있는지**입니다. 이것이 model-agnostic 하네스와 가장 잘 맞는 구조입니다. [\[15\]](https://www.fsc.go.kr/no010101/83594?curPage=&srchBeginDt=&srchCtgry=&srchEndDt=&srchKey=&srchText=)

### 상용 API와 오픈웨이트 자체호스팅의 의사결정 프레임

상용 API와 자체호스팅은 “둘 중 하나”가 아니라 **업무 단위 라우팅** 문제입니다. OpenAI의 실무 가이드도 큰 모델을 무조건 쓰기보다 **정확도 목표를 맞춘 뒤 더 작은 모델로 비용·지연을 최적화**하라고 권고하고, Azure도 여러 작은 모델을 조합해 적절한 전문가 모델에 질의를 보내면 비용과 정확도를 함께 개선할 수 있다고 설명합니다. 이 프레임을 금융권에 옮기면, **민감도 높은 대량 업무는 로컬 소형·중형 모델**, **희소하지만 실패 비용이 큰 고난도 업무는 premium 외부 모델**로 보내는 것이 정석입니다. [\[22\]](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/)

## 컴퓨팅·전력·TCO

### TCO 산식

금융권 로컬 LLM의 월 TCO는 다음처럼 단순화할 수 있습니다.

`월 TCO = 월 감가상각 + 유지보수/보증 + 상면/네트워크 + 운영인건비 + 전력비`

전력비는 다시,

`월 전력비 = (IT부하 kW × PUE × 24 × 30 × 전력량요금) + (계약전력 kW × 기본요금)`\
`실청구 = (전기요금계 × 1.10 VAT) + (전기요금계 × 0.027 전력산업기반기금)` 정도로 볼 수 있습니다. 한국전력은 산업용(갑)Ⅱ 고압A 선택Ⅱ 기준 기본요금 7,140원/kW, 전력량요금은 시간대별로 53.70원/kWh에서 107.70원/kWh 수준을 제시하고 있습니다. [\[23\]](https://home.kepco.co.kr/kepco/front/html/CY/E/E/CYEEHP00103.html)

토큰 기준 자체호스팅 비용은

`자체호스팅 원가/1M토큰 = 월 TCO / 월 처리토큰수 × 1,000,000`

이고, 월 처리토큰수는

`월 처리토큰수 = 실효 throughput(tokens/s) × 월초수 × 평균활용률`

입니다. 여기서 가장 큰 민감도는 **throughput과 활용률**입니다. 전력 단가는 분명 중요하지만, 실제 비용을 뒤집는 것은 유휴 GPU입니다. 같은 하드웨어라도 utilisation 20%와 60%는 단가를 세 배 차이 낼 수 있습니다. [\[24\]](https://docs.nvidia.com/nim/benchmarking/llm/latest/performance.html)

### 하드웨어 적정성

NVIDIA가 2026년 4월 갱신한 NIM 벤치마크를 보면, Llama 3.1 8B는 **H100 1장**에서 인터랙티브 워크로드(200 in / 200 out) 기준 동시성 1에서 214.56 tokens/s입니다. 같은 페이지에서 Llama 3.1 70B는 **H100 4장**에서 같은 워크로드·동시성 5에서 387.55 tokens/s이고, Llama 3.3 70B는 **H200 2장**에서 500 in / 2000 out·동시성 5에서 261.83 tokens/s입니다. 이 수치만 놓고 보면, 은행 규모의 팀 단위 코파일럿·지식검색 용도라면 8B~32B는 단일 GPU 또는 1~2 GPU 서버, 70B급은 2~4 GPU 서버가 현실적 출발점입니다. 초대형 reasoning 모델 본체는 여전히 다GPU 인프라를 요구합니다. [\[25\]](https://docs.nvidia.com/nim/benchmarking/llm/latest/performance.html)

최적화의 효과도 큽니다. vLLM은 양자화가 더 작은 메모리 풋프린트를 만들어 더 넓은 장치 범위에서 대형 모델을 돌릴 수 있다고 설명하고, FP8은 모델 메모리 요구량을 2배 줄이고 throughput을 최대 1.6배 개선할 수 있다고 적습니다. NVIDIA는 speculative decoding으로 Llama 3.3 70B throughput을 최대 3.55배 높일 수 있다고 공개했습니다. 즉 “이 회사에 자원이 있느냐”를 보는 기준은 단순히 GPU 장수만이 아니라, **양자화 허용 여부, 컨텍스트 길이, 동시성 목표, tool-use latency 요구**까지 포함해야 합니다. [\[26\]](https://docs.vllm.ai/en/latest/features/quantization/)

이때 API와의 break-even은 크게 갈립니다. GPT-4.1 mini의 blended 가격은 70/30 입력·출력 가정 시 **\$0.76 / 1M tokens**입니다. 이는 대략 1,026원/1M tokens 수준이므로, 4xH100 70B 시나리오가 이 API보다 싸지려면 월 약 **97억 토큰**이 필요합니다. 현실적 인터랙티브 사내 코파일럿 용도에서는 사실상 어렵습니다. 반면 GPT-5.5는 70/30 가정 시 **\$12.5 / 1M tokens**, 즉 약 16,875원/1M tokens 정도여서, 같은 4xH100 시나리오의 break-even은 월 약 **5.9억 토큰** 수준입니다. 50/50 가정이면 약 **4.2억 토큰**으로 내려갑니다. 즉 **저가 API와는 경쟁하지 말고**, **고가 frontier 모델을 대체하는 대량 민감업무**에서 로컬을 주력으로 삼는 것이 합리적입니다. [\[34\]](https://developers.openai.com/api/docs/models/gpt-4.1-mini)

또 하나의 포인트는 전력입니다. 예시 가정(80원/kWh, PUE 1.3)으로 보면 1xH100급 IT 부하 0.7kW의 월 전력·기본요금은 부가세·기금 포함 약 **6.6만원**, 4xH100급 2.8kW는 약 **26.6만원** 수준입니다. 이는 로컬 TCO에서 **GPU 감가상각과 운영인력보다 훨씬 작습니다**. 금융권의 그린 AI 논의도 결국은 “전기료 절감”보다 **과대형 모델 상시 가동을 줄이고, 적정 모델을 라우팅해 유휴 전력을 줄이는 것**이 핵심입니다. [\[23\]](https://home.kepco.co.kr/kepco/front/html/CY/E/E/CYEEHP00103.html)

## 하이브리드와 로컬 충분성의 시연 논거

금융권 데모에서 입증해야 할 문장은 “로컬도 돌아간다”가 아니라, 더 정확히는 **“거버넌드 하네스가 있으므로 로컬이 실무적으로 충분하다”**입니다. OpenAI는 에이전트 구축의 기본 요소를 모델·도구·상태·오케스트레이션으로 설명하고, 도구 사용 가이드에서 built-in tools, function calling, remote MCP를 제시합니다. LangGraph와 Deep Agents는 내구 실행, human-in-the-loop, 평가, 추적, 그리고 무엇보다 **모델 공급자 독립성**을 전면에 둡니다. 따라서 하네스가 잘 설계되면, 동일한 업무 흐름에서 로컬 모델과 상용 모델을 **정책적으로** 교체·승격할 수 있습니다. [\[35\]](https://developers.openai.com/tracks/building-agents)

작은 로컬 모델의 실무 가능성도 과소평가할 필요가 없습니다. Microsoft는 3.8B 규모의 Phi-4-mini에 대해 function calling, instruction following, long context, reasoning을 강조합니다. 이 메시지의 핵심은 “작은 모델도 도구를 잘 쓰면 쓸모 있다”는 것입니다. 금융 업무의 상당수는 사실 자유 대화보다 **정형 도구 호출**입니다. 예를 들면 사내 규정 검색, 표준 양식 채우기, 상품 조건 비교, 내부 API 질의, SQL/룰엔진 호출, 리스크 템플릿 생성입니다. 이런 업무에서 작은 로컬 모델은 “모든 것을 자체 추론”하는 대신 **조회·검증·계산을 도구에 위임**하면 됩니다. [\[36\]](https://azure.microsoft.com/en-us/blog/empowering-innovation-the-next-generation-of-the-phi-family/)

벤치마크도 이 방향을 뒷받침합니다. GAIA는 인간 92% 대비 GPT-4+plugins baseline 15%를 보고했는데, 이는 웹·도구·멀티스텝 실행이 필요한 문제에서 모델 자체만으로는 한계가 크다는 뜻입니다. 반대로 SWE-agent는 모델을 특정하지 않고 **하네스 자체**를 공개하며, mini-SWE-agent가 SWE-bench Verified 65%를 달성했다고 밝힙니다. 금융권에 그대로 대응시키면, “작은 로컬 모델은 안 된다”가 아니라 **“작은 모델 단독 채팅은 부족할 수 있지만, 툴·서브에이전트·검증루프가 있는 하네스에서는 충분히 많은 업무가 가능하다”**가 더 정확한 해석입니다. [\[37\]](https://openreview.net/forum?_hsenc=p2ANqtz-_PWlnjKxyBNXEe91UAT1mq3JzmUPs-bzE6LoL1yfFdN-JzUgSdn1qoJpzMpxOYDDV1XoP_&id=fibxvahvs3&utm_campaign=The+Batch&utm_medium=email&utm_source=hs_email)

### 제품·시연 적용 시사점

JB LocalGuard OS의 메시지는 “우리는 외부 LLM을 안 쓴다”가 아니라, **“우리는 민감데이터를 외부에 내보내지 않는 기본경로를 갖고 있고, 필요할 때만 통제된 승격을 한다”**여야 설득력이 큽니다. 아래 구성은 그 메시지를 가장 잘 드러냅니다. [\[15\]](https://www.fsc.go.kr/no010101/83594?curPage=&srchBeginDt=&srchCtgry=&srchEndDt=&srchKey=&srchText=)

- **PII 비반출 기본값**: 고객식별정보·상담기록·내부 규정·상품문서는 로컬 모델 경로로만 가게 한다. 외부 호출은 정책엔진이 차단하거나 승인 대기시킨다. 외부 API는 “가능한 옵션”이지 기본값이 아니다. [\[38\]](https://developers.openai.com/api/docs/guides/your-data)
- **동일 하네스, 다른 모델**: 같은 에이전트 하네스가 Codex·Claude(OAuth)·로컬 모델을 모두 받을 수 있음을 보여 준다. 모델은 교체 가능하고, 정책과 기록 체계는 유지된다는 점이 핵심이다. [\[39\]](https://github.com/langchain-ai/deepagents)
- **로컬 1차, 외부 2차 승격**: 공개 웹 검색, 비민감 코드 작업, 외부 공개자료 요약은 Codex·Claude 경로를 허용하되, 내부 고객 데이터가 섞이면 자동으로 로컬 경로로 강등되는 모습이 중요하다. [\[40\]](https://openai.com/solutions/use-case/agents/)
- **작은 로컬 모델의 실무성 시연**: 로컬 8B~32B 모델이 규정 QA, 상담 스크립트 초안, 상품 비교표 생성, 내부 API 호출 같은 업무를 무리 없이 처리하는 장면을 보여 준다. “전부 다 잘함”보다 “핵심 반복업무를 안정적으로 잘함”이 더 설득력 있다. [\[41\]](https://azure.microsoft.com/en-us/blog/empowering-innovation-the-next-generation-of-the-phi-family/)
- **검증 루프와 근거 제시**: 답변에 인용 근거, 도구 로그, 사람 승인, 실패 시 fallback를 붙이면 모델 크기보다 체감 신뢰가 높아진다. Morgan Stanley가 OpenAI 도입에서 eval framework를 전면에 둔 이유가 바로 이것이다. [\[12\]](https://openai.com/index/morgan-stanley/)
- **비용 메시지**: “모든 토큰을 premium API에 태우면 비싸고, 모든 것을 대형 로컬에 올리면 과투자”라는 점을 수치로 보여 준다. 대량·반복·민감 업무는 로컬이, 희소·고난도는 외부가 낫다는 곡선을 함께 제시하면 된다. [\[34\]](https://developers.openai.com/api/docs/models/gpt-4.1-mini)

## 갭과 미검증
