# D20 요약 — 금융 AX 에이전트는 특화모델을 묶는 운영체제다

## ⚡ 30초 핵심
금융 AX의 주 모델은 LLM이 아니라 특화모델·규칙엔진·레거시 어댑터를 묶는 하네스다.  
LLM은 점수화·예측·탐지를 직접 맡지 말고, 의도 분류·도구 선택·근거 패키징을 맡아야 한다.  
승부처는 모델 성능보다 계정계 접합, 승인 게이트, 추적로그, 비동기 상태 관리다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- **Score Skill 분리**: 신용평가·여신심사는 XGBoost/LightGBM/Random Forest, FDS는 규칙엔진+GNN/시퀀스, 조기경보는 RF/GBM, 문서 IE/OCR은 LayoutLMv3·Donut, 추천은 ranking 모델로 라우팅한다.
- **LLM 역할 고정**: LLM은 고객 의도 분류 → 필요한 근거 범주 결정 → score API/OCR/규칙엔진/코어뱅킹 어댑터 호출 → 점수·규칙 결과·문서 추출물·시스템 응답 요약까지만 맡긴다.
- **Legacy Adapter Skill 설계**: 카카오뱅크식 비동기 작업 체인, 상태 전이, 실시간 상태 대시보드, 기존 계정계 프레임워크 재사용 패턴을 차용한다.
- **Tool Registry + Policy Engine 우선 배치**: MCP, CLI, 내부 REST, 배치 잡, 검색 서버마다 테넌트·데이터 등급·승인 필요 여부·허용 파라미터·output redaction 계약을 둔다.
- **Audit Skill은 실행 로그로**: LLM 호출, 도구 호출, 상태 전이, 승인 이벤트, 규칙 판정, 모델 점수, 레거시 반영 결과를 OpenAI Agents SDK tracing식으로 남긴다.
- **Harness Benchmark로 설득**: 같은 모델에서 하네스 전후 pass rate, tool success rate, approval burden, time-to-resolution, legacy error rate를 비교 지표로 잡는다.

## 🙋 전문가에게 묻는다면
- **Q. LLM이 신용평가나 여신심사 엔진을 대체하나?** A. 아니다. 표형 수치·범주 변수는 XGBoost·LightGBM·Random Forest가 더 검증 가능하고 비용도 낮다.
- **Q. 그럼 LLM은 어디에 쓰나?** A. 의도 해석, 규정·문서 요약, 도구 선택, 결과 설명에 쓴다. 계산은 특화모델과 규칙엔진에 맡긴다.
- **Q. MCP를 붙이면 금융권 보안 문제가 풀리나?** A. 아니다. MCP는 연결 표준이고 authorization도 optional이다. 사내 하네스가 SSO/OAuth, allowlist, sandbox, 감사로그를 강제해야 한다.
- **Q. 왜 자체 하네스가 차별점인가?** A. Anthropic은 같은 모델도 scaffolding에 따라 SWE-bench 성능이 크게 달라진다고 밝혔다. 금융에서는 컨텍스트 배치, 승인 게이트, 툴 계약, 재시도, 로깅이 성능과 안전성을 만든다.
- **Q. 레거시 은행 시스템에는 어떻게 붙이나?** A. 빅뱅 교체를 피한다. 표준 API, 어댑터·래퍼, 비동기 작업 체인, 상태 대시보드, strangler형 점진 도입으로 붙인다.

## ⚠️ 주의·한계
- 한국 은행의 FDS·신용평가·추천 운영 성능 수치는 공개 1차 자료가 부족하다. 은행별 AUC·탐지율·오탐률은 대체로 비공개다.
- 벤더 블로그의 “N배 개선”, “정확도 99%”, “시간 98% 절감”류 숫자는 핵심 근거로 쓰지 않는다.
- ₩ON, BC Card Finance LLM Benchmark, KR-FinBert 계열은 존재 근거만 쓴다. 모델카드·README를 현업 운영 품질로 비약하지 않는다.
- MCP는 신흥 표준이다. 연결 표준의 성숙도와 금융 운영 보안 통제는 별개로 봐야 한다.
