# 전세사기 보호 담당자 역할 하네스 v3 — 업무 루프 콘솔 (2026-07-03)

원칙: **역할 = 독립 업무 하네스**, 그리고 **MoveValue의 부동산 데이터 아이디어는 "가격 데이터 엔진"으로만 흡수** —
화면/보드/메뉴/에이전트/케이스 생명주기는 전세사기 보호 업무 전용으로 새로 설계했다.
AI는 위험 "신호"와 확인 항목만 제안하고, 전세사기 여부·법률·보증·피해자 결정은 항상 담당자가 판단한다.

- roleKey: `jeonse-protection` · route base: `#/roles/jeonse-protection` · activeView: `jeonse-protection-harness`
- 저장소: `jpo-ops-db-v2` (`jpoRepository` 5개 진입점, scope 미지정 시 `role scope is required` 예외)

## 4. 공공데이터 엔진

- 프록시(`scripts/api-proxy.mjs`, :8020): 국토부 실거래 7종(dataset별 전용 키 → MOLIT_SERVICE_KEY →
  MOLIT_API_KEY → PUBLIC_DATA_API_KEY → DATA_GO_KR_KEY 체인) + 서울 열린데이터 키 프로브
  (SEOUL_OPEN_API_KEY/SEOUL_API_KEY/MOVEVALUE_SEOUL_OPEN_API_KEY). **키는 환경변수로만.**
- adapter(`fetchJeonseMarketData`): ?live=1 + 프록시 정상 → `live_api`; 아니면 지역 스냅샷(`snapshot`);
  둘 다 없으면 `fallback` — 이때 위험도를 낮게 확정하지 않고 최소 medium + "담당자 확인 필요".
- 공시가격 API 미연동: 매매 중앙값 기반 "추정" 공시가격만 표기(확정값 아님).

## 5. 위험 스코어링 (computeJeonseRiskAssessment)

신호 10종: JEONSE_RATIO_HIGH · DEPOSIT_OVER_OFFICIAL_PRICE · ABOVE_NEIGHBORHOOD_MEDIAN ·
LOW_COMPARABLE_COUNT · REGISTRY_RIGHTS_UNKNOWN · GUARANTEE_STATUS_UNKNOWN ·
AUCTION_OR_FORECLOSURE_DEADLINE · LANDLORD_RISK_MANUAL_REQUIRED · ILLEGAL_BUILDING_MANUAL_REQUIRED ·
CONTRACT_DATE_URGENT. 규칙: 등기/보증 unknown ⇒ 최소 medium · 경공매 ⇒ high+(D-14 이내 critical) ·
high/critical ⇒ requiresHumanReview=true & 자동 완료 금지(훅이 needsReview로 강등) ·
피해자 결정/법률/보증 ⇒ "담당자 확인 필요"로만.

## 6. Seed (JEONSE-0001~0006)

화곡동 다세대 전세가율 91%(high) · 신림동 오피스텔 보증 확인 필요 · 개봉동 아파트 전월세 비교(보강 중) ·
인천 미추홀 다가구 경공매 D-9(critical·fallback 스냅샷) · 대전 반환 지연(high·법률/금융/심리 연계) ·

## 8. v3 — 담당자 업무 루프 고도화

**루프**: Intake → Triage → Data Enrichment → Risk Analysis → **Evaluator Check(분리)** → Human Review → Referral → Audit/State.

- **화면 언어**: 내부 개발 용어를 담당자 업무 언어로 전면 교체(표시층만, 내부 key/id 유지) —
  `JEONSE-0001→전세위험-0001` · `CUST-JS→익명 고객` · `JEONSE-SNAP→시세 비교 기록` · `스냅샷→저장 기준` ·
  `중앙값→인근 거래 기준가` · `sourceMode→데이터 연계 상태` · `fallback→대체 기준 사용` ·
  `role scope 집계→전세보호 업무 기준 집계`. e2e가 4개 화면에서 내부 용어 부재를 스캔.
- **상세 패널(6섹션)**: 기본 식별 → 분석 결과(검토 필요 사유·확정 판정 아님 고지) → 담당 AI 업무지원(5 agent 최근 실행) →
  검토/승인 정책 → 근거 피드(jeonse_evidence) → 감사 로그. "상세 페이지 열기" 버튼 제공.
- **상세 페이지**: `/roles/jeonse-protection/cases/:caseId/full` — 탭 8(개요/위험 신호/시세·실거래/권리관계/보증·HUG/
  피해지원/AI 분석/근거·감사), 근거 무결성 확인 버튼, 새로고침 유지, 좁은 화면(≤1180px)은 카드 클릭 시 바로 상세 페이지.
- **Evaluator 분리**: `jpo-evaluator`(Loop Evaluator Agent, 11번째) + `runJeonseLoopEvaluator` — 생성 함수와 분리 실행.
  판정: 검증 통과/담당자 확인 필요/금지 표현 수정 필요/근거 부족/개인정보 노출 차단. 대체·저장 기준이면 담당자 확인 강제,
  high/critical 자동 완료 차단. 결과는 EVALUATOR_CHECKED 감사 + evidence + UI(하네스·AI 분석 탭)에 기록.
- **Evidence/Audit/State**: jeonse_evidence(price/comparable/registry/guarantee/referral/dataSource/evaluator) 신설,
  감사 액션 확장(CASE_SELECTED/DATA_ENRICHED/RISK_SIGNAL_CREATED/AGENT_RUN_CREATED/EVALUATOR_CHECKED/
  STATUS_CHANGED/SUPPORT_REFERRAL_CREATED), jpoState.loop(선택 케이스·연계 상태·검증 상태·Human Inbox 수).
- **Automation(준비만)**: 일일 분류·장기 미처리·SLA 임박·시세 갱신·검증 실패 **감지 전용**(JPO_AUTOMATION_RULES +
  `/jeonse-daily-triage`), UI 표기 "자동 실행 준비됨" — 자동 종결/자동 승인 없음.
- **lifecycle alias**: JPO_STATUS_CANONICAL이 확장 상태명(enrichingData 등 9종)을 canonical 7종으로 매핑.
