# 05 · RM(관계관리 담당자) 역할 하네스

> 역할 하네스는 화면 필터가 아니라 독립 업무 하네스다. 이 문서는 RM 역할 하네스(scope `rm-officer`)의
> 제품·UX·에이전트·MD 산출물 설계를 정리한다.
> 기준 구현: `jeonse-protection` / 두 번째 선례: `corporate-credit`.
> 상호 참조(볼트 산출물): `08_본선/03_제품/07_역할-하네스/_HARNESS-RM.md` (커밋 dfc6ffb),
> 실행뷰 설계도 `08_본선/03_제품/설계도/03-에이전트-실행뷰.md`.

## 1. 제품 한 줄 설명

RM이 담당 고객의 여신 상담 케이스를 **"급한 순 근거"로 모아 보고**, 케이스마다 필요한 AI 에이전트를
**키보드 Enter 한 번으로 승인·실행**해 개별/통합 **MD 리포트**로 검토하는 내부 업무 지원 콘솔.
(실제 승인·금리·한도·신용평가·정책자금 대상 확정은 하지 않는다 — 모든 산출물은 내부 업무 참고용.)

## 2. 핵심 사용자 여정

1. 역할 레일에서 **RM** 진입 → 업무보드가 급한 순으로 케이스를 정렬해 보여준다.
2. 카운트 헤더(작업 전/중/후)와 필터(전체·작업 전·중·후)로 오늘 처리할 일을 좁힌다.
3. **숫자키로 케이스를 선택** → SUB에 케이스 헤더 + 상태 칩 + **에이전트 승인 큐**가 열린다.
4. 승인 큐에서 현재 선택된 에이전트에 **Enter로 승인·실행** → (모의) 실행이 개별 `.md` 산출물을 만든다.
5. 배정 에이전트를 모두 실행하면 **통합본 `.md`**가 자동 생성된다(개별 md를 옵시디언식 링크로 연결).
6. LLM 지연은 "다른 케이스 보기"로 흡수 — 실행 중에도 보드로 돌아가거나 다른 케이스에 진입할 수 있다.
7. 완료/고위험 케이스는 **승인 라우팅**으로 넘어가 사람 검토를 받는다.

## 3. 화면별 UX 구성 (Figma SSOT 3장면)

- **화면 A · 업무보드**: 좌측 조직 레일 + 전용 사이드바("RM 업무지원 포털"), 우측 필터
  (`전체` / `처리해야할 작업 전`(기본) / `처리해야할 작업 중` / `처리해야할 작업 후`),
  배너 "RM님 업무 급한 순으로 모아왔어요" + 힌트 "키보드 숫자를 눌러 Case를 확인하세요",
  카운트 헤더(진행 전/중/후 개수), 케이스 카드 가로 리스트(상태 배지·고객·케이스번호·지역·은행·
  주제·상황·우선순위 근거·필요 에이전트 칩).
  - **SUB**: 업무보드(위)와 시각적으로 분리된 영역(좌측 레일 `SUB` 라벨 + 배경 밴드 + 구분선,
    `rmo-sub-band`/`rmo-board-sub-divider`). 헤더 카드(Case·날짜·위험등급·제목·고객/사업자·도메인·
    계열사·처리 목표·현재 위험 신호 칩·`고객 정보` 버튼) + 상태 칩(작업 전/중/후) + **에이전트 업무
    계층도(Agent Work Map, 10장)** + **산출물 보드**(개별 md·근거 데이터·통합 리포트·승인/재실행 버튼).
    실행 중 카드는 딤 오버레이 + "조금만 기다려주세요" + **불확정(indeterminate) 스피너**.
- **화면 B · 진행 상태 + 통합 MD 뷰어**: 케이스 배지가 `작업 완료`로 전환, 다른 케이스는 `분석중`.
  뷰어 탭 `[통합본 | 개별 에이전트.md ×N]` + 문서 본문 + 접이식 섹션
  `[사용 에이전트/스킬 표(에이전트·산출물.md·사용 데이터·날짜·관여율) | 사용 출처(링크)]`.
- **화면 C · 문서 모달**: 보드 딤 처리 + MD 플로팅 모달(메타데이터 블록 → 제목 → Summary → 근거 표).
  통합본 안에서 개별 md로 링크 이동(옵시디언식).

## 5. 키보드 인터랙션 규칙

| 키 | 동작 |
| --- | --- |
| 숫자 `1`~`9` | 업무보드에서 해당 순번 케이스 선택(급한 순 정렬 기준) |
| `←` `→` | 케이스 카드를 클릭(포커스)한 뒤 이전/다음 케이스로 이동(업무보드 순서 기준) |
| `↑` `↓` | 업무 계층도 노드(총괄→브랜치→리포트) 이동 |
| `Space` | 포커스된 노드의 상세 펼치기(사용 데이터·도구/스킬·산출물 경로) |
| `Enter` | 포커스된 노드 실행 승인(개별 md 생성, report는 선행 브랜치 전원 완료 후에만 가능) |
| `R` | 완료/반려 노드 재실행(이전 산출물을 정리하고 새로 생성) |
| `A` | 케이스 통합 보고서 직원 최종 승인(report가 완료/승인대기 상태일 때만 활성화) |
| `Esc` | 문서 모달 닫기 / 케이스 선택 해제 후 보드 복귀 |

- 모든 바인딩은 RM 콘솔 스코프(`rmoModeActive()`) 안에서만 활성화되며 타 콘솔·전역 단축키를 침범하지 않는다.
- **Enter/A 하나에 모든 책임을 주지 않도록** 현재 선택 상태(케이스·노드)는 `rmo-node-focused` 강조 테두리로 항상 시각적으로 명확하다.
- 입력 필드(input/textarea/select) 포커스 중에는 단축키를 가로채지 않는다. 마우스 클릭(노드 카드·버튼)도 키보드와 동일하게 모두 지원한다.
- 승인하지 않을 노드는 그대로 둘 수 있다(회색·파랑 유지). 실행 중=로딩, 완료=딤/체크, 반려=빨강, 승인대기=보라로 상태 변화를 표시한다.

## 6. 에이전트 워크플로우

Case → **Case Orchestrator**(RM Case Triage Orchestrator, 우선순위 근거 산정·자동 완료) →
**분석 브랜치 에이전트**(케이스 유형별 3~5개, 상황분석/거래패턴/정책·내규검토/외부데이터조회 등) →
**최종 보고서 작성 Agent**(브랜치 전원 완료 후에만 실행 가능) → **Human Approval**(직원 최종 승인).

- 기존 4개 케이스 유형(재해·상환·생활금융·정책창업)은 등록된 11개 에이전트를 재사용하고,
  신규 3개 케이스 유형(기업여신·기술신용/보이스피싱 대응/농수산 사후관리)은 케이스 서사에 맞춘
  전용 에이전트 16개를 추가해 총 27개 에이전트가 등록되어 있다(`rmOfficerAgents.registry.js`).
- 에이전트는 계층적으로(에이전트 → 스킬/도구 → 데이터/출처) 차별성이 드러나되 기술적으로 과하게 보이지 않게 구성.
  각 노드는 "왜 이 에이전트가 필요한지"(role/reason)와 사용 데이터·도구를 함께 노출한다.
- 실행은 **시뮬레이션**(모의 실행 + `rm_officer_agent_runs`/`rm_officer_agent_handoffs` 기록)이며,
  선택적으로 로컬 Ollama 모델을 붙일 수 있다(실패 시 graceful degrade + `needsReview` 기록).
- **high/critical 위험은 자동 완료하지 않고** report 노드를 `needsApproval`(보라)로 남겨
  `humanReview` + 승인 라우팅으로 강등된다(가드레일 불변식, 10장 참고).

## 7. MD 결과물 구조

- **개별 에이전트 `.md`**: 프론트매터(case·agent·org·date·note) → `# 제목` → `## Summary`(3~4문장) →
  `## 상황 분석`(케이스 서사 기반 단락) → `## 근거`(4~6행 표: 항목·내용·출처, 구체적 더미 수치 포함) →
  `## 판단 및 권고`(단계별 1)2)3) 목록) → `## 다음 조치 태스크`(체크리스트) → `## 예상 기대값` →
  `## 한계 및 주의`(가드레일 인용문 — 실제 승인·금리·한도·신용평가·정책자금 대상 확정이 아님을 고지).
  에이전트 종류(리스크 분석/케어/액션/보고 등)별로 문체와 근거 내용을 차별화했다(`RMO_DELIVERABLE_TEMPLATES`).
- **통합본 `.md`**: 상황 → 우선순위 근거 → **개별 산출물 링크(`[[파일명]]`)** →
  **사용 에이전트/스킬 표**(에이전트·산출물.md·사용 데이터·날짜·관여율) → **사용 출처** 목록.
- 뷰어/모달은 자체 생성 MD만 렌더하며 `escapeHtml`로 안전하게 표시한다. AI 산출물 시맨틱은
  **"내부 업무 참고용 · 담당자 검토 필요"**.

## 8. MVP 범위와 고도화 범위

- **MVP(현재)**: 업무보드(급한 순·근거)·키보드 승인 큐·개별/통합 MD·우선순위 근거·승인 라우팅·
  scope 강제 저장소·보안 훅(PII/자동 종결/승인)·자체 검증 루프·모의 실행 + 선택적 로컬 모델.
- **고도화**: 실데이터 커넥터(기상특보·급여주기·정책자금 요건) 연결, 에이전트 실행 병렬화·재시도,
  통합본 자동 요약 품질 강화, 콜백/알림 채널 연동, 승인 매트릭스(L1~L4) 정교화, RAG 근거 인용 링크 심화.

## 9. 해커톤 발표 차별점

1. **근거 기반 우선순위** — 단순 점수가 아니라 "왜 급한지" 문장과 데이터 출처를 함께 노출한다.
2. **키보드 퍼스트 승인 큐** — 숫자키 선택 → Enter 승인으로 케이스를 빠르게 처리(마우스 최소화).
3. **개별 → 통합 MD 파이프라인** — 에이전트별 산출물을 옵시디언식 링크로 엮은 통합 리포트로 검토 동선 단축.
4. **가드레일 내장** — 실제 금융 판단 금지·PII 차단·high/critical 자동 종결 금지를 코드/훅/검증기로 강제.
5. **그룹 확장성 증명** — 전세보호·기업여신·JB우리캐피탈에 이은 4번째 독립 역할 하네스로 표준 계약을 재입증.
6. **케이스 업무 계층도** — "AI가 다 한다"가 아니라 업무가 어떻게 쪼개지고 누가 검토하는지 보여주는
   조직도형 UI로 신뢰를 만든다(10장).

## 10. 케이스 업무 계층도 (Agent Work Map)

### 기능 정의문

> 케이스를 클릭하면 하네스는 해당 업무를 처리하기 위한 AI 에이전트 계층도를 자동으로 펼친다.
> 직원은 각 에이전트가 어떤 근거를 보고 어떤 산출물을 만드는지 확인한 뒤 실행을 승인한다.
> 에이전트들은 개별 MD 보고서를 생성하고, 최종 통합 에이전트가 담당자용 검토 보고서를 만든다.
> 최종 판단과 대외 조치는 반드시 직원이 승인한다.

### 핵심 개념

"AI가 알아서 다 한다"가 아니라 **이 케이스를 처리하기 위해 어떤 업무가 어떤 순서로 쪼개지고, 어떤
에이전트가 어떤 근거를 보고, 어떤 산출물을 만드는지** 보여주는 구조다. 케이스 분해 형태:

```
Case
 └─ Case Orchestrator(총괄, 자동 완료 — 우선순위 근거 산정)
     ├─ 분석 브랜치 1..N (케이스 유형별 3~5개 — 상황 분석 / 거래·입금 패턴 /
     │   정책·내규 검토 / 외부 데이터 조회 / 위험도 판단 등)
     └─ 최종 보고서 작성 Agent(모든 브랜치 완료 후에만 실행 가능) → Human Approval
```

케이스 선택 → 케이스 요약(상단) → 업무 목표 정의(처리 목표) → 필요한 에이전트 계층도 표시(중단) →
각 에이전트 실행 승인(Enter) → 결과 MD 생성 → 통합 보고서 생성(모든 브랜치 완료 시 report 노드 실행
가능) → 직원 최종 승인(A, 하단)의 흐름을 화면 하나에서 보여준다.

### 노드 스키마 (11필드)

`agentId, agentName, role, reason, inputData, tools, expectedOutput, status, riskLevel,
requiresApproval, outputMdPath` — `rm_officer_agent_assignments` 행에 저장되며(`kind: "branch" |
"report"`), `role`은 "이 케이스에서 하는 일"(예: "최근 30일 입금 공백과 지출 증가 원인 분석"),
`reason`은 에이전트 자체의 일반 역량 설명이다. Space 키로 `inputData`/`tools`/`outputMdPath`를
펼쳐볼 수 있다.

### 상태 6색 (실행 상태가 살아있는 도식)
