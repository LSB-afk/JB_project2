# 로컬 백엔드 서버 구현

> 목적: 기존 정적 SPA/localStorage 데모를 유지하면서, 실제 API 서버와 영속 저장소를 갖춘 백엔드 기준선을 만든다.

## 현재 구현 범위

- 런타임: Node 내장 모듈 기반 HTTP 서버 (`server/index.mjs`)
- 저장소: 파일 기반 JSON DB (`server/data/localguard-db.json`, git ignore)
- Supabase 저장소: `JB_DB_DRIVER=supabase`일 때 Supabase REST API의 `jb_backend_state` JSONB 테이블 사용
- seed: 역할, 케이스, 감사 로그 초기 데이터 (`server/lib/seed.mjs`)
- repository: 직렬 mutation queue와 atomic write (`server/lib/repository.mjs`)
- 정적 앱 서빙: `/index.html`, `/styles.css` 등 기존 `app/` 파일을 같은 서버에서 제공
- API prefix: `/api/*`

새 의존성을 추가하지 않았다. 개발 환경에서 `npm install` 이후 바로 실행된다.

## 실행

```bash
npm run backend
```

기본 주소:

```txt
http://127.0.0.1:8010
```

환경변수:

| 이름 | 기본값 | 의미 |
| --- | --- | --- |
| `JB_BACKEND_PORT` 또는 `PORT` | `8010` | 백엔드 포트 |
| `JB_DB_DRIVER` | `json` | `json` 또는 `supabase` |
| `JB_BACKEND_DB` | `server/data/localguard-db.json` | 로컬 JSON DB 경로 |
| `SUPABASE_URL` 또는 `JB_SUPABASE_URL` | `https://wtahqybymvtiwypmzsbw.supabase.co` | Supabase API URL |
| `SUPABASE_SERVICE_ROLE_KEY` 또는 `SUPABASE_SECRET_KEY` | 없음 | 서버 전용 Supabase key |
| `JB_SUPABASE_TABLE` | `jb_backend_state` | Supabase 상태 테이블 |
| `JB_SUPABASE_STATE_ID` | `localguard` | 상태 row id |
| `JB_STATIC_ROOT` | `app` | 정적 앱 루트 |
| `JB_PUBLIC_DATA_PROXY` | `http://127.0.0.1:8020` | 기존 공공데이터 프록시 주소 |
| `JB_OLLAMA_PROXY` | `http://127.0.0.1:8030` | 기존 Ollama 프록시 주소 |

## Supabase API 저장소

Supabase 방식은 대시보드 링크가 아니라 API URL과 서버 전용 key를 사용한다.

현재 프로젝트 ref 기준 URL:

```txt
https://wtahqybymvtiwypmzsbw.supabase.co
```

1. Supabase Dashboard의 SQL Editor에서 `server/sql/supabase-api-state.sql`을 실행한다.
2. 로컬 터미널에 서버 전용 key를 환경변수로 설정한다.
3. `JB_DB_DRIVER=supabase npm run backend`로 서버를 실행한다.

예시:

```bash
export JB_DB_DRIVER=supabase
export SUPABASE_URL="https://wtahqybymvtiwypmzsbw.supabase.co"
export SUPABASE_SERVICE_ROLE_KEY="<server-only-key>"
npm run backend
```

주의:

- `SUPABASE_SERVICE_ROLE_KEY` 또는 `SUPABASE_SECRET_KEY`는 브라우저에 넣지 않는다.
- `.env`, `.env.local`은 git ignore 대상이다.
- 현재 Supabase 구현은 기존 API 계약을 빠르게 보존하기 위한 `payload jsonb` 상태 row 방식이다.
- 운영 전환 시 케이스, 파일, 실행 기록, 산출물, 감사 로그를 정규화 테이블로 분리해야 한다.

## API 계약

| Method | Path | 역할 |
| --- | --- | --- |
| `GET` | `/api/health` | 서버/DB health |
| `GET` | `/api/roles` | 역할 목록 |
| `GET` | `/api/roles/:roleKey/dashboard` | 역할별 대시보드 요약, 케이스, 실행 큐, 최근 감사 |
| `GET` | `/api/cases?roleKey=&status=&q=` | 케이스 목록/검색 |
| `POST` | `/api/cases` | 신규 케이스 접수 및 감사 로그 생성 |
| `GET` | `/api/cases/:id` | 케이스 상세 + 파일 + 산출물 + 감사 로그 |
| `POST` | `/api/cases/:id/files` | JSON 또는 multipart 파일 업로드 |
| `POST` | `/api/agent-runs` | 결정론적 에이전트 실행, 산출물 생성, 감사 로그 기록 |
| `GET` | `/api/agent-runs/:id` | 실행 기록 조회 |
| `GET` | `/api/deliverables?roleKey=&caseId=` | 산출물 조회 |
| `GET` | `/api/audit-logs?roleKey=&caseId=` | 감사 로그 조회 |
| `GET` | `/api/public-data/jeonse/market` | 공공데이터 프록시 연동, 미연결 시 snapshot fallback |
| `GET` | `/api/model-runtime/health` | Ollama 프록시 health, 미연결 시 fallback |
| `POST` | `/api/model-runtime/run` | Ollama 실행 프록시, 미연결 시 결정론적 fallback |

개발 모드 전용:

| Method | Path | 역할 |
| --- | --- | --- |
| `POST` | `/api/dev/reset` | seed DB 재생성 (`NODE_ENV=production`에서는 비활성) |

## 업무 흐름

1. `POST /api/cases`로 케이스가 접수된다.
2. 서버는 케이스를 DB에 저장하고 `CASE_CREATED` 감사 로그를 append 한다.
3. `POST /api/cases/:id/files`로 등기 요약, 상담 메모 같은 분석 자료를 첨부한다.
4. 서버는 파일 메타/preview를 저장하고 `FILE_UPLOADED` 감사 로그를 append 한다.
5. `POST /api/agent-runs`로 역할 에이전트를 실행한다.
6. 서버는 실행 기록, MD 산출물, `AGENT_RUN_COMPLETED` 감사 로그를 한 번의 mutation으로 저장한다.
7. `GET /api/cases/:id`에서 케이스, 파일, 산출물, 감사 로그를 한 흐름으로 조회한다.

## 보안 경계

현재 백엔드는 데모용 로컬 서버다. 운영 전환 전 다음이 필요하다.

- SSO/IAM 기반 인증
- RBAC와 서버 측 row-level scope 강제
- PostgreSQL 또는 은행 운영 DB로 저장소 교체
- 감사 로그 hash chain/WORM 저장
- 파일 원문 저장소 분리와 바이러스/PII 스캔
- 외부 API credential 시크릿 매니저 관리
- 운영망 배포와 접근 로깅

현재 구현은 그 전 단계인 "실제 서버가 로컬에서 동작하고, UI가 나중에 fetch 어댑터로 갈아탈 수 있는 백엔드 표면"이다.

## 검증

```bash
npm run backend:test
npm test
```

`tests/backend/server.test.mjs`가 다음을 증명한다.

- 서버 health와 역할 대시보드 API
- Supabase REST API 저장소 드라이버의 seed/write/read 흐름
- 케이스 생성, 파일 업로드, 에이전트 실행, 산출물 생성, 감사 로그 저장
- 공공데이터/Ollama 프록시 부재 시 fallback
- API 서버가 정적 앱 shell도 함께 서빙
