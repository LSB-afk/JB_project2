# 백엔드 구축 루프 검증 기록

> 목적: 백엔드 구축을 한 번의 프롬프트가 아니라 발견 → 구현 → 검증 → 실패 보완 → 재검증 루프로 남긴다.

## 루프 원칙

- 목표: 로컬에서 실제 API 서버가 뜨고, 케이스/파일/에이전트 실행/산출물/감사 로그가 DB에 영속화된다.
- 상태 파일: 이 문서가 진행 상태와 검증 근거를 남긴다.
- 검증 기준: 문법 검사, 백엔드 API 테스트, 정적 계약 검증을 모두 통과해야 한다.
- 실패 처리: 실패 로그를 먼저 기록하고, 원인을 좁게 수정한 뒤 같은 검증을 재실행한다.

## Loop 1 — 백엔드 표면 생성

### 발견

- 기존 앱은 정적 SPA와 `localStorage` mock DB 중심이다.
- 기존 보조 서버는 공공데이터 프록시와 Ollama 프록시뿐이다.
- 운영용 서버 API, DB, 케이스/감사/파일 endpoint는 없었다.

### 구현

- `server/index.mjs`: HTTP API 서버와 정적 앱 서빙
- `server/lib/seed.mjs`: 역할/케이스 seed DB
- `server/lib/repository.mjs`: JSON 파일 저장소와 직렬 mutation
- `tests/backend/server.test.mjs`: API 동작 테스트

### 검증

```bash
node --check server/index.mjs
node --check server/lib/repository.mjs
npm run backend:test
```

결과: 통과.

## Loop 2 — 문서/계약 보강

### 발견

- 코드만 추가하면 다음 세션에서 "백엔드가 어디까지 구현됐는지" 다시 추측해야 한다.
- 기존 정적 검증 스크립트는 `server/` 파일을 필수 계약으로 보지 않는다.

### 보완

- `docs/06-백엔드-서버.md`: API 계약과 실행 방법 문서화
- `docs/07-백엔드-루프-검증.md`: 루프 상태 파일
- `scripts/verify_static.py`: 백엔드 파일, 문서, npm script, Node 문법 검사를 정적 계약에 편입

### 재검증

```bash
npm test
npm run backend:test
```

결과: 통과.

## Loop 3 — 실서버 smoke 실패 보완

### 실패

`node server/index.mjs`로 실제 서버를 띄운 뒤 `/api/health`를 확인하는 smoke test에서 연결 실패가 발생했다.

추가 확인 결과:

- `node server/index.mjs` 직접 실행 감지가 상대 경로 실행에서 안정적이지 않았다.
- 빈 임시 DB 파일이 이미 있으면 seed DB를 쓰지 않아 이후 읽기 단계가 깨질 수 있었다.

### 보완

- `server/index.mjs`: `fileURLToPath(import.meta.url)`와 `path.resolve(process.argv[1])` 비교로 CLI 직접 실행 감지 수정
- `server/lib/repository.mjs`: DB 파일이 없거나 비어 있으면 seed DB를 생성하도록 보완
- `tests/backend/server.test.mjs`: 빈 DB seed 회귀 테스트와 CLI entrypoint 회귀 테스트 추가

### 재검증

```bash
node --check server/index.mjs
node --check server/lib/repository.mjs
node --check tests/backend/server.test.mjs
npm run backend:test
npm test
git diff --check
```

결과: 통과.

실서버 smoke:

```bash
JB_BACKEND_DB=<empty-temp-db> JB_BACKEND_PORT=8124 node server/index.mjs
curl http://127.0.0.1:8124/api/health
curl http://127.0.0.1:8124/api/roles
curl http://127.0.0.1:8124/api/roles/jeonse-protection/dashboard
```

결과: `ok=true`, 역할 5개, 전세사기 대시보드 1건, 실행 큐 4개 응답 확인.

## Loop 4 — Supabase API 저장소 전환

### 발견

- 사용자 선택은 Supabase API 방식이다.
- 기존 백엔드 API 핸들러는 `repository.read/mutate/reset`에 의존하므로 저장소 드라이버만 교체하면 API 계약을 유지할 수 있다.
- 실제 service role key는 로컬에 없으므로 원격 Supabase 실접속 대신 fake Supabase REST 서버로 계약을 검증해야 한다.

### 구현

- `server/lib/supabaseRepository.mjs`: Supabase REST `/rest/v1` 호출 기반 저장소 드라이버
- `server/sql/supabase-api-state.sql`: Supabase SQL Editor에서 실행할 `jb_backend_state` 테이블
- `server/index.mjs`: `JB_DB_DRIVER=supabase` 환경변수로 저장소 전환
- `tests/backend/server.test.mjs`: fake Supabase REST 서버로 header, seed, upsert, read 검증

### 검증

```bash
node --check server/index.mjs
node --check server/lib/supabaseRepository.mjs
node --check tests/backend/server.test.mjs
npm run backend:test
npm test
git diff --check
```

결과: 통과. Supabase 실 key가 없으므로 원격 프로젝트 실접속은 하지 않았고, fake Supabase REST 서버로 `/rest/v1` header, seed, upsert, read 계약을 검증했다.
